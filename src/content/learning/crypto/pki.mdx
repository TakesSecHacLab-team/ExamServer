## 初めて訪れるサーバーをブラウザはなぜ信頼できるのか

これまで一度も訪れたことのない新しいデバイス、新しいネットワークから`https://google.com`にアクセスする場面を考える。

ブラウザは`https://`で接続し、鍵のアイコンが表示される。
このときブラウザは、接続先が本当にGoogleのサーバーであることを、どうやって確認しているのだろうか。

HTTPSのサイトを作ることそのものは難しくない。
SSL証明書(後で説明する仕組み)は、Let's Encryptのような団体から無料で取得できる。
攻撃者がフィッシング用のサイトをHTTPS化することも技術的には可能である。

では、フィッシングサイトとGoogleの本物のサイトの違いはどこにあるのか。
答えは**証明書チェーン**の信頼性にある。
ブラウザはOSから受け継いだ「信頼できる認証局のリスト」を使って、接続先が提示する証明書が正当なものかを検証する。

## 公開鍵証明書：公開鍵とドメインを結びつける仕組み

公開鍵暗号にはある問題がある。
公開鍵そのものは、誰が見ても「ただのデータ」であり、それが本当に名乗っている相手のものかを、受け取った側だけでは確認できない。

```
利用者: 「Googleの公開鍵をください」
攻撃者: 「これがGoogleの公開鍵です」（実際は攻撃者自身の鍵）

利用者: 攻撃者の公開鍵でデータを暗号化して送信
  ↓
攻撃者: 自分の秘密鍵でそのデータを復号できる
```

暗号化したつもりが、攻撃者に内容を読まれてしまう。
この問題を解決するために、「この公開鍵は本当にこのドメインのものである」という事実を、信頼できる第三者が保証する仕組みが作られた。
これが**公開鍵証明書**(X.509という規格に基づく形式)である。

```
【証明書の内容の例】
ドメイン: google.com
公開鍵: [Googleの公開鍵]
発行者: Google Trust Services LLC
有効期間: 2024/06/01 〜 2025/06/01
署名: [発行者の秘密鍵による署名]
```

証明書の末尾にある署名が重要な役割を持つ。
発行者(この例ではGoogle Trust Services)が、自分の秘密鍵を使って証明書全体に署名している。

```
受信側の検証手順:
  1. 証明書に書かれた公開鍵を取り出す
  2. 発行者の公開鍵を使って、証明書の署名を検証する
  3. 「本当にGoogle Trust Servicesが署名したものか」を確認する
     一致する → このドメインの公開鍵は本物だと判断できる
     一致しない → 何者かが改ざんしたと判断し、信用しない
```

この手順を踏んでも、一つ問題が残る。
「Google Trust Servicesの公開鍵自体が本物か」という疑問に、まだ答えていない。

## 認証局と証明書チェーンによる信頼の連鎖

証明書を発行する第三者を**認証局**(CA、Certificate Authority)と呼ぶ。
「発行者の公開鍵は本物か」という疑問に答えるため、認証局は階層構造を作っている。

```
【信頼チェーン】

ブラウザの信頼ストア（OSが管理）
  ↓ 信頼している
ルートCA（例：大手認証局のルート証明書）
  ↓ 署名によって保証する
中間CA（例：Google Trust Services）
  ↓ 署名によって保証する
サーバー証明書（google.com の証明書）
```

実際の検証は次の順序で進む。

```
1. ブラウザが google.com に接続する
2. サーバーが「サーバー証明書」を提示する
3. ブラウザが中間CAの公開鍵を使って、サーバー証明書の署名を検証する
   検証に通れば、Google Trust Servicesが署名したことが確かめられる
4. ブラウザが続けて、ルートCAの公開鍵を使って、中間CAの証明書を検証する
   検証に通れば、ルートCAが中間CAを認可したことが確かめられる
5. ルートCAの公開鍵は、ブラウザの信頼ストアにあらかじめ組み込まれている
   つまり、ルートCAについてはOSとブラウザがすでに信頼している
```

この連鎖があるからこそ、初めて訪れたサーバーでも、証明書をたどることで本物か偽物かを判定できる。
連鎖の出発点(ルートCA)だけは、あらかじめ信頼するものとしてブラウザに組み込まれている。

## ブラウザの信頼ストアとCAの責任

Windows、macOS、Linux、主要なブラウザ(Chrome、Firefox、Safari)は、あらかじめ「信頼できるルートCA」のリストを保持している。
このリストを**信頼ストア**と呼ぶ。

```
信頼ストアに含まれるCAの例:
  - DigiCert Global Root CA
  - GlobalSign Root CA
  - Comodo RSA CA
  ...（数百件規模）
```

新しいCAがこのリストに追加されるには、一定の基準を満たす必要がある。

- セキュリティの運用体制が十分であるか
- 不正な証明書を発行していないか
- 外部の監査を受けているか

一度信頼ストアから除外されると、そのCAが過去に発行したすべての証明書が、まとめて「信頼できない」と判定される。

2011年、オランダの認証局DigiNotarが攻撃者に侵害され、フィッシング用のサイトに対して証明書が発行される事件が起きた。
この事件の後、主要ブラウザはDigiNotarを信頼ストアから削除し、同社は事実上事業を継続できなくなった。

<Callout title="認証局に求められる責任の重さ">
認証局が不正な証明書を一件でも発行すれば、その認証局を信頼する仕組み全体の信頼性が揺らぐ。ブラウザベンダーは認証局に対して厳格な基準を適用しており、重大な不正が発覚すれば、数時間以内に信頼ストアから削除される場合もある。
</Callout>

## ドメイン検証と企業認証と拡張認証という三つの確認水準

証明書を発行する際、認証局がどこまで身元を確認するかによって、証明書には三つの種類がある。

**ドメイン検証**（DV、Domain Validation)は、認証局が確認するのが「申請者がそのドメインを管理している」という事実だけにとどまる種類である。

```
申請者が domain.com 用の証明書を申請する
認証局: 「本当に domain.com の管理者か」を確認する
  → 確認メールへの応答、または DNS レコードへの確認用文字列の追加によって確認する
認証局: 確認できた → 証明書を発行する
```

確認にかかる時間は数分から数時間程度であり、Let's Encryptが発行する証明書もこの種類に当たる。

DV証明書が証明するのは、ドメインの管理権限だけである。
「そのドメインが特定の企業のものか」は確認の対象に含まれない。
そのためフィッシング用のドメインであっても、DV証明書は取得できる。

```
攻撃者: evil.example.com というドメインを取得する
認証局にDV証明書を申請する
認証局: evil.example.com の管理権限を確認する
  → 確認できる(攻撃者自身がそのドメインを管理しているため)
認証局: evil.example.com 用のDV証明書を発行する

利用者がブラウザで見る情報:
  URL: evil.example.com
  https:// と鍵アイコンが表示されている
  証明書: 有効なDV証明書

利用者が「httpsだから安全」と判断してクリックし、フィッシングの被害を受ける
```

**企業認証**（OV、Organization Validation)は、ドメインの管理権限に加えて、申請する企業の実在を確認する種類である。

```
申請者が domain.com 用の証明書を申請する（企業名：Example Corp）
認証局: 「本当に Example Corp の管理下にあるドメインか」を確認する
  → 企業の登記情報を確認する
  → 企業の所在地と電話番号を確認する
  → WHOIS情報に登録されている企業情報を確認する
```

確認には数日から数週間かかる。
OV証明書が発行されているという事実は、その背後に実在する企業があることの一つの裏付けになる。

**拡張認証**（EV、Extended Validation)は、三つの中で最も厳格な確認を行う種類で、法人登記、営業実績、第三者機関による監査など、複数の観点から確認する。

```
金融機関や大手企業が利用する例が多い
過去のブラウザでは、アドレスバーに企業名を表示する仕様もあった
例: 【Google Inc.】https://google.com
```

現在では、多くのブラウザがEV証明書とDV証明書の表示を区別しなくなっており、EV証明書が利用者に与える視覚的な効果は以前より小さくなっている。

<Callout title="HTTPSであることと安全であることは同じではない">
HTTPSでの接続だからといって、そのサイトの運営者が信頼できるとは限らない。フィッシングサイトもDV証明書を取得できる。HTTPSが保証するのは「通信が暗号化されている」という事実だけである。運営する企業の実在性を確認するにはOV以上の証明書が一つの目安になり、最終的にはアドレスバーに表示されたドメイン名そのものを確認する必要がある。
</Callout>

## 証明書の失効をどう伝えるか：CRLとOCSP

発行済みの証明書は、秘密鍵が漏洩した、ドメインの管理権限を失った、といった理由で、有効期間の途中で失効することがある。
失効した事実をどう関係者に伝えるかにも、複数の方式がある。

**CRL**（Certificate Revocation List、証明書失効リスト)は、失効した証明書のシリアル番号を一覧にして配布する方式である。

```
失効リストの例:
  シリアル: 1a2b3c4d... → 失効日時
  シリアル: 5e6f7g8h... → 失効日時
  ...
```

ブラウザは定期的にこのリストをダウンロードして参照する。
欠点は、発行される証明書の数が増えるにつれてリストが大きくなり、配布の効率が下がることである。

**OCSP**（Online Certificate Status Protocol)は、証明書が有効かどうかを認証局へリアルタイムに問い合わせる方式である。

```
ブラウザ: 「シリアル番号 1a2b3c4d の証明書はまだ有効ですか」
認証局のサーバー: 「はい、有効です」
ブラウザ: サイトへの接続を続ける
```

リアルタイムに確認できる一方、認証局のサーバーへの問い合わせが遅延すると、接続全体が遅くなる場合がある。

この遅延を抑える工夫が**OCSPステープリング**である。
サーバー側があらかじめ認証局からOCSPの応答を取得しておき、クライアントとの接続時に、認証局の署名付きの「失効していない」という情報をその場で提示する。

## Let's Encryptが変えた証明書の取得しやすさ

2015年より前、SSL証明書の取得には年間数千円から数万円の費用がかかるのが一般的だった。
費用が壁となり、小規模なサイトの運営者はHTTPSの導入を見送る場合が多かった。

**Let's Encrypt**は非営利団体が運営する認証局で、DV証明書を無料で自動発行する仕組みを提供している。

```
Let's Encrypt の特徴:
  - 利用者の登録だけで誰でも利用できる
  - 証明書は90日ごとに自動更新される
  - ACMEというプロトコルを通じて発行と更新を自動化できる
```

Let's Encryptの登場以降、HTTPSを採用するサイトの割合は大きく伸びた。

```
2015年: HTTPS化されたサイトの割合は約40%
2024年: HTTPS化されたサイトの割合は約99%
```

Let's Encryptの証明書はDV証明書に限られ、有効期間も90日と短い。
ただし自動更新の仕組みを前提に運用されているため、実際の運用で問題になることは少ない。
Let's Encryptが発行する証明書も、主要ブラウザの信頼ストアに登録された認証局によるものとして扱われる。

フィッシング対策という観点では、HTTPSの普及そのものよりも、前のページで扱ったDMARCのようなメール認証の仕組みのほうが効果が大きい。
HTTPSだけでは、正規の企業のサイトとフィッシングサイトを区別できない。

<Callout title="ホスティングサービスにおけるLet's Encryptの利用例">
Vercel、Netlify、Railwayといったホスティングサービスや、CloudflareのようなCDNの多くは、Let's Encryptを内部に統合しており、デプロイと同時にHTTPS証明書を自動的に取得し更新する。
</Callout>

## PKI：これらの要素を統合した信頼の基盤

ここまで見てきた認証局、証明書、検証の手順、失効情報の配布を、まとめて一つの仕組みとして捉えたものが**PKI**（Public Key Infrastructure、公開鍵基盤)である。

```
【PKIの構成要素】
- 認証局（CA）：証明書を発行し、管理する
- 利用者：証明書を取得し、自分の公開鍵を証明書として公開する
- 検証者（ブラウザなど）：受け取った証明書を検証する
- リポジトリ（CRL / OCSP）：失効情報を配布する
```

PKIという基盤があるからこそ、初対面のサーバーであっても、証明書チェーンをたどることで信頼できるかどうかを判定できる。
このモデルが、現在のインターネット上の暗号化された通信を支えている。

## ここで押さえること

- **公開鍵証明書（X.509)**は、ある公開鍵が確かにそのドメインのものであることを、認証局が署名によって保証する仕組みである
- **証明書チェーン**は、ルートCAから中間CA、サーバー証明書へと続く信頼の連鎖であり、信頼は上位から下位へ伝わる
- ブラウザの**信頼ストア**には、OSやブラウザがあらかじめ信頼するルートCAのリストが組み込まれている
- **ドメイン検証(DV)**はドメインの管理権限の確認だけで発行され、フィッシングサイトでも取得できる
- **企業認証(OV)**と**拡張認証(EV)**は企業の実在性まで確認するが、発行には時間がかかる
- 「HTTPSであること」は「安全であること」を意味しない。企業の実在性を判断するには、証明書の種類とドメイン名そのものの確認が必要になる
- 証明書の失効情報は**CRL**(一覧形式)と**OCSP**(リアルタイムの問い合わせ)で配布される
- **Let's Encrypt**は無料でDV証明書を自動発行する仕組みを提供し、HTTPSの普及を大きく後押しした
- **PKI**は、認証局と証明書、検証者、失効情報の配布を組み合わせた、信頼を支える基盤全体を指す
