## 「ブラウザはなぜ初めてのサーバーを信頼できるのか」

あなたが Google.com にアクセスします。今まで訪れたことのない新しいデバイス、新しいネットワークから。

ブラウザは `https://` で接続します。でもそのサーバーが「本当に Google か」、どうやって確認しているんでしょう？

ハッカーが HTTPS サイトを作ることは簡単です。SSL 証明書だって取得できます。（Let's Encrypt が無料で配布してるから）

では、フィッシングサイトとGoogle の本サイトの違いは何か。

答えは **証明書チェーンの信頼性**。ブラウザが OS から受け継いだ「信頼できる認証局（CA）のリスト」を使って、サーバーの証明書が本物か検証するんです。

じつは、この仕組みの複雑さにこそ、セキュリティの本質が詰まっています。

## 公開鍵証明書（X.509）：「この公開鍵はたしかにこのドメインのもの」

公開鍵暗号の問題点：

```
あなた: 「Google の公開鍵をください」
ハッカー: 「これが Google の公開鍵です」←嘘

あなた: ハッカーの公開鍵で暗号化して送信
  ↓
ハッカー: あなたのデータを復号できる
```

暗号化のはずが、ハッカーに盗まれてしまいます。

解決策：**「この公開鍵は本当に Google のものですよ」という証明を、信頼できる第三者が保証する。**

これが **公開鍵証明書**（X.509形式）です。

```
【証明書の内容】
ドメイン: google.com
公開鍵: [Google の公開鍵]
発行者: Google Trust Services LLC
有効期間: 2024/06/01 ～ 2025/06/01
署名: [発行者の秘密鍵で署名]
```

重要なのは最後の「署名」。発行者（ Google Trust Services）が秘密鍵で署名しています。

これなら：

```
受信側が検証:
  1. 証明書に書かれた公開鍵（Google のもの）を抽出
  2. 発行者の公開鍵で、証明書の署名を検証
  3. 「本当に Google Trust Services が署名したか」を確認
     YES → このドメインの公開鍵は本物
     NO → 誰かが改ざんした。信用するな
```

ただし「Google Trust Services の公開鍵は本物か？」という問題が残ります。

## 認証局（CA）と証明書チェーン

信頼の道筋を遡るため、**階層構造** が使われます。

```
【信頼チェーン】

ブラウザの信頼ストア（OS）
  ↓ 信頼している
ルート CA（例：Symantec Root CA）
  ↓ 署名を信頼
中間 CA（例：Google Trust Services）
  ↓ 署名を信頼
サーバー証明書（google.com の証明書）
```

プロセス：

```
1. ブラウザが google.com に接続
2. サーバーが「サーバー証明書」を提示
3. ブラウザが中間 CA の公開鍵で、サーバー証明書の署名を検証
   → OK なら「Google Trust Services が署名したのは本物」
4. ブラウザが次に、ルート CA の公開鍵で、中間 CA の証明書を検証
   → OK なら「Symantec が Google Trust Services を認可した」
5. ルート CA の公開鍵は、ブラウザの信頼ストアに事前インストール
   → つまり「OS が信頼している」
```

この「チェーン」があるから、**最初に訪れたサイトでも「本物か偽物か」判定できる**。

## ブラウザの信頼ストア：誰が信頼できるか

Windows、macOS、Linux、各ブラウザ（Chrome、Firefox、Safari）が、あらかじめ「信頼できるルート CA」のリストを持っています。

```
ブラウザの信頼ストア（信頼できるCA の公開鍵）:
  - Symantec Root CA
  - Comodo RSA CA
  - GoDaddy Secure CA
  - DigiCert Global Root CA
  ...（数百個）
```

新しい CA がこのリストに追加されるには、基準があります：

- セキュリティ体制が十分か
- 不正な証明書を発行していないか
- 監査を受けているか

一度信頼ストアから除外されたら、その CA が発行したすべての証明書が「信頼できない」と判定されます。

実例：2011年、オランダの CA である DigiNotar が中東のハッカーに侵害され、フィッシングサイト用の証明書を発行されました。その後、全ブラウザから信頼ストアから削除され、事実上営業不可能に。

<Callout title="CA の責任は重い">
CA が不正な証明書を発行すると、業界全体の信頼が揺らぐ。だからブラウザは CA に対して極めて厳格。不正が発覚したら、数時間でリストから削除されることもある。
</Callout>

## ドメイン検証（DV）vs 企業認証（OV）vs 拡張認証（EV）

証明書発行時に「どの程度の身元確認を行ったか」で、3つの種類があります。

**DV（Domain Validation）:**

CA が確認するのは「**ドメインのコントロール」だけ**。

```
あなたが domain.com 用の証明書を申請
CA: 「本当に domain.com のオーナーか？」
  → メール確認（domain.com@... 宛に確認メール）
  または DNS レコード追加（domain.com の DNS に証明ファイル置く）
CA: 「OK、確認できた」→ 証明書発行
```

確認: **数分～数時間で発行可能。** Let's Encrypt も DV 証明書です。

重要なポイント：**「ドメインが本当にその企業のか」は確認しない。** だからフィッシングサイトでも DV 証明書は取得できます。

```
悪意のハッカー: evil.example.com ドメインを所有
CA に DV 証明書申請
CA: evil.example.com のコントロール確認
  → OK
CA: evil.example.com 用の DV 証明書を発行

ブラウザ:
  URL: evil.example.com
  https://
  証明書: 有効な DV 証明書
  
ユーザー: 「https だし大丈夫」とクリック（フィッシング被害）
```

**OV（Organization Validation）:**

企業の身元まで確認します。

```
あなたが domain.com 用の証明書を申請（企業名：Example Corp）
CA: 「本当に Example Corp のオーナーか？」
  → 企業登記簿確認
  → 企業の住所・電話番号確認
  → WHOIS に企業情報が載ってるか確認
```

確認: **数日～数週間。** OV 証明書が付いたサイトは「実在する企業」の証拠。

**EV（Extended Validation）:**

最も厳格。法人登記、営業実績、第三者による監査など、複数の確認。

```
金融機関、大手企業が発行する証明書に使用
ブラウザのアドレスバーに企業名を表示
例: 【Google Inc.】https://google.com
```

現在、EV 証明書の価値は低下しており、多くのブラウザは EV と DV の表示を区別していません。

<Callout title="「https = 安全」は誤解">
HTTPS だからといって安全ではありません。フィッシングサイトも DV 証明書を取得できます。本来は「通信が暗号化されている」だけ。企業の信頼性を示すには OV 以上が必要。さらに確実には、正規のドメインか確認（アドレスバーをよく見る）が重要。
</Callout>

## 証明書の失効：CRL と OCSP

発行した証明書が「秘密鍵が漏洩した」「ドメイン管理を失った」などの理由で失効することがあります。

失効情報をどう配布するか：

**CRL（Certificate Revocation List）:**

失効した証明書のシリアル番号を一覧で配布。

```
失効一覧:
  シリアル: 1a2b3c4d... → 失効日時
  シリアル: 5e6f7g8h... → 失効日時
  ...
```

ブラウザは定期的に CRL をダウンロードして参照。欠点は**リストが大きくなりすぎて、配布効率が悪い**。

**OCSP（Online Certificate Status Protocol）:**

CA に「この証明書はまだ有効か？」をリアルタイムで問い合わせ。

```
ブラウザ: 「シリアル番号 1a2b3c4d の証明書は有効ですか？」
CA サーバー: 「はい、有効です」
ブラウザ: サイトへ接続
```

リアルタイムだから正確ですが、**CA サーバーへの問い合わせが遅延することがある**。

最近は **OCSP Stapling** という最適化手法が使われます。サーバー側が CA に定期的に OCSP レスポンスを取得して、クライアントに「証明書は失効していません」という CA 署名付きの情報をあらかじめ返す。

## Let's Encrypt と証明書の民主化

2015年以前、SSL 証明書は「年数千円～数万円の有償」でした。だから小規模サイトは HTTPS を導入しづらかった。

**Let's Encrypt** は非営利団体で、**無料で DV 証明書を自動発行**。

```
Let's Encrypt:
  - 誰でも無料
  - 自動更新（90日ごと）
  - API 経由で自動化可能
  - ACME プロトコル（自動化標準）
```

Let's Encrypt の登場で、HTTPS 普及が爆発的に進みました。

```
2015 年: HTTPS サイトの割合 ~40%
2024 年: HTTPS サイトの割合 ~99%
```

デメリットは「DV 証明書のみ」＋「90日有効期限」。ただし自動更新が前提なので、実運用では問題になりません。

一方、Let's Encrypt も信頼ストアに登録されており、ブラウザは「Let's Encrypt の証明書 = 信頼できる」と判定します。

フィッシング対策に関しては、**DMARC（前の章で説明）などメール認証の方が効果的**。HTTPS だけでは区別できません。

<Callout title="Let's Encrypt の利用例">
多くの実行ホスティング（Vercel、Netlify、Railway など）や CDN（Cloudflare）は、Let's Encrypt を統合して、デプロイと同時に HTTPS 証明書を自動取得・更新しています。
</Callout>

## PKI（公開鍵基盤）：信頼を支える全体システム

ここまでの要素を統合したシステムが **PKI（Public Key Infrastructure）** です。

```
【PKI の構成】
- 認証局（CA）：証明書を発行・管理
- 利用者：証明書を取得して公開
- 検証者（ブラウザなど）：証明書を検証
- リポジトリ（CRL / OCSP）：失効情報を配布
```

PKI があるから、初対面のサーバーも信頼できる——このモデルが現代の Internet の基礎です。

## ここで押さえること

- **公開鍵証明書（X.509）**：「この公開鍵はたしかにこのドメインのもの」を認証局が保証
- **証明書チェーン**：ルート CA → 中間 CA → サーバー証明書。信頼は上から下へ
- **ブラウザの信頼ストア**：OS が信頼する CA のリスト。事前インストール
- **ドメイン検証（DV）**：ドメイン所有の確認だけ。数分で発行。フィッシングサイトも取得可能
- **企業認証（OV）/拡張認証（EV）**：企業の身元まで確認。時間がかかるが信頼性が高い
- **「HTTPS = 安全」は誤解**。DV 証明書でも有効。企業信頼性を判定するには正規ドメイン確認が必要
- **失効管理**：CRL（一覧）と OCSP（リアルタイム問い合わせ）で失効情報を配布
- **Let's Encrypt**：無料で DV 証明書を自動発行。HTTPS 普及を加速
- **PKI**：認証局・証明書・検証者が組み合わさった信頼インフラ
