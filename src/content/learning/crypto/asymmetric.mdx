## 「鍵を渡さずに秘密を共有できる」

このタイトルを初めて聞いたとき、多くの人が「えっどういうことだ？」となります。

鍵を渡さずに？秘密を？

共通鍵暗号の鍵配送問題をどう解くのか——1970年代の暗号学者たちもこれが不可能だと思っていました。

でも、**数学の巧妙さ** がこれを可能にしました。

## 公開鍵と秘密鍵のペア

公開鍵暗号の仕組みは、「2つの関連した鍵」を使います。

**公開鍵**：名前の通り「公開してOK」。世界中に配布できる。
**秘密鍵**：絶対に秘密。本人だけが持つ。

興味深いのは、この2つは数学的に結びついていて：

- **公開鍵で暗号化したデータは、秘密鍵でのみ復号できる**
- **秘密鍵で署名したデータは、公開鍵で検証できる**

初めて聞くと「えっ？秘密鍵で処理したら、そんなの秘密じゃないじゃん」と思うはずです。でも数学的には秘密鍵の「存在」を知らなくても、公開鍵で検証できる仕組みなんです。

```
ボブ（あなた）: 「私の公開鍵はこれです」→ 広く配布
アリス（友人）: 「この公開鍵でメッセージを暗号化」→ ボブへ送信
ボブ: 「秘密鍵で復号」→ メッセージが読める
（中間で傍受した人）: 「暗号文は手に入ったが、秘密鍵がないので読めない」
```

## RSA：素因数分解の困難性に基づく

最も有名な公開鍵暗号が **RSA** です。1977年に発表されて以来、銀行のオンラインサービス、HTTPS、デジタル署名など、世界中で使われています。

RSA の安全性の基盤は「**素因数分解が難しい**」という数学的事実。

```
2つの大きな素数を選ぶ：
  p = 61
  q = 53

2つを掛ける：
  n = p × q = 3,233

公開鍵に n = 3,233 を含める
秘密鍵には p と q を含める
```

問題は「n = 3,233 を見て、p と q が何か求めよ」——小さい n なら手計算で因数分解できます。

でも n が 2,048ビット（600桁以上）になると、**現代のスーパーコンピュータでさえ、数百年かかってしまう**。

RSA-2048 が現在のセキュリティスタンダード。RSA-4096 はさらに安全ですが、計算負荷が大きいため、一般的には 2048 で十分とされています。

<Callout title="量子コンピュータの脅威">
量子コンピュータが実用化されると、素因数分解が劇的に速くなるという理論があります。そのため、「ポスト量子暗号」（格子ベース暗号など）の研究が進んでいます。現在、NISTは「量子耐性のあるアルゴリズム」の標準化を進めています。
</Callout>

## Diffie-Hellman 鍵交換：鍵を送らずに共通の秘密を作る

RSA が「一方向的な暗号化」なら、**Diffie-Hellman（DH）** は「鍵を送らずに共通の秘密を作る」アルゴリズム。

想像してください。

アリスとボブが、盗聴者イブに傍受される通信路を通じて「共通の秘密」を作りたい。

**DH の手順：**

```
1. アリスとボブが「公開情報」を事前に決める（素数 p と g）
2. アリスが秘密の数 a を選ぶ（誰にも教えない）
3. ボブが秘密の数 b を選ぶ（誰にも教えない）
4. アリスが g^a mod p を計算して、ボブへ送信
5. ボブが g^b mod p を計算して、アリスへ送信
6. （イブは傍受したが、g^a と g^b から a や b は求められない）
7. アリスが (g^b)^a mod p を計算
8. ボブが (g^a)^b mod p を計算
9. 数学的に：(g^b)^a = (g^a)^b = g^(ab) mod p
10. アリスとボブだけが共通の秘密 g^(ab) mod p を知っている
```

このアルゴリズムの秀逸な点は「**一度も秘密の値 a や b を送信していない**」のに、2人だけが共通の秘密を作られる。

イブが傍受できるのは g^a と g^b だけ。これから a や b を求めるのは、**離散対数問題**という計算上困難な問題。数百年かかります。

HTTPS ハンドシェイク（ TLS 1.3）でも DH が使われます。クライアント（あなた）とサーバーが鍵を送信せず、鍵交換後は共通鍵暗号で通信する——その仕組みの中心にあるのが DH です。

## 楕円曲線暗号（ECC）へのちょっと先の話

RSA や DH は「大きな数の計算」に基づいています。

最近は **楕円曲線暗号（ECC）** が注目されています。

楕円曲線とは：

```
y^2 = x^3 + ax + b
```

という式で表される曲線上の「点の演算」を使う暗号。

ECC の利点：

- RSA と同等のセキュリティを、**はるかに短い鍵長** で実現
  - ECC-256 ≈ RSA-3072 程度のセキュリティ
- 計算が高速で、スマートフォンやIoTデバイスに向いている
- TLS 1.3、暗号通貨（Bitcoin/Ethereum）で採用されている

実は、ECDH（楕円曲線版の Diffie-Hellman）が TLS 1.3 のデフォルト鍵交換方式。あなたが HTTPS でブラウジングするたびに、楕円曲線が活躍しているんです。

<Callout title="楕円曲線の種類">
複数の楕円曲線が標準化されています。P-256（secp256r1）が HTTPS では一般的。Ed25519 は署名用として採用が広がっている。
</Callout>

## 公開鍵暗号と TLS：Web通信を支える基盤

HTTPS では「鍵交換」と「秘密通信」を組み合わせたハイブリッド方式が使われます。

現代の TLS 1.3 では、先ほど説明した ECDH で鍵交換を行い、その後の通信は高速な共通鍵暗号（AES-GCM）で行います。

```
TLS 1.3 の流れ（簡略）：
1. クライアント → サーバー: 「こんにちは + 自分のECDH公開値」
2. サーバー → クライアント: 「自分のECDH公開値 + 証明書」
3. 双方が ECDH で共通の秘密を独立に計算（鍵そのものは送らない）
4. 以降：その共通秘密から導出した AES-GCM 鍵で暗号化・復号
```

旧来の TLS 1.2（RSA 鍵交換）では、クライアントが公開鍵で共通鍵を暗号化して送るという方式でした。TLS 1.3 はこれを廃止し、常に ECDH を使います。

なぜ公開鍵暗号だけで全部やらないのか？

- **公開鍵暗号は遅い**（RSA-2048 の処理は AES の 1000 倍以上遅い）
- だから「鍵交換」だけに公開鍵暗号を使い、大量のデータは高速な AES-GCM で暗号化するハイブリッド方式になっている

## ここで押さえること

- **公開鍵暗号**は2つの関連した鍵を使う：公開鍵（広く配布）と秘密鍵（秘密）
- **RSA** は「素因数分解が困難」という数学的事実に基づく。RSA-2048 が現在の標準
- **鍵交換**（DHやECDH）で「鍵を送らずに共通の秘密を作る」——これが HTTPS のハンドシェイク
- **楕円曲線暗号（ECC）** は RSA より短い鍵長で同等のセキュリティを実現。スマートフォンに向いている
- **HTTPS は公開鍵と共通鍵を組み合わせたハイブリッド**。鍵交換は公開鍵暗号、データ転送は高速な AES-GCM
