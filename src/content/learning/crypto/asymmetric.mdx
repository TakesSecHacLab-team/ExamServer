## 鍵を渡さずに秘密を共有できるという発想

共通鍵暗号には鍵配送問題があった。
送り手と受け手が同じ鍵を持つ必要があり、その鍵を安全に渡す方法そのものが課題になっていた。

「鍵を渡さずに秘密を共有する」という言い方は、一見すると矛盾している。
渡さなければ共有できないはずだからである。

1970年代の暗号学者たちも、当初はこれが不可能だと考えていた。
それでも数学的な工夫によって、鍵そのものを送信せずに、送り手と受け手だけが知る秘密を作り出す方法が見つかった。
この方法が**公開鍵暗号**である。

## 公開鍵と秘密鍵という二つの鍵

公開鍵暗号では、一人につき関連した二つの鍵を用意する。

**公開鍵**は誰に渡しても問題のない鍵で、広く配布できる。
**秘密鍵**は本人だけが持つ鍵で、決して他人に渡してはならない。

この二つの鍵は数学的な関係で結びついている。

- 公開鍵で暗号化したデータは、対応する秘密鍵でのみ復号できる
- 秘密鍵で署名したデータは、対応する公開鍵で検証できる

公開鍵で暗号化した文書を、暗号化に使った公開鍵自身で復号することはできない。
復号できるのは、対になっている秘密鍵だけである。
ここに公開鍵暗号の核がある。

```
受け手: 「私の公開鍵はこれです」→ 広く配布
送り手: 「この公開鍵でメッセージを暗号化」→ 受け手へ送信
受け手: 「秘密鍵で復号」→ メッセージが読める
（通信を傍受した第三者）: 「暗号文は手に入ったが、秘密鍵がないので読めない」
```

公開鍵を盗み見られても困らない。
暗号文を作れるのは公開鍵を持つ誰でもだが、それを元のメッセージに戻せるのは秘密鍵を持つ本人だけである。

## RSA：素因数分解の難しさに基づく方式

公開鍵暗号の方式として最初に広く使われたのが**RSA**である。
1977年に発表されて以来、銀行のオンラインサービスやHTTPS、デジタル署名など、幅広い場面で使われている。

RSAの安全性は、ある数学的な事実に基づく。
二つの大きな素数を掛け合わせるのは簡単だが、その積から元の二つの素数を求める**素因数分解**は、数が大きくなるほど計算量が爆発的に増える。

```
2つの大きな素数を選ぶ：
  p = 61
  q = 53

2つを掛ける：
  n = p × q = 3,233

公開鍵に n = 3,233 を含める
秘密鍵には p と q を含める
```

3,233のような小さな数なら、61と53という組を手計算でも見つけられる。
ところがnが2,048ビット（10進数で600桁以上）になると、現在知られているどの計算機を使っても、素因数分解には数百年単位の時間がかかる。

RSA-2048（鍵長2,048ビットのRSA）が現在の標準的な強度とされる。
RSA-4096はさらに強度が高いが、計算負荷も大きくなるため、用途によって2,048で十分とされる場合が多い。

<Callout title="量子コンピュータが素因数分解を変える可能性">
量子コンピュータが実用化されれば、ショアのアルゴリズムという手法によって素因数分解が現在よりはるかに高速になるという理論がある。
そのため、量子コンピュータでも解読が難しいとされる方式（格子問題などを基盤にする「ポスト量子暗号」）の研究が進んでいる。
NISTはすでに、量子耐性を持つアルゴリズムの標準化作業を進めている。
</Callout>

## Diffie-Hellman鍵交換：鍵を送らずに共通の秘密を作る

RSAは「公開鍵で暗号化し、秘密鍵で復号する」という一方向の処理を提供する方式である。
これとは別の発想で、通信する二者が鍵そのものを一度も送らずに、共通の秘密の値を作り出す方法がある。
この方法を**Diffie-Hellman鍵交換**（DH）と呼ぶ。

アリスとボブが、盗聴者イブに通信内容を見られる経路を使って、共通の秘密を作りたいとする。
DHの手順は次のとおりである。

```
1. アリスとボブが「公開情報」を事前に決める（素数 p と生成元 g）
2. アリスが秘密の数 a を選ぶ（誰にも教えない）
3. ボブが秘密の数 b を選ぶ（誰にも教えない）
4. アリスが g^a mod p を計算して、ボブへ送信
5. ボブが g^b mod p を計算して、アリスへ送信
6. （イブは g^a と g^b を傍受できるが、そこから a や b を求めることはできない）
7. アリスが (g^b)^a mod p を計算する
8. ボブが (g^a)^b mod p を計算する
9. 数学的に (g^b)^a と (g^a)^b は同じ値 g^(ab) mod p になる
10. アリスとボブだけが、共通の秘密 g^(ab) mod p を知っている状態になる
```

`mod`は剰余演算であり、ある数を別の数で割った余りを取る計算を指す。
DHの安全性は、g^aとg^bの値を知っていても、そこから指数であるaやbを逆算するには現実的な時間で終わらない計算量が必要という事実(**離散対数問題**)に基づく。

この手順の特徴は、aやbという秘密の数そのものを一度も送信していない点にある。
イブが傍受できるのはg^aとg^bだけであり、そこから元の指数を求めるには、現在の計算機では数百年単位の時間がかかる。

HTTPSの接続を確立する手順（TLSハンドシェイク）でも、DHの考え方が使われている。
クライアントとサーバーは鍵そのものを送信せず、鍵交換が終わった後の通信は共通鍵暗号で行う。
この鍵交換の中心にあるのがDHである。

## 楕円曲線暗号という効率の良い方式

RSAとDHはいずれも、大きな整数の掛け算や指数計算を基盤にしている。
これとは別に、**楕円曲線暗号**（ECC、Elliptic Curve Cryptography）という方式がある。

楕円曲線とは、次の式で表される曲線である。

```
y^2 = x^3 + ax + b
```

ECCは、この曲線上の点同士の演算を使って公開鍵暗号を構成する方式である。
ECCには次の利点がある。

- RSAと同等の安全性を、より短い鍵長で実現できる（ECC-256はRSA-3072程度の安全性に相当する）
- 計算量が少なく、スマートフォンやIoT機器のような計算資源が限られた環境に向いている
- TLS 1.3や、BitcoinやEthereumなどの暗号資産で採用されている

DHの楕円曲線版を**ECDH**と呼び、TLS 1.3では鍵交換の標準的な方式として使われている。
ブラウザでHTTPSのサイトを開くたびに、背後でECDHが動いている。

<Callout title="楕円曲線にも複数の種類がある">
標準化されている楕円曲線は一つではない。
HTTPSではP-256（secp256r1という曲線）が一般的に使われ、署名用にはEd25519という曲線の採用が広がっている。
</Callout>

## TLSにおける公開鍵暗号と共通鍵暗号の組み合わせ

HTTPSでは、鍵交換と実際の通信の暗号化に、それぞれ異なる方式を組み合わせて使う。

TLS 1.3では、先に説明したECDHで鍵交換を行い、その後の実際の通信は高速な共通鍵暗号（AES-GCM)で暗号化する。

```
TLS 1.3 の流れ（簡略）：
1. クライアント → サーバー: 「こんにちは + 自分のECDH公開値」
2. サーバー → クライアント: 「自分のECDH公開値 + 証明書」
3. 双方がECDHで共通の秘密を独立に計算する（鍵そのものは送らない）
4. 以降は、その共通秘密から導出したAES-GCM鍵で暗号化し復号する
```

旧来のTLS 1.2では、クライアントが公開鍵で共通鍵を暗号化してサーバーに送る方式（RSA鍵交換)も使われていた。
TLS 1.3はこの方式を廃止し、常にECDHを使う。

なぜ通信全体を公開鍵暗号だけで処理しないのか。
RSA-2048による暗号化と復号は、AESによる処理と比べて1,000倍以上遅い。
鍵交換という一度限りの処理にだけ公開鍵暗号を使い、その後の大量データのやり取りには高速な共通鍵暗号を使う構成のほうが、全体として効率がよい。

## ここで押さえること

- **公開鍵暗号**は公開鍵(配布用)と秘密鍵(本人専用)という関連した二つの鍵を使う
- **RSA**は素因数分解の困難さに基づく方式で、RSA-2048が現在の標準的な強度である
- **Diffie-Hellman鍵交換**(DHやECDH)は、鍵そのものを送らずに共通の秘密を作る手法で、HTTPSのハンドシェイクの中心にある
- **楕円曲線暗号(ECC)**はRSAより短い鍵長で同等の安全性を実現し、スマートフォンなどの計算資源が限られた環境に向く
- HTTPSは公開鍵暗号と共通鍵暗号を組み合わせたハイブリッド方式である。鍵交換に公開鍵暗号、データ転送に高速な共通鍵暗号を使う
