## パスワードが流出しても被害が違う理由

データベースの流出事件は珍しくない。
2012年のLinkedInの流出、2016年のAdobeの流出など、大規模な利用者情報の漏洩がこれまでに何度も起きている。
報道では「パスワードはハッシュ化されて保存されていた」と説明される場合と、「平文のまま保存されていた」と説明される場合がある。

平文のパスワード一覧を手に入れた攻撃者は、その場でログインを試せる。
一方、ハッシュ化されたパスワードを手に入れた攻撃者は、その値を見ても元のパスワードがわからない。
同じ「流出」という出来事でも、保存方式によって被害の深刻さが変わる。

この差を理解するには、「ハッシュ化」と「暗号化」が別の処理であることを知る必要がある。

## ハッシュ関数とは何をする関数か

**ハッシュ関数**は、任意の長さの入力を受け取り、固定された長さの出力に変換する関数である。
この出力を**ハッシュ値**と呼ぶ。

```
入力: "password123"（11字）
    ↓ SHA-256 ハッシュ関数
出力: "ef92b778bafe771e89245d171bafbaa6...（64字の16進数）"
```

同じ入力を与えると、必ず同じ出力が得られる。

```
入力: "password123"
出力: ef92b778bafe771e89245d171bafbaa6...（毎回同じ）

入力: "password124"（最後の数字だけが違う）
出力: a4e1a5b5a82a...（まったく別の出力）
```

暗号化と決定的に違う点が一つある。
暗号化には、正しい鍵を使って元のデータに戻す（復号する)手順が存在する。
ハッシュ関数には、ハッシュ値から元の入力を復元する手順が存在しない。
これがハッシュ関数を理解する出発点になる。

ハッシュ化と暗号化を同じものだと考えている人は少なくない。
この区別を押さえておくと、後で見るパスワード保存の実装の妥当性を判断できるようになる。

## ハッシュ関数に求められる三つの性質

暗号学的に安全とされるハッシュ関数には、三つの性質が求められる。

**一方向性**（原像計算困難性)は、ハッシュ値から元の入力を逆算するのが計算上不可能であるという性質を指す。
`ef92b778...`というハッシュ値を見て、元の文字列`password123`を直接導き出す方法は存在しない。
確認する側にできるのは、`password123`という候補を実際に入力してみて、出力されたハッシュ値が一致するかを照合することだけである。
攻撃者は、候補を一つずつ試して当たりを引くまで繰り返すしかない。

**衝突耐性**（第二原像計算困難性)は、異なる二つの入力が同じハッシュ値を生成する事態が起きにくいという性質を指す。
異なる入力が同じハッシュ値を生成する事態を**衝突**と呼ぶ。

```
入力A: "password"
入力B: "drowssap"（まったく別の文字列）

もし同じハッシュ値が出力されたら、衝突が発生したことになる
```

衝突が起きやすいハッシュ関数では、無関係な入力同士が同じハッシュ値を持つ事態が現実的な計算量で発生し、ハッシュ値による照合が信頼できなくなる。

**雪崩効果**（アバランシェ効果)は、入力をわずかに変えるだけで、出力が大きく変わるという性質を指す。

```
入力: "password"
ハッシュ: 5baa61e4c9b93f3f0682250b6cf8331b7ee68fd8

入力: "password1"（文字を一つ追加しただけ)
ハッシュ: fcea920f7412b5da7be0cf42b8c93759de898e48
```

入力の差がわずかでも、出力は無関係な値同士に見えるほど変わる。
出力の一部が似ているということは起きない。
この性質があるからこそ、ハッシュ値を使ってデータの改ざんを検知できる。

## MD5とSHA-1が「弱い」とされる理由

**MD5**と**SHA-1**は、1990年代から2000年代初頭にかけて広く使われたハッシュ関数である。
現在ではどちらも、安全性を求める用途には使うべきではないとされている。

その理由は、衝突を現実的な計算量で見つける手法が実際に発見されたことにある。
2004年に中国の研究者である王小雲のグループがMD5の衝突を発見し、2005年にはSHA-1についても同様の手法が示された。
これは「理論上は衝突が存在しうる」という指摘ではなく、実際に衝突する入力の組を計算機で作れることが示された結果である。

これにより、次のような事態が現実の脅威になる。

- 異なる二つのパスワードが同じMD5ハッシュ値を持つ組を、現実的な計算量で見つけられる
- 内容が異なる二つのファイルに、意図的に同じSHA-1ハッシュ値を持たせられる

Google Chromeは2017年、SHA-1で署名されたHTTPS証明書の信頼を打ち切った。
衝突を作る攻撃が現実的になった以上、証明書の正当性をSHA-1だけで保証できなくなったためである。

<Callout title="衝突を実際に作って示した実例">
2017年、GoogleはSHA-1の衝突を実際に作成して公開した。内容の異なる二つのPDFファイルが同じSHA-1ハッシュ値を持つという例である。これは理論上の脆弱性の指摘ではなく、攻撃が実行可能であることを示す実証だった。
</Callout>

## SHA-256とSHA-3という現在の標準

MD5やSHA-1に代わって使われているのが、SHA-2ファミリー（SHA-256、SHA-384、SHA-512を含む)とSHA-3である。

SHA-256は2001年から、SHA-3は2015年から標準として定められており、現時点では実用的な衝突は発見されていない。
ブロックチェーン、デジタル署名、TLS通信など、多くの重要なインフラがSHA-256に依存している。

ただし「安全」という評価は、「現時点で衝突が見つかっていない」という意味にとどまる。
計算能力が将来大きく向上すれば、SHA-256にも衝突の脅威が及ぶ可能性は残る。
このため暗号学者は、新しいハッシュ関数の研究を継続している。

## パスワード保存には計算コストの高いハッシュ関数を使う

通常のハッシュ関数（SHA-256など)は、高速に計算できるように設計されている。

```
"password" → SHA-256計算 → 0.00001秒で完了
```

この速さは、パスワードを保存する用途では不利に働く。
よく使われるパスワードのリストを使い、各候補のハッシュ値を計算して、流出したハッシュ値と照合する**辞書攻撃**という手法では、計算が高速であるほど、攻撃者が大量の候補を短時間で試せてしまう。

そのため、パスワード保存専用に設計された、意図的に計算が遅いハッシュ関数が使われる。
**bcrypt**、**scrypt**、**Argon2**がその代表である。

bcryptの計算は次のような形を取る。

```
bcrypt("password", salt, cost=12)
→ 計算に意図的に0.1秒前後かかるように設計されている
```

`cost`は反復計算の回数を指定するパラメータで、値が大きいほど一回の計算に時間がかかる。

- cost=10：約100ミリ秒
- cost=12：約250ミリ秒
- cost=14：約1秒

cost=12の設定では、100万個のパスワード候補を試すのに250秒(4分以上)かかる。
SHA-256でそのまま保存している場合と比べると、攻撃者が同じ時間内に試せる候補の数が大きく変わる。

**Argon2**はbcryptをさらに発展させた方式で、計算時間に加えてメモリ使用量も増やす設計になっている。

```
Argon2("password", salt, time=2, memory=65MB)
```

メモリを大量に使う設計には理由がある。
CPUの数を増やして並列に攻撃を試みるのは比較的容易だが、メモリの容量を同じように増やすのは難しい。
このため、メモリ使用量の大きいハッシュ関数は、並列攻撃への耐性が高い。
セキュリティの実践基準を公開しているOWASP（Open Web Application Security Project)は、パスワード保存にArgon2を第一の選択肢として推奨している。

## レインボーテーブル攻撃とソルトによる対策

流出したパスワードのリストがすべてハッシュ化されていても、攻撃者には別の手段が残っている。

**レインボーテーブル**は、よく使われるパスワードとそのハッシュ値の対応をあらかじめ計算しておいた表である。

```
よくあるパスワード10億個のリストを用意する
各パスワードのMD5ハッシュ値を計算する
結果を表にして保存する（数百GB規模になる）

流出したハッシュ値をこの表で検索する
一致すれば、そのユーザーのパスワードが判明する
```

この表を作る作業には時間がかかるが、一度作れば何度でも再利用できる。
インターネット上には、こうした表が無料で配布されている例もある。

この攻撃への対策が**ソルト**である。
ソルトは、各パスワードのハッシュ値を計算する前に付け加えるランダムな値を指す。

```
ユーザーAのパスワード: "password"
生成されたソルト: "a7f3b2c1"
保存するハッシュ: SHA-256("password" + "a7f3b2c1")

ユーザーBのパスワード: "password"（同じ）
生成されたソルト: "9d2e5k8m"
保存するハッシュ: SHA-256("password" + "9d2e5k8m")
```

同じパスワードでも、ソルトが異なれば出力されるハッシュ値も異なる。
攻撃者は利用者ごとに異なるソルトを織り込んだ表を新たに作る必要があり、事前に計算しておいたレインボーテーブルが使えなくなる。

bcryptとArgon2は、ソルトの生成と保存を内部で自動的に処理する。
実装者が別途ソルトを管理する処理を書く必要はない。

<Callout title="パスワード保存の現在の指針">
パスワードの保存にはArgon2を使うのが現在の標準的な指針である。bcryptも十分な安全性を持つが、Argon2のほうが並列攻撃への耐性で優れる。MD5やSHA-1、素のSHA-256をパスワード保存にそのまま使ってはならない。
</Callout>

## ファイルの改ざん検知への応用

ハッシュ関数の使い道はパスワード保存だけではない。
ファイルが配布の途中で改ざんされていないかを確認する用途にも使われる。

OSのインストーラーを配布する公式サイトには、「SHA-256: 3c4f...」のような値が記載されている場合がある。

```
ダウンロードしたファイル → SHA-256計算
→ 出力: 3c4f...
→ 公式サイトに記載された値と一致するか確認
   一致する → ファイルは改ざんされていない
```

攻撃者がファイルの内容を改変すると、ハッシュ値も変わる。
雪崩効果により、わずかな改変でもハッシュ値はまったく別の値になるため、「内容だけ変えてハッシュ値は変えない」という操作は成立しない。

Gitのコミットを識別するID(コミットハッシュ)も、SHA-1によるハッシュ値である。
あるコミットの内容が改変されていないかは、そのコミットのSHA-1を再計算して確認できる。

## ここで押さえること

- **ハッシュ関数**は任意長の入力を固定長の出力に変換する関数で、出力から入力を復元する手順は存在しない（一方向性）
- 暗号化には鍵を使った復号の手順があるが、ハッシュ関数にはその手順がない
- ハッシュ関数に求められる性質は**一方向性**、**衝突耐性**、**雪崩効果**の三つである
- MD5とSHA-1は衝突を作る手法が実証済みのため、安全性を求める用途では使わない。SHA-256とSHA-3が現在の標準である
- パスワード保存には**bcrypt**や**Argon2**のような、意図的に計算が遅いハッシュ関数を使う。通常のハッシュ関数は高速すぎて辞書攻撃に弱い
- **ソルト**は、パスワードごとに異なるランダムな値を加えることで、事前に計算したレインボーテーブルを無効化する
- ハッシュ関数はファイルの改ざん検知にも使える。配布元が公開したハッシュ値と再計算した値が一致すれば、内容が改ざんされていない証拠になる
