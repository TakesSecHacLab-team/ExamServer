## 「パスワードが盗まれても、ハッシュ化されてれば安全？」

2012年のLinkedIn漏洩、2016年のAdobe漏洩——大規模なデータベース流出が頻繁に起きました。ニュースで「パスワードがハッシュ化されていた」と報じられた場合と「平文で保存されていた」と報じられた場合がある。

その差は何か。プレーンテキストのパスワードリストを受け取ったハッカーがすぐに使えるのと、暗号化された(はずの)パスワードを前にして「これ何だ?」となるのでは、被害の深刻さが全く違います。

まず「ハッシュ化」と「暗号化」の違いから解きほぐします。

## ハッシュ関数とは

ハッシュ関数は「任意の長さのデータを受け取って、固定の長さの数字に変換する」という仕事をします。

```
入力: "password123"（11字）
    ↓ SHA-256 ハッシュ関数
出力: "ef92b778bafe771e89245d171bafbaa6...（64字の16進数）"
```

同じ入力には必ず同じ出力が出ます。

```
入力: "password123"
出力: ef92b778bafe771e89245d171bafbaa6...（毎回同じ）

入力: "password124"（最後の数字が違う）
出力: a4e1a5b5a82a...（全く別の出力）
```

重要なポイント：**正しい鍵があれば元に戻す方法が存在するのが暗号化。ハッシュには元に戻す方法が存在しない。** これが暗号化との決定的な違いです。

実際、「ハッシュ化=暗号化」と混同しているケースは非常に多いです。でもこの違いが分かると、実装の安全さがガラリと変わります。

## ハッシュの3つの特性

暗号学的に安全なハッシュ関数には、3つの性質が求められます。

**1. 一方向性（原像計算困難性）**

与えられたハッシュ値から、元の入力を逆算で導き出すのが計算上不可能。

「ef92b778... から元の文字列 'password123' を復元できるか」——できません。最初から 'password123' を入力して、その出力が 'ef92b778...' と一致するかを確認するしかない。つまりハッカーは「あたり」を引くまで試し続けるしかない。

**2. 衝突耐性（第二原像計算困難性）**

異なる2つの入力が同じハッシュ値を作り出すのが極めて難しい（これを「衝突」と呼ぶ）。

```
入力A: "password"
入力B: "drowssap"（全く別の文字列）

もし同じハッシュが出力されたら → 衝突が発生した
```

衝突が起きやすいハッシュ関数は、全く無関係なパスワードでも同じハッシュが出てしまい、セキュリティが崩壊します。

**3. 雪崩効果（アバランシェ効果）**

入力をほんの少し変えるだけで、出力が劇的に変わる。

```
入力: "password"
ハッシュ: 5baa61e4c9b93f3f0682250b6cf8331b7ee68fd8

入力: "password1"（文字を1つ追加）
ハッシュ: fcea920f7412b5da7be0cf42b8c93759de898e48
```

出力の一部が同じだったり、「少し似てる」ことはない。全く別の値になる。これがあるとデータの整合性チェックに使える。

## MD5、SHA-1はなぜ「弱い」のか

ここが理解を深めるために大事なポイント。MD5とSHA-1は、1990年代から2000年代初頭に広く使われてきたハッシュ関数です。

でも「弱い」と言われるのは、**衝突が現実的な時間・計算量で発見できてしまった** から。

2004年、中国の密码学者・王小雲がMD5の衝突を発見しました。2005年にはSHA-1の衝突も発見されました。これらは「理論的には衝突があり得る」という話じゃなく、**実際にハッカーが衝突を作れる** ということ。

つまり：

- 異なるパスワード「password」と「drowssap」が同じMD5ハッシュを生成するケースを、現実的な計算量で見つけられる
- PDFファイルAとファイルBの内容は全く違うのに、意図的に同じSHA-1ハッシュを作ることができる

Google Chromeは2017年にSHA-1を使った署名付きHTTPS証明書を信頼しなくなった。理由は「衝突されるリスクが許容できないレベル」だから。

<Callout title="衝突の実例：Google Shattered Project">
2017年、GoogleはSHA-1の衝突を実際に作成してみせました。異なる2つのPDFファイルが同じSHA-1ハッシュを持つ例を公開。これは「理論的な脆弱性」ではなく「実世界での攻撃が可能」を証明した。
</Callout>

## SHA-256、SHA-3の今

MD5やSHA-1に代わるハッシュ関数として、SHA-2ファミリー（SHA-256、SHA-384、SHA-512）とSHA-3が使われています。

SHA-256は2001年から、SHA-3は2015年から標準として定められ、今のところ実用的な衝突は発見されていません。ブロックチェーン、デジタル署名、TLS通信など、重要なインフラがSHA-256に依存しています。

ただし「安全」というのは「今のところ衝突が見つかっていない」という意味。将来的に計算能力が飛躍的に上がれば、SHA-256にも衝突の脅威が出てくる可能性はあります。だから暗号学者は新しいハッシュ関数の研究を続けているんです。

## パスワードハッシング：計算コストを意図的に上げる

注意点がある。通常のハッシュ関数（SHA-256など）は、**とても高速に計算される** ように設計されています。

つまり：

```
"password" → SHA-256計算 → 0.00001秒で完了
```

ハッカーにとっては、この速さが有利に働きます。辞書攻撃——「よくあるパスワード1000個」のリストから、各パスワードのSHA-256を計算して、盗まれたハッシュと照らし合わせる——これが一瞬で終わってしまいます。

だから **パスワード保存専用のハッシュ関数** が必要とされました。bcrypt、scrypt、Argon2 などです。

**bcrypt の仕組み：**

```
bcrypt("password", salt, cost=12)
→ 計算に意図的に0.1秒かかるように設計
```

`cost` というパラメータで「何回反復計算するか」を指定でき、 cost が高いほど計算に時間がかかります。

- cost=10：約100ミリ秒
- cost=12：約250ミリ秒
- cost=14：約1秒

ハッカーが100万個のパスワードを試し終わるのに、bcost=12では250秒（4分以上）かかります。bcrypt が普及しているサービスと、SHA-256でパスワードを保存している脆弱なサービスでは、攻撃の実行可能性が全く異なるんです。

**Argon2 はさらに進化：**

bcrypt は時間計算量を増やしますが、Argon2 は **メモリ使用量も増やす**。

```
Argon2("password", salt, time=2, memory=65MB)
```

メモリを大量に使うと、ハッカーが並列攻撃（複数PCで同時に試す）をやりづらくなります。メモリは CPUほど簡単に増設できないから。OWASP（セキュリティのベストプラクティス機関）は Argon2 を「パスワード保存の最優先選択肢」と推奨しています。



## レインボーテーブル攻撃とソルト

「パスワードリストが手に入った。でも全部ハッシュ化されてる...」

こんなときハッカーが使う裏技が **レインボーテーブル** です。

レインボーテーブルは、事前に計算された「よくあるパスワード → そのハッシュ値」の辞書。

```
よくあるパスワード10億個のリストを用意
各パスワードのMD5を計算
結果を表にして保存（数百GB）

盗まれたハッシュ値をこのテーブルで検索
→ 一致したら「このユーザーのパスワードは何か」がバレる
```

このテーブル作成は時間がかかりますが、一度作ってしまえば何度でも使えます。インターネットには「Rainbow Tables」として無料で配布されているものもあります。

対策が **ソルト** です。ソルトは「各パスワードに付け加えるランダムな数字」。

```
ユーザーAのパスワード: "password"
ランダムなソルト生成: "a7f3b2c1"
保存するハッシュ: SHA-256("password" + "a7f3b2c1")

ユーザーBのパスワード: "password"（同じ）
ランダムなソルト生成: "9d2e5k8m"
保存するハッシュ: SHA-256("password" + "9d2e5k8m")
```

同じパスワードでも、ソルトが違えば出力も違う。だから事前に計算されたレインボーテーブルが使えなくなります。

bcrypt と Argon2 はソルトの生成と保存を自動で処理します。実装者が特別な操作をしなくても、内部的にソルトが付加されます。

<Callout title="ハッシュストレージのベストプラクティス">
2024年時点で、パスワード保存には Argon2 が標準。bcrypt は「十分安全だが Argon2 ほどではない」というレベル。MD5/SHA-1/SHA-256をパスワード保存に使ってはいけない。
</Callout>

## ファイルの整合性確認への応用

ハッシュ関数が役立つのはパスワード保存だけではありません。**ファイルが改ざんされていないか** を確認するのに使われます。

OSのインストーラーをダウンロードするとき、公式サイトに「SHA-256: 3c4f...」と書いてあることがあります。

```
ダウンロードしたファイル → SHA-256計算
→ 出力: 3c4f...
→ 公式サイトの値と一致？ YES → ファイルは改ざんされていない
```

ハッカーがファイルの中身を改変すると、ハッシュ値も変わってしまいます。小さな改変も雪崩効果で全く別のハッシュになるので、「ここだけ改ざんしたけどハッシュは変えない」は不可能。

Git のコミット ID も実はハッシュ（SHA-1）です。「このコミットのコードが改ざんされたかどうか」は、SHA-1を再計算すれば検証できます。

## ここで押さえること

- **ハッシュ関数**は「任意長の入力→固定長の出力」で、**一方向**（元に戻せない）
- 暗号化は双方向（鍵があれば復号できるが、ハッシュは不可）
- ハッシュの3つの特性：**一方向性**、**衝突耐性**、**雪崩効果**
- **MD5/SHA-1は衝突が実証済み**のため、セキュリティ用途では使うな。SHA-256/SHA-3 が現在の標準
- **パスワード保存には専用の遅いハッシュ**（bcrypt/Argon2）を使う。通常のハッシュは高速すぎて辞書攻撃に弱い
- **ソルト**で「事前計算テーブル（レインボーテーブル）攻撃」を防ぐ
- ハッシュは「改ざん検知」にも使える。ファイルのハッシュが一致すれば、内容が改ざんされていない証拠
