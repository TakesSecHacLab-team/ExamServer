## 「同じ鍵でロックして、同じ鍵で開ける」

例えば、信頼できる友人に秘密のメッセージを送りたいとします。

「この内容は絶対に第三者には見せない」——そのために暗号化する。

共通鍵暗号の考え方は シンプルです。**あらかじめ、送り手と受け手で同じ鍵を持つ**。送り手がそれで「ロック」して、受け手が同じ鍵で「開ける」。

```
送り手: "Hello World" + 鍵 → 暗号化 → "X#$%@!K#..."
受け手: "X#$%@!K#..." + 鍵 → 復号 → "Hello World"
```

問題は1つ。**その鍵をどうやって相手に渡すか**。知らない相手に荷物を送りたいが、先に鍵を送る必要がある。でも「鍵の配送」が一番危ないんです。

ここまではシンプルですが、実装の細部に潜む落とし穴が、セキュリティを左右します。

## AES（Advanced Encryption Standard）の概要

共通鍵暗号の最現代的な標準が **AES** です。米国の NIST が2001年に採用して以来、世界中のシステムで使われています。

AES の基本的な特徴：

- **ブロック暗号**：データを固定サイズ（128ビット単位）に分割して、ブロックごとに暗号化
- **鍵長の選択肢**：128ビット、192ビット、256ビットの3種類。長いほど総当たり攻撃に強い
- **計算速度**：ハードウェア、ソフトウェアともに高速。実用的なデータ量でも暗号化・復号に秒単位かかることはない

```
平文: "This is a secret message..."
鍵: 256ビットのランダムな数列
AES-256 暗号化: "K3Xp9mL2Q8..."（暗号文）
```

では、なぜ AES が安全とされるのか。**現代のスーパーコンピュータでさえ、正しい鍵を総当たりで試し終わるのに、宇宙の年齢より長い時間がかかる** から。

AES-256 だと 2^256 通りの鍵の組み合わせがあります。これは約「100,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000,000 通り」。

## 暗号化モード：ECB、CBC、GCM

ここからが、実装の落とし穴の話です。

AES は「128ビット固定のブロック」を暗号化します。でもファイルサイズは 128ビットより大きい。だから複数ブロックを暗号化する必要があります。

「複数ブロックをどうやって繋ぎ合わせるか」——それが **暗号化モード** です。

### ECB（Electronic CodeBook）モード：最も危険

最もシンプルな方法：各ブロックを独立して暗号化する。

```
平文ブロック1: "ABCDEFGH" + 鍵 → "X#$%@!K#"
平文ブロック2: "ABCDEFGH" + 鍵 → "X#$%@!K#"（同じ平文なら同じ暗号文）
平文ブロック3: "IJKLMNOP" + 鍵 → "Q7Z9M2R$"
```

**問題：同じ平文が同じ暗号文になる。** つまり、元のファイルに「同じ部分」があれば、暗号化後も「同じ部分」が見える。

これを見える化したのが有名な例。**ペンギン画像**です。

ペンギンの写真をECBモードで暗号化すると、暗号文から「ペンギンの形」が浮かぶんです。色は変わるけど、構造は丸見け。

なぜなら「黒い部分」（ペンギン）は同じピクセル値が繰り返され、暗号化しても「同じペンギンの形をしたブロック」として認識できてしまうから。

ECB モードは業界では「教科書的な悪い例」としてしか語られません。実務では絶対に使ってはいけないモード。

<Callout title="ECB の悪名：Tux の幽霊">
Linux のマスコットキャラ「Tux」をECBで暗号化すると、暗号文の画像データからも「Tux の形」が見える。これは ECB の脆弱性を示すために使われる有名な例。
</Callout>

### CBC（Cipher Block Chaining）モード：より安全

前のブロックの暗号文を、次のブロックに「混ぜる」。

```
IV（初期化ベクトル）: ランダム値
ブロック1: 平文1 XOR IV + 鍵 → 暗号文1
ブロック2: 平文2 XOR 暗号文1 + 鍵 → 暗号文2
ブロック3: 平文3 XOR 暗号文2 + 鍵 → 暗号文3
```

同じ平文でも、前のブロックが異なれば出力が変わる。つまり ECB のような「パターン漏洩」がない。

ただし CBC にも弱点がある。**パディングオラクル攻撃** という手法で、暗号文から元の平文を推測される可能性がある（実装が悪いと）。

### GCM（Galois/Counter Mode）モード：現在の標準

CBC より優れた設計。データを暗号化しつつ、同時に **改ざん検知（認証タグ）** も付ける。

```
平文 + 鍵 + ノンス → 暗号文 + 認証タグ
```

GCM は「秘密を隠す」＆「改ざんされていないか確認する」を同時にやるので、実装の余地が少なく、よほどのことがない限り安全です。

HTTPS の TLS 1.3、パスワードマネージャー、ディスク暗号化（BitLocker）——最新のセキュリティ機構はほぼ GCM を使っています。

<Callout title="AES-256 vs AES-128：誤解を招く表現">
「AES-256 はAES-128 より2倍安全」と書く人がいますが、正確ではありません。確かに鍵長が長いほど総当たり攻撃に強いのは事実。ですが実務の脅威は総当たり（2^256 世代の試行）より、**実装ミス・鍵管理の問題・IV の再利用** などの方が圧倒的に多い。ECB モードのAES-256 より、GCM モードのAES-128 の方が実質的に安全です。
</Callout>



## 鍵配送問題：共通鍵暗号の致命的な矛盾

ここが、共通鍵暗号の「根本的な問題」です。

あなたがクレジットカード情報をオンラインショップに送りたい。その情報を暗号化したい。でも暗号化するには、事前にショップと「同じ鍵」を共有しておく必要があります。

その鍵をどうやって「安全に」ショップに届ける？

```
あなた: 「鍵を送るね」
  ↓ 鍵を何らかの方法で送信
  ↓（この過程で鍵が盗まれたら...）
ショップ: 「鍵を受け取った」
```

メールで送ってはいけない（メール経由で盗まれる）。電話で言う？（ハッカーが通話を傍受する）。郵便？（物理的に遮断できない）。

これが **鍵配送問題** です。1970年代から、暗号学者が苦労してきた根本的な問題。

解決策は **公開鍵暗号**。次の章で詳しく説明しますが、「鍵を送らずに秘密を作る」という革新的な手法です。

## ストリーム暗号とブロック暗号：軽く知っておく

参考までに、共通鍵暗号にはブロック暗号のほかにもう1つのカテゴリがあります。

**ブロック暗号**（AES など）：データを固定サイズに分割して処理

**ストリーム暗号**（ChaCha20、RC4 など）：データを1ビット（または1バイト）ずつ、連続して処理

ストリーム暗号は「一度に全データを読み込む必要がない」のでメモリ効率が良く、通信ストリームに向いています。TLS 1.3 では ChaCha20 がサポートされています。

実務的には「ブロック暗号か、モダンなストリーム暗号か」どちらか。古い実装（RC4など）は脆弱性が見つかっているため、使うべきではありません。

## ここで押さえること

- **共通鍵暗号**は「同じ鍵でロック・アンロック」。AES が現在の標準
- **ブロック暗号**として 128ビット単位でデータを処理。鍵長 256ビット推奨
- **暗号化モード**によって安全性が左右される：ECB は脆弱（パターン漏洩）、CBC は中程度、GCM（認証付き）が現在の標準
- **ECB で AES-256 を使っても、GCM で AES-128 使う方が安全**——実装が安全さを決める
- **鍵配送問題**：共通鍵をどう安全に相手に渡すか。この問題を解くのが次の公開鍵暗号
