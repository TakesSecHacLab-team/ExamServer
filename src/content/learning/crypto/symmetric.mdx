## 暗号化とは何をすることか

スマートフォンでメッセージアプリを開いて、友人に「今日の集合場所は駅前」と送る。

このメッセージは、自分のスマートフォンから友人のスマートフォンまで、いくつものネットワーク機器を経由して届く。
途中には、自宅のWi-Fiルーター、契約している通信会社の設備、インターネットの中継地点がある。
これらの機器を運用している人や、回線を流れる通信を盗み見ようとする第三者がいれば、メッセージの内容をそのまま読めてしまう。

**暗号化**とは、データを「鍵」と呼ばれる値を使って、意味のわからない見た目に変換する処理である。
変換後のデータを**暗号文**、変換前のデータを**平文**と呼ぶ。
暗号文を元の平文に戻す処理を**復号**と呼ぶ。

```
平文: "今日の集合場所は駅前"
  ↓ 鍵を使って暗号化
暗号文: "X#$%@!K#..."
```

この暗号文を盗聴者が手に入れても、鍵を持たなければ元のメッセージはわからない。
鍵を持つ友人だけが、暗号文を平文に戻せる。

暗号化の方式は大きく二種類に分かれる。
一つは、暗号化と復号に同じ鍵を使う方式で、これを**共通鍵暗号**と呼ぶ。
もう一つは、暗号化と復号に別の鍵を使う方式で、これを公開鍵暗号と呼ぶ。
このページでは共通鍵暗号を扱う。

## 共通鍵暗号の仕組み

共通鍵暗号では、送り手と受け手が、通信を始める前に同じ鍵をあらかじめ共有しておく。
送り手はその鍵でメッセージを暗号化し、受け手は同じ鍵で復号する。

```
送り手: "Hello World" + 鍵 → 暗号化 → "X#$%@!K#..."
受け手: "X#$%@!K#..." + 鍵 → 復号 → "Hello World"
```

鍵が玄関の鍵だとすると、送り手と受け手が同じ鍵を持っている状態に近い。
送り手が鍵をかけ、受け手が同じ鍵で開ける。

この方式には一つの前提がある。
送り手と受け手は、通信を始める前に、何らかの方法で同じ鍵を共有していなければならない。
その共有の方法に弱点があれば、暗号化そのものがどれだけ強くても意味を持たない。
盗聴者が鍵そのものを手に入れてしまえば、暗号文を平文に戻せてしまうからである。

この「鍵をどう安全に渡すか」という課題は、共通鍵暗号の構造そのものに由来する未解決の問題として後で扱う。
まずは、共通鍵暗号で実際に使われている暗号化の方式そのものを見ていく。

## AES：現在の標準的な共通鍵暗号

共通鍵暗号の方式は歴史の中でいくつも提案されてきたが、現在最も広く使われている方式が**AES**（Advanced Encryption Standard）である。
米国の標準化機関である**NIST**（National Institute of Standards and Technology、米国国立標準技術研究所）が2001年に標準として採用した。

AESには次の特徴がある。

- **ブロック暗号**である：データを128ビットという固定サイズの単位（**ブロック**と呼ぶ）に分割し、ブロックごとに暗号化する
- **鍵長**を128ビット、192ビット、256ビットの中から選べる：鍵長が長いほど、鍵の総当たりによる解読に強くなる
- 暗号化と復号の計算が高速である：専用のハードウェア支援がなくても、実用上は瞬時に終わる

```
平文: "This is a secret message..."
鍵: 256ビットのランダムな数列
AES-256 暗号化: "K3Xp9mL2Q8..."（暗号文）
```

AESが安全だとされる理由は、鍵の総当たり攻撃に必要な計算量が、現実的な時間内では終わらないという事実にある。
AES-256では鍵の組み合わせが2の256乗通りあり、これは10進数で78桁ほどの数になる。
現在知られているどの計算機を使っても、この組み合わせを総当たりで試し切るには、宇宙の年齢をはるかに超える時間がかかる。

## 暗号化モードという落とし穴

AESそのものは128ビット単位のブロックしか暗号化できない。
ところが実際に暗号化したいファイルは、写真でも文書でも、128ビットよりはるかに大きい。
そのため、複数のブロックを連続して暗号化する手順が必要になる。

この「複数ブロックをどう処理するか」を決める規則を**暗号化モード**と呼ぶ。
同じAESを使っていても、暗号化モードの選び方次第で、安全性が大きく変わる。

### ECBモード：最も単純で最も危険

**ECB**（Electronic CodeBook）モードは、各ブロックを互いに無関係に、独立して暗号化する方式である。

```
平文ブロック1: "ABCDEFGH" + 鍵 → "X#$%@!K#"
平文ブロック2: "ABCDEFGH" + 鍵 → "X#$%@!K#"（同じ平文なら同じ暗号文になる）
平文ブロック3: "IJKLMNOP" + 鍵 → "Q7Z9M2R$"
```

ECBモードには明確な欠陥がある。
同じ平文ブロックは常に同じ暗号文ブロックになる。
元のデータに同じパターンが繰り返されていれば、暗号化したあとのデータにも、同じパターンがそのまま現れる。

この欠陥を視覚的に示す有名な例が、ペンギンの写真をECBモードで暗号化した画像である。
写真の中で同じ色が続く部分（ペンギンの黒い体の部分など)は、暗号化後もブロックごとに同じ暗号文になる。
結果として、暗号化したはずの画像データから、ペンギンの輪郭がそのまま見えてしまう。
色は変わっても、形という構造情報は暗号化前と変わらず残る。

<Callout title="ECBの悪名：暗号化してもキャラクターの形が見える例">
Linuxのマスコットキャラクターである「Tux」の画像をECBモードで暗号化すると、暗号文として出力された画像データからもTuxの輪郭が判別できる。この現象は、ECBモードがパターンを隠せないことを示す例として、セキュリティの教材で繰り返し引用されている。
</Callout>

ECBモードは、暗号化の入門で「使ってはいけない例」として紹介されるのが定番であり、実務での利用は避けるべきモードである。

### CBCモード：前のブロックを次に混ぜる

**CBC**（Cipher Block Chaining）モードは、前のブロックの暗号文を、次のブロックの暗号化に混ぜ込む方式である。

```
IV（初期化ベクトル）: ランダムな値
ブロック1: 平文1 XOR IV + 鍵 → 暗号文1
ブロック2: 平文2 XOR 暗号文1 + 鍵 → 暗号文2
ブロック3: 平文3 XOR 暗号文2 + 鍵 → 暗号文3
```

`XOR`は二つのビット列の排他的論理和を取る演算であり、同じビットなら0、異なるビットなら1を返す。
ここでは、平文と前のブロックの暗号文（最初のブロックでは**IV**、Initialization Vector、初期化ベクトルと呼ばれるランダムな値）を混ぜ合わせるために使われている。

CBCモードでは、同じ平文ブロックでも、直前のブロックの内容が異なれば暗号文も変わる。
ECBモードのようにパターンがそのまま漏れることはない。

CBCモードにも弱点がある。
実装によっては、復号時のパディング処理(ブロックの末尾を埋める処理)の挙動の違いから、暗号文の一部を少しずつ書き換えて反応の違いを観察し、平文を推測する**パディングオラクル攻撃**という手法が成立する場合がある。

### GCMモード：暗号化と改ざん検知を同時に行う

**GCM**（Galois/Counter Mode）モードは、データを暗号化しながら、同時に改ざんの有無を確認できる**認証タグ**という値も生成する方式である。

```
平文 + 鍵 + ノンス → 暗号文 + 認証タグ
```

`ノンス`は一度しか使わない値という意味の使い捨ての初期値で、CBCモードのIVに近い役割を持つ。

GCMモードは「内容を隠す」処理と「改ざんされていないか確認する」処理を一つの手順の中でまとめて行う。
そのため実装者が処理を分けて組み合わせる必要がなく、実装ミスの入り込む余地が少ない。

HTTPS通信を支えるTLS 1.3、多くのパスワード管理ソフト、ディスク全体を暗号化するBitLockerなど、現在広く使われているセキュリティ機構の多くがGCMモードを採用している。

<Callout title="鍵長の長さだけで安全性は決まらない">
「AES-256はAES-128より2倍安全」という言い方は、正確ではない。鍵長が長いほど総当たり攻撃に強いのは事実である。ただし実務で発生する被害は、総当たり攻撃よりも、実装の誤りや鍵管理の不備、IVの再利用といった運用上の問題に起因することが多い。ECBモードで動かしているAES-256よりも、GCMモードで動かしているAES-128のほうが、実際の運用では安全である場合がある。
</Callout>

## 鍵配送問題：共通鍵暗号が抱える構造的な課題

共通鍵暗号には、仕組みそのものに由来する課題が一つ残っている。

オンラインショップにクレジットカード情報を送りたい場面を考える。
情報を暗号化して送るには、ショップとあらかじめ同じ鍵を共有しておく必要がある。
ではその鍵を、どうやって安全にショップへ渡すのか。

```
利用者: 「鍵を送るね」
  ↓ 鍵を何らかの方法で送信
  ↓（この過程で鍵が盗まれたら、その後の暗号化は意味を失う）
ショップ: 「鍵を受け取った」
```

鍵をメールで送れば、メールの経路で盗まれる可能性がある。
電話で伝えれば、通話を傍受される可能性がある。
郵便で送れば、配送の途中で開封される可能性がある。
鍵を安全に渡す手段そのものが、暗号化したい通信と同じ問題を抱えている。

この課題を**鍵配送問題**と呼ぶ。
1970年代の暗号学者たちが取り組んだ、共通鍵暗号の構造に根を持つ問題である。

鍵配送問題への解決策が公開鍵暗号であり、鍵そのものを送らずに秘密を作り出す手法を使う。
この手法は次のページで扱う。

## ストリーム暗号という別系統の方式

共通鍵暗号には、ブロック暗号のほかにもう一つの系統がある。
データを固定サイズのブロックに分けて処理するのがブロック暗号であるのに対し、データを1ビットまたは1バイトずつ連続して処理する方式を**ストリーム暗号**と呼ぶ。
ChaCha20やRC4がこの系統に属する。

ストリーム暗号は、データ全体を一度に読み込む必要がないため、通信のように途切れなく流れてくるデータの暗号化に向いている。
TLS 1.3ではChaCha20がブロック暗号の代替として使える。

RC4のような古いストリーム暗号には、出力に統計的な偏りがあることが判明しており、現在は使うべきではない。
実務でストリーム暗号を選ぶなら、ChaCha20のような検証済みの方式を使う。

## ここで押さえること

- **共通鍵暗号**は暗号化と復号に同じ鍵を使う。AESが現在の標準的な方式である
- AESは128ビット単位のブロック暗号であり、鍵長は128、192、256ビットから選べる。256ビットを推奨する
- 複数ブロックの処理方法を決める**暗号化モード**が安全性を左右する。ECBはパターンが漏れるため避け、GCM（認証付き）が現在の標準である
- 鍵長を長くするだけでは安全にならない。実装の正しさと鍵の管理が安全性を決める
- **鍵配送問題**は、共通鍵をどう安全に相手へ渡すかという課題である。この課題への解決策が公開鍵暗号である
