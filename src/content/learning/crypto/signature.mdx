## 「メールの From はなぜ詐称できるのか」

あなたはメールを受け取ったとき、本当に「送信者本人」からのメールか確認しますか？

```
From: noreply@amazon.co.jp
Subject: アカウント確認が必要です
```

でも実は、この `From` はハッカーが簡単に偽造できます。

SMTP（メール送信プロトコル）には、送信者の本人確認機能がもともとない。好き勝手に `From` を書いて送信できるんです。

```
実は送信元: ハッカーのサーバー
From: noreply@amazon.co.jp（嘘）
本文: 「クリックしてパスワード確認」（フィッシング）
```

このメールを受け取った多くの人が「Amazon からのメール」だと信じて、クリックしてしまいます。

でも「メールを改ざんされたのか」「本当に本人が送ったのか」を、数学的に検証する方法がある——それが **電子署名** です。

## 電子署名とは

電子署名の仕組み：

```
送信者: メッセージ + 秘密鍵 → 署名を付けたメッセージを送信
受信者: メッセージ + 署名 + 送信者の公開鍵 → 「本当に送信者が送ったか」を検証
```

重要なポイント：

- **改ざん検知**：メッセージの1ビットでも改ざんされたら、署名が一致しない
- **送信者認証**：秘密鍵は送信者だけが持っているから、「この署名は本当に本人が作ったのか」を検証できる

一般的には秘密鍵で「メッセージそのもの」に署名するのではなく、**メッセージのハッシュ値** に署名します。

```
送信者:
  1. メッセージ → SHA-256計算 → ハッシュ（64字）
  2. ハッシュ + 秘密鍵 → RSA署名
  3. 元のメッセージ + 署名を送信

受信者:
  1. 受信したメッセージ → SHA-256計算 → ハッシュ
  2. 受信した署名 + 送信者の公開鍵 → RSA検証
  3. 計算したハッシュと、検証したハッシュが一致？
     YES → 「改ざんされていない」「本当に送信者が送った」
     NO → 「誰かが改ざんした」または「本人じゃない」
```

なぜハッシュに署名するのか。**メッセージが巨大でも、ハッシュは固定長（256ビット）だから、署名・検証が高速** になるから。

## 「改ざん検知」と「送信者認証」を同時に実現

電子署名が凄いのは、この2つを同時に達成できる点。

**改ざん検知：**

メッセージが改ざんされたら、ハッシュが変わる（雪崩効果）。ハッシュが変わったら、署名が一致しなくなる。

```
元のメッセージ: "Send $100 to account A"
ハッシュ: abc123...

改ざん後: "Send $1000 to account A"
ハッシュ: xyz789...（全く別）

署名は元のハッシュ（abc123...）から作られているので、新しいハッシュ（xyz789...）では検証できない
```

**送信者認証：**

署名は秘密鍵でのみ生成できます。秘密鍵を持っているのは送信者だけ。

だから「この署名は本当に秘密鍵の持ち主が作ったのか」を公開鍵で検証できる。

```
誰かが署名を偽造しようとする
  ↓ でも秘密鍵を持っていない
  ↓ 
正しい署名を数学的に生成することは、RSA-2048なら計算上不可能
```

フィッシングメール対策に戻りましょう。

メールに電子署名があれば：

```
From: noreply@amazon.co.jp
署名: [秘密鍵で署名されたハッシュ]

メーラーが検証:
  Amazon の公開鍵でこの署名を検証
  → 秘密鍵で生成された署名か？
  → NO → このメールは偽造。警告表示
```

ただし「公開鍵が本当に Amazon のものか」を確認する必要がある。ここで次の章の「証明書」が登場します。

## Git コミット署名：実装レベルでの活用

実際の使用例として、Git の署名機能があります。

```bash
git commit -S -m "Fix security bug"
```

オプション `-S` で署名すると、このコミットには GPG署名が付きます。

```
commit abc123...
Author: Alice <alice@company.com>
gpgsig: -----BEGIN PGP SIGNATURE-----
        [署名の内容]
        -----END PGP SIGNATURE-----

Fix security bug
```

後から「本当に Alice が書いたコミットか」を検証できます。

なぜ大事か。重要な本番環境へのコード変更は、「署名済みコミット」だけを許可するルールを企業が作ることがあります。

```
pull request の merge 条件:
  - テストが通っている
  - コードレビューが完了している
  - 全コミットが署名済み（秘密鍵を持ってるメンバーだけが書いたこと）
```

これなら、ハッカーが盗んだアカウントで「正当な人になりすまして」コミットを混ぜることを防げます。

<Callout title="GitHub での署名表示">
GitHub は署名済みコミットに「Verified」バッジを表示します。オープンソースプロジェクトでも、公開鍵を GitHub にアップロードすれば、コミットの信頼性が可視化される。
</Callout>

## メール認証：DKIM、SPF、DMARC

フィッシング対策として、メール認証の標準がいくつか定められています。

**SPF（Sender Policy Framework）：**

「このドメインからメールを送信できるサーバーはどれか」をDNSに記録。

```
example.com の DNS:
  v=spf1 include:_spf.google.com ~all
  （意味：Google のメールサーバーからのみ送信OK）
```

メーラーが受信したメールの送信元 IP が、SPF レコードに記録されたサーバーか確認。違ったら警告。

**DKIM（DomainKeys Identified Mail）：**

電子署名の実装版。メールサーバーがメッセージに秘密鍵で署名。

```
メールヘッダ:
  DKIM-Signature: ...
  （秘密鍵で署名されたハッシュ）
```

受信側が DKIM 公開鍵（DNS に公開）で検証。

**DMARC（Domain-based Message Authentication, Reporting and Conformance）：**

SPF と DKIM の検証結果に基づいて「検証に失敗したメールはどう処理するか」を指定。

```
example.com の DMARC ポリシー:
  p=reject （検証失敗したメールは受信しない）
  p=quarantine （スパムフォルダに）
```

大手企業（Gmail、Microsoft、Yahoo など）は DMARC を `p=reject` で設定しているため、偽の From を使ったメールは届きにくくなっています。

## コード署名証明書の役割

ソフトウェア配布の際にも署名が重要です。

Windows でアプリをダウンロードすると、「発行者: Microsoft」とか「発行者: Adobe」という表示を見ることがあります。

```
EXE ファイル + コード署名証明書（秘密鍵で署名）
  ↓
Windows が実行時に検証
  ↓
「このアプリは信頼できる発行者から来ている」
```

署名がない、または改ざんされたアプリを実行しようとすると、Windows は警告を出します。

## ここで押さえること

- **電子署名**は「改ざん検知」と「送信者認証」を同時に実現
- 秘密鍵で署名、公開鍵で検証。秘密鍵を持つ人だけが有効な署名を作れる
- 実装的には「**メッセージのハッシュに署名**」する。ハッシュが固定長なので処理が高速
- **Git コミット署名**で「本当に本人が書いたか」を後から検証可能
- **メール認証**（SPF・DKIM・DMARC）でフィッシング対策。企業は DMARC で検証失敗メールを拒否
- **コード署名**で、ダウンロードしたソフトが改ざんされていないか確認。Windows / macOS が実行時に検証
