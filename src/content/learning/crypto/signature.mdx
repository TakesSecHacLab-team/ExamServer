## メールの送信元はなぜ偽れるのか

メールを受け取ったとき、その送信者が本当に名乗っている本人かどうかを確認する人は少ない。

```
From: noreply@amazon.co.jp
Subject: アカウント確認が必要です
```

この`From`欄は、攻撃者が比較的簡単に偽装できる。
メールの送信に使われる**SMTP**（Simple Mail Transfer Protocol、メール送信プロトコル)には、送信者が本人であることを確認する機能がもともと組み込まれていない。
送信するメールサーバーは、`From`欄に好きな文字列を書いて送信できる。

```
実際の送信元: 攻撃者が用意したサーバー
From: noreply@amazon.co.jp（偽り）
本文: 「クリックしてパスワードを確認してください」（フィッシングの誘導）
```

`From`欄だけを見て「Amazonからのメール」だと信じ、本文中のリンクをクリックしてしまう被害は後を絶たない。

メールが改ざんされていないか、本当に名乗っている本人が送ったものかを、数学的に検証する方法がある。
これが**電子署名**である。

## 電子署名の仕組み

電子署名は次のような手順で成立する。

```
送信者: メッセージ + 秘密鍵 → 署名を付けたメッセージを送信
受信者: メッセージ + 署名 + 送信者の公開鍵 → 「本当に送信者が送ったか」を検証
```

電子署名が果たす役割は二つある。

- **改ざん検知**：メッセージの内容が1ビットでも変われば、署名が一致しなくなる
- **送信者認証**：秘密鍵を持っているのは送信者本人だけなので、有効な署名が作れること自体が、送信者本人の関与を示す

実際の実装では、秘密鍵で署名する対象はメッセージそのものではなく、メッセージのハッシュ値である場合が多い。

```
送信者:
  1. メッセージ → SHA-256計算 → ハッシュ値（64字)
  2. ハッシュ値 + 秘密鍵 → RSA署名
  3. 元のメッセージと署名をまとめて送信

受信者:
  1. 受信したメッセージ → SHA-256計算 → ハッシュ値
  2. 受信した署名 + 送信者の公開鍵 → RSA検証
  3. 計算したハッシュ値と、検証で得られたハッシュ値が一致するか確認
     一致する → 改ざんされておらず、本当に送信者が送ったものと判断できる
     一致しない → 誰かが改ざんしたか、本人ではない誰かが署名したと判断できる
```

メッセージそのものに直接署名せず、ハッシュ値に署名する理由がある。
ハッシュ値は固定長（256ビットなど)であるため、署名と検証にかかる計算量がメッセージの長さに左右されず、処理が速くなる。

## 改ざん検知と送信者認証が同時に成り立つ理由

電子署名の特徴は、改ざん検知と送信者認証という二つの保証を、一つの仕組みで同時に得られる点にある。

改ざん検知が成り立つ理由は、ハッシュ関数の雪崩効果にある。
メッセージの内容が変われば、ハッシュ値も大きく変わる。
ハッシュ値が変われば、元のハッシュ値に対して作られた署名と一致しなくなる。

```
元のメッセージ: "Send $100 to account A"
ハッシュ: abc123...

改ざん後のメッセージ: "Send $1000 to account A"
ハッシュ: xyz789...（まったく別の値）

署名は元のハッシュ値（abc123...)から作られているため、改ざん後のハッシュ値（xyz789...)では検証が通らない
```

送信者認証が成り立つ理由は、秘密鍵を持つ者が一人に限られることにある。
秘密鍵を持っているのは送信者本人だけであるため、その秘密鍵から生成された有効な署名が存在すること自体が、送信者本人による処理だったことを示す。

```
何者かが署名を偽造しようとする
  ↓ しかし秘密鍵を持っていない
  ↓
正しい署名を秘密鍵なしに数学的に生成することは、RSA-2048の鍵長では計算上不可能である
```

メールの送信元偽装の話に戻ると、電子署名がある場合、受信側のメールソフトは次のように検証できる。

```
From: noreply@amazon.co.jp
署名: [秘密鍵で署名されたハッシュ値]

メーラーによる検証:
  Amazon の公開鍵でこの署名を検証する
  → 秘密鍵によって正しく生成された署名か確認する
  → NO（不一致）→ このメールは偽造されたものと判定し、警告を表示する
```

ただしこの検証には前提が一つ残る。
検証に使う公開鍵が本当にAmazonのものであることを、どうやって確認するのか。
この確認の仕組みが、次のページで扱う証明書である。

## Gitのコミット署名という実装例

電子署名の実用例として、Gitのコミット署名機能がある。

```bash
git commit -S -m "Fix security bug"
```

`-S`オプションを付けてコミットすると、そのコミットにGPG署名が付与される。

```
commit abc123...
Author: Alice <alice@company.com>
gpgsig: -----BEGIN PGP SIGNATURE-----
        [署名の内容]
        -----END PGP SIGNATURE-----

Fix security bug
```

このコミットを後から検証すれば、本当にAliceが作成したコミットかどうかを確認できる。

本番環境に影響するコードの変更では、署名済みコミットだけを許可する運用ルールを定める組織がある。

```
プルリクエストをマージする条件の例:
  - テストが通っている
  - コードレビューが完了している
  - 全コミットが署名済みである（秘密鍵を持つメンバーだけが作成したことを示す）
```

このルールがあれば、何者かが盗んだアカウントを使って、正当なメンバーになりすましてコミットを混入させる事態を防げる。

<Callout title="GitHubにおける署名の可視化">
GitHubは署名済みコミットに「Verified」というバッジを表示する。オープンソースのプロジェクトでも、公開鍵をGitHubに登録しておけば、各コミットの署名の有無が可視化される。
</Callout>

## SPFとDKIMとDMARCによるメール認証

フィッシング対策として、メールの送信元を検証するための標準がいくつか定められている。

**SPF**（Sender Policy Framework)は、あるドメインからメールを送信してよいサーバーをDNSに記録する仕組みである。

```
example.com の DNS レコード:
  v=spf1 include:_spf.google.com ~all
  （意味：Googleのメールサーバーからの送信のみを許可する）
```

メールを受け取ったサーバーは、送信元のIPアドレスがSPFレコードに記録されたサーバーと一致するかを確認する。
一致しなければ、そのメールは疑わしいと判定される。

**DKIM**（DomainKeys Identified Mail)は、電子署名をメール送信に応用した仕組みである。
メールを送信するサーバーが、メッセージに対して秘密鍵で署名を行う。

```
メールヘッダ:
  DKIM-Signature: ...
  （秘密鍵で署名されたハッシュ値）
```

受信側は、DNSに公開されたDKIM用の公開鍵を使って、この署名を検証する。

**DMARC**（Domain-based Message Authentication, Reporting and Conformance)は、SPFとDKIMの検証結果を踏まえて、検証に失敗したメールをどう扱うかを指定する仕組みである。

```
example.com の DMARC ポリシーの例:
  p=reject（検証に失敗したメールは受信しない）
  p=quarantine（スパムフォルダへ振り分ける）
```

GmailやMicrosoft、Yahooといった大手のメールサービスは、DMARCのポリシーを`p=reject`に設定している場合が多い。
これにより、偽った`From`を使ったメールは、宛先に届く前に拒否されやすくなっている。

## コード署名証明書によるソフトウェアの検証

ソフトウェアの配布にも電子署名が使われる。

Windowsでアプリケーションをダウンロードすると、「発行者: Microsoft」や「発行者: Adobe」といった表示を見ることがある。

```
実行ファイル + コード署名証明書（秘密鍵で署名済み）
  ↓
Windowsが実行時に署名を検証する
  ↓
「このアプリケーションは信頼できる発行者から提供されている」と判定する
```

署名がない実行ファイルや、署名後に内容が改ざんされた実行ファイルを実行しようとすると、Windowsは警告を表示する。

## ここで押さえること

- **電子署名**は、改ざん検知と送信者認証を一つの仕組みで同時に実現する
- 署名の生成には秘密鍵を、検証には公開鍵を使う。秘密鍵を持つ者だけが有効な署名を作れる
- 実装上は、メッセージそのものではなく**メッセージのハッシュ値**に署名する。ハッシュ値が固定長であるため、処理が高速になる
- **Gitのコミット署名**を使うと、コミットを本当に書いた本人を後から検証できる
- **SPF、DKIM、DMARC**はメールの送信元を検証する標準で、組織はDMARCの設定によって検証失敗メールの扱いを決められる
- **コード署名証明書**により、ダウンロードしたソフトウェアが改ざんされていないかをOSが実行時に検証できる
