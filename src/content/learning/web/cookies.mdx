## HTTPはなぜ「前のことを覚えていない」のか

クッキーの話を始める前に、HTTPそのものを確認します。

**HTTP**は、ブラウザとサーバーがデータをやり取りするための通信規約です。

ブラウザが「このページをくれ」という**リクエスト**を送り、サーバーが「はい、これです」という**レスポンス**を返す。

この往復だけでWebページは表示されています。

リクエストとレスポンスには、本文（HTMLやJSONなど）だけでなく、**ヘッダー**という付帯情報が付きます。

ヘッダーは「名前: 値」の形をした、本文に関するメタ情報です。

たとえばレスポンスのヘッダーには「このファイルの種類はHTMLです」「このファイルは1234バイトです」といった情報が乗ります。

ここで一つ確認しておきたい性質があります。

HTTPは**ステートレス**だと言われます。

ステートレスとは「状態を持たない」という意味で、ここでは「サーバーが前のリクエストの内容を覚えていない」ことを指します。

サーバーは1件のリクエストを受け取って1件のレスポンスを返すだけで、そのリクエストを送ってきたのが誰で、5秒前に何をした人かを、HTTPの仕組みそのものでは判断できません。

この性質が、これから説明するクッキーが必要になる理由です。

ログイン機能を考えると分かりやすくなります。

ユーザーがIDとパスワードを送って「ログインに成功した」としても、次にユーザーが別のページを開くと、それは新しい別のリクエストです。

サーバーは「さっきログインに成功した人」と「今ページを開こうとしている人」が同じかどうかを、何も手を打たなければ判定できません。

そこでサーバーは、最初のレスポンスで「次回からはこの目印を一緒に送ってください」という指示をブラウザに渡します。

ブラウザがその目印を覚えておき、次のリクエストで送り返すと、サーバーは「ああ、さっきログインした人だ」と認識できます。

この「サーバーが渡し、ブラウザが覚えて送り返す目印」が**クッキー**です。

クッキーは、ステートレスなHTTPの上に「状態」を後付けする仕組みだと言えます。

## ドメインとオリジンは何を区別しているのか

クッキーの送り先を理解するには、ドメインとオリジンという二つの概念を区別しておく必要があります。

**ドメイン**は `example.com` のような、サーバーの住所に相当する文字列です。

**オリジン**は、ドメインに加えて「スキーム（`https://` か `http://` か）」と「ポート番号」までを含めた単位です。

たとえば `https://example.com` と `http://example.com` は、ドメインは同じでもオリジンは別です。

ブラウザのセキュリティ機構の多くは、このオリジンを基準に「同じサイトかどうか」を判定しています。

クッキーも、どのドメイン宛てのリクエストに付けるかを、発行時に指定したドメインの範囲で管理しています。

## DevToolsでクッキーを実際に見てみる

仕組みの説明に入る前に、実物を見ておきます。

ブラウザで任意のサイトを開き、F12キー（Macなら Command+Option+I）を押してDevToolsを開きます。

「Application」タブの「Cookies」を開くと、そのサイトが保存しているクッキーの一覧が表示されます。

名前、値、ドメイン、有効期限、HttpOnlyなど、複数の列を持つ表が並んでいます。

これは、サーバーがブラウザに預けている目印の保管庫そのものです。

<Callout title="DevToolsで実際に確認する">
  DevTools の Application → Cookies で、各クッキーの属性値が確認できる。HttpOnly 列にチェックが入っているクッキーは、ブラウザのコンソールから `document.cookie` を実行しても値が取れない。実際にコンソールを開いて試すと、属性の効果が体感できる。
</Callout>

## Set-CookieヘッダーとCookieヘッダーの往復

クッキーは、先ほど説明したヘッダーを使ってサーバーとブラウザの間でやり取りされます。

まず、サーバーがブラウザにクッキーを渡す場面です。

```
HTTP/1.1 200 OK
Set-Cookie: sessionId=abc123def456; Path=/; HttpOnly; Secure; Max-Age=3600
```

`Set-Cookie` は「これを保存しておいてください」というサーバーからの指示です。

ブラウザはこのヘッダーを受け取ると、指示された名前と値を端末にローカル保存します。

次に、ブラウザが保存済みのクッキーをサーバーに送り返す場面です。

```
GET /dashboard HTTP/1.1
Host: example.com
Cookie: sessionId=abc123def456
```

ブラウザは、同じドメイン宛てのリクエストを送るたびに、保存しているクッキーを自動で `Cookie` ヘッダーに乗せます。

ここで押さえておきたいのは「自動で」という点です。

ユーザーが何か操作する必要はなく、ページ側でJavaScriptを書く必要もありません。

ブラウザが、保存条件に合致するクッキーを毎回勝手に付けて送ります。

この自動送信という挙動が、クッキーの便利さの根拠であり、同時にこの講義の後半で説明するCSRFという攻撃が成立する根拠でもあります。

## なぜブラウザは送り先を選んでクッキーを送るのか

ブラウザは、保存しているクッキーを無差別に全リクエストへ付けているわけではありません。

クッキーには発行時に「どのドメインの、どのパスへのリクエストに付けるか」という条件が指定されており、ブラウザはその条件に合致するリクエストにだけクッキーを付けます。

`example.com` が発行したクッキーは、原則として `example.com` 宛てのリクエストにしか付きません。

`other.com` へのリクエストには付かないので、`other.com` のサーバーが `example.com` のクッキーを盗み見ることはできません。

この「発行元のサイトにしか送らない」という制約があるおかげで、クッキーはサイトをまたいだ識別子の漏洩を防いでいます。

ただし、この制約には例外が存在します。

後述する「サードパーティクッキー」は、この前提を一部崩す形で使われてきた仕組みです。

## Set-Cookieの属性を一つずつ読む

実際のSet-Cookieヘッダーには、複数の属性がセミコロンで連結されています。

```
Set-Cookie: sessionId=abc123; Path=/; HttpOnly; Secure; SameSite=Lax; Max-Age=3600
```

それぞれの属性が何を制御しているか、一つずつ確認します。

### name=value（クッキー本体）

`sessionId=abc123` の部分が、保存する名前と値の組です。

認証用途では `sessionId=ランダムな文字列` という形が典型的です。

値そのものにユーザー名やメールアドレスなどの個人情報を直接入れるべきではありません。

理由は次のセッションIDの節で説明します。

### Path=/

クッキーをどのパス配下へのリクエストに付けるかを指定します。

`Path=/api` と指定すれば、`/api` 配下へのリクエストにだけクッキーが付きます。

サイト全体に付けたい場合は `Path=/` を指定します。

### HttpOnly

`document.cookie` というJavaScriptの命令からクッキーを読めなくする属性です。

この属性が必要になる理由を理解するには、**XSS**（クロスサイトスクリプティング）という攻撃を知っておく必要があります。

XSSは、攻撃者が被害者のブラウザ上で、被害者の意図しないJavaScriptを実行させる攻撃です。

たとえば、コメント欄に入力した文字列がそのままページに埋め込まれて実行されてしまう不備があると、攻撃者はそこに `document.cookie` を読み取って外部に送信するスクリプトを仕込めます。

`HttpOnly` を付けたクッキーは、ページ上でどんなJavaScriptが実行されても `document.cookie` の結果に含まれません。

XSSでスクリプトを仕込まれても、HttpOnly付きのクッキーまでは盗まれない、というのがこの属性の役割です。

### Secure

HTTPS接続のときだけクッキーを送信し、暗号化されていないHTTP接続では送信しない属性です。

暗号化されていない通信は、途中の通信経路で内容を読み取られる可能性があります。

`Secure` を付けることで、クッキーが平文の通信に乗って漏れる経路を断ちます。

本番環境のクッキーには、原則として `Secure` を付けます。

### SameSite

別ドメインのサイトを経由したリクエスト（クロスサイトリクエスト）のときに、クッキーを送るかどうかを制御する属性です。

値は3種類あります。

`SameSite` は、後述するCSRFという攻撃への対策として重要な役割を持っています。

### Max-Age=3600

クッキーが何秒後に削除されるかを指定します。

`3600` は3600秒、つまり1時間です。

`Max-Age` を指定しないクッキーは「セッションクッキー」と呼ばれ、ブラウザのウィンドウやタブを閉じると消えます。

ここで言う「セッションクッキー」という呼び名は、次に説明する「セッションID」とは別の概念なので混同しないでください。

セッションクッキーは「いつ消えるか」についての分類で、セッションIDは「何のために使うクッキーか」についての分類です。

## セッションIDという識別子の仕組み

認証で最もよく使われるパターンは、クッキーに**セッションID**という識別子を一つだけ入れる方法です。

```
Set-Cookie: sessionId=8f3a9b7c4d12e6f0; HttpOnly; Secure; Path=/
```

`8f3a9b7c4d12e6f0` という値そのものには、ユーザー名やメールアドレスなどの情報は含まれていません。

これは単なるランダムな文字列です。

ユーザー情報は、サーバー側で別に保持しています。

```javascript
// サーバー内のセッションストア(概念的なイメージ)
sessionStore = {
  "8f3a9b7c4d12e6f0": {
    userId: 42,
    username: "taro",
    loginAt: "2026-06-20T10:30:00Z"
  }
}
```

ブラウザからこのIDが届くたびに、サーバーは「このIDに対応するユーザーは誰か」をセッションストアから引いてきます。

パスワードを使うのはログインのその1回だけで、その後の各リクエストではこのランダムな識別子だけでユーザーを判別しています。

なぜパスワードそのものをクッキーに入れないのか、機構を確認します。

セッションIDが盗まれても、その値からパスワードを逆算することはできません。

また、セッションIDはサーバー側のセッションストアから削除するだけで、次のリクエストから即座に無効化できます。

パスワードそのものをクッキーに入れていた場合、パスワードが漏れた時点でユーザーは全サービスでの再利用も含めて被害を受けますが、セッションIDの漏洩はそのセッションだけの被害にとどめられます。

## HttpOnlyを入れただけでは守れないもの

ここまでの説明を読むと、「HttpOnlyを付けておけばクッキーは安全だ」と考えたくなるかもしれません。

しかし、HttpOnlyが防いでいるのは「JavaScriptからセッションIDを読み取られること」だけです。

これは具体的には、XSSで仕込まれたスクリプトが `document.cookie` を実行してセッションIDを盗み出す経路への対策です。

**CSRF**（クロスサイトリクエストフォージェリ）は、これとは別の経路を使う攻撃です。

CSRFは、被害者のブラウザを使って、被害者の名前でリクエストを送らせる攻撃です。

攻撃者が用意した別サイトのページに、被害者のサイトへの送金フォームなどを埋め込んでおき、被害者がそのページを開いた瞬間に自動でリクエストが送られるようにします。

ブラウザは、リクエスト先のドメインに対応するクッキーを自動で付けて送るので、被害者のセッションIDが乗ったままリクエストが送信されてしまいます。

この送信は被害者本人がボタンをクリックする操作だけで成立し、JavaScriptで `document.cookie` を読み取る必要もありません。

つまりHttpOnlyは、このCSRFという経路には関与しません。

CSRFへの対策には、`SameSite` 属性の指定、またはCSRFトークン（リクエストごとに発行される、推測できない値をフォームに埋め込んでサーバー側で検証する仕組み）が別途必要です。

「HttpOnlyを入れたから安全」という考え方は、対策している攻撃の種類を取り違えています。

HttpOnly、Secure、SameSiteの3属性を組み合わせて初めて、実用的な防御になります。

## サードパーティクッキーはなぜ制限されてきたのか

`example.com` のページを見ているのに、広告ネットワークである `ads.tracking.io` がそのページ内のリクエストに対して `Set-Cookie` を送ってくる場合があります。

このように、いま開いているサイトとは別のドメインが発行するクッキーを**サードパーティクッキー**と呼びます。

広告ネットワークは、複数のサイトに自社の広告枠や計測タグを埋め込んでおくことで、同じ識別子を使って利用者の行動を複数サイトにわたって追跡できます。

これがサードパーティクッキーの主な使われ方でした。

Chromeは2024年から段階的にサードパーティクッキーの送信を制限しています。

Firefox とSafariは、それより前から制限を導入しています。

この流れを受けて、GoogleはPrivacy Sandboxという代替の仕組みを提案していますが、業界の反発もあり、仕様は現在も変化を続けています。

ここで言う制限は、サードパーティクッキーに対するものです。

`example.com` 自身が発行する**ファーストパーティクッキー**には影響しません。

ログイン状態の保持や、表示言語の設定保存といった用途には、ファーストパーティクッキーを引き続き使えます。

## ここで押さえること

- HTTPはステートレスで、サーバーは前のリクエストを覚えていない。クッキーは、サーバーが渡した目印をブラウザが覚えて送り返すことで、この欠けている「状態」を補う仕組みである。
- クッキーは `Set-Cookie` ヘッダーでサーバーから指示され、ブラウザがローカルに保存し、以後同じドメイン宛てのリクエストに `Cookie` ヘッダーとして自動で付けて送る。
- `HttpOnly` はJavaScriptからの読み取りを防ぎ、XSSによるセッションIDの窃取に対策する。`Secure` はHTTPS接続でのみ送信し、通信の盗聴に対策する。`SameSite` はクロスサイトでの送信を制御し、CSRFに対策する。
- 認証にはセッションIDが使われる。パスワードではなくランダムな識別子をクッキーに入れ、ユーザー情報はサーバー側のセッションストアで管理する。
- HttpOnlyが防ぐのはXSS経由の窃取であり、CSRFには無力である。CSRF対策にはSameSiteかCSRFトークンが別途必要になる。
