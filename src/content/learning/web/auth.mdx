## ログイン機能がなぜ必要なのか

Webサイトを開くと、多くの場合まずメールアドレスとパスワードの入力を求められます。

これは単なる手続きではありません。

サーバーは、リクエストを送ってきた相手が誰なのかを、何も知らない状態から始まります。

ブラウザから届くリクエストには、送り主の名前を保証するものは何も付いていません。

「自分は田中です」と名乗るリクエストが本当に田中本人から来たのか、それとも田中になりすました別人から来たのかを、サーバーは見分ける必要があります。

ログイン機能は、この「相手が誰であるかを確認する」ための仕組みです。

これから、確認の仕組みを土台から積み上げていきます。

## HTTPのリクエストとレスポンスとは何か

仕組みを理解するには、まずWebがどう動いているかを知っておく必要があります。

ブラウザとサーバーは、**HTTP**という約束事に従ってデータをやり取りします。

ブラウザが「このページをください」とサーバーに送るデータを**リクエスト**と呼びます。

サーバーがそれに応えて返すデータを**レスポンス**と呼びます。

リクエストには、何を求めているかという情報（URLやメソッド）と、追加の情報（ヘッダーや本文）が含まれます。

ログインフォームで「ログイン」ボタンを押すと、ブラウザは入力されたメールアドレスとパスワードを本文に詰めて、こういうリクエストを送ります。

```
POST /login HTTP/1.1
Host: example.com
Content-Type: application/json

{"email": "taro@example.com", "password": "secret123"}
```

サーバーはこのリクエストを受け取り、データベースを調べて、結果をレスポンスとして返します。

ここで重要な事実があります。

**HTTPには「状態」を覚える仕組みがありません。**

1回目のリクエストでログインに成功しても、2回目のリクエストを送ったサーバーは、それが同じ人からのものかどうかを、デフォルトでは知りません。

リクエストはそれぞれ独立していて、サーバー側に「さっきの続き」という記憶は残らないからです。

この性質を**ステートレス**と呼びます。

ログインした状態を次のページでも維持するには、何らかの工夫が必要になります。

その工夫が、後で説明する「セッション」です。

## 平文保存とは何か、なぜ危険なのか

ログイン機能を作るとき、最初に決めるべきことがあります。

データベースにパスワードをどう保存するかです。

最も単純な発想は、ユーザーが入力した文字列をそのままデータベースに入れることです。

```sql
-- 危険な例
INSERT INTO users (email, password) VALUES ('taro@example.com', 'secret123');
```

この `secret123` のように、暗号化や変換を一切していない元の文字列を**平文**と呼びます。

平文保存が危険なのは、攻撃者がデータベースに何らかの方法で侵入したとき、保存されている全ユーザーのパスワードがそのまま読めてしまうからです。

データベースへの侵入は、SQLインジェクション、設定ミスによる外部公開、内部関係者による持ち出しなど、複数の経路で起こり得ます。

多くの人は複数のサービスで同じパスワードを使い回しています。

そのため、1つのサービスのデータベースが漏洩すると、同じメールアドレスとパスワードの組み合わせで別のサービスにもログインされてしまう被害（パスワードリスト攻撃）が起こります。

ここから導かれる結論は1つです。

パスワードは、データベースに侵入されても読めない形で保存しなければなりません。

その方法を次に説明します。

## ハッシュ関数とは何か、一方向性とは何か

パスワードを安全に保存するための鍵になる道具が**ハッシュ関数**です。

ハッシュ関数とは、どんな長さの入力を渡しても、決まった長さの文字列（ハッシュ値）を返す関数のことです。

同じ入力を渡すと、必ず同じ出力が返ってきます。

入力が1文字でも変われば、出力はまったく違う値になります。

ハッシュ関数には、もう1つ重要な性質があります。

**一方向性**です。

一方向性とは、ハッシュ値から元の入力を計算で逆算できない、という性質を指します。

逆算できないというのは「計算量的に非現実的」という意味です。

理論上は同じハッシュ値を持つ入力が他にも存在しますが、それを実用的な時間で探し出す方法が知られていません。

この一方向性のおかげで、次のことが成り立ちます。

データベースにはパスワードのハッシュ値だけを保存しておき、ログイン時には入力されたパスワードを同じ関数でハッシュ化して、保存済みのハッシュ値と一致するかどうかを比較する。

一致すればパスワードが正しいと判断できますが、ハッシュ値自体から元のパスワードを取り出すことはできません。

攻撃者がデータベースに侵入してハッシュ値を盗んでも、そこから直接パスワードはわかりません。

これが、平文保存と比べてハッシュ化が安全とされる理由です。

```sql
-- 正しい例（ハッシュ値を保存する）
INSERT INTO users (email, password) VALUES ('taro@example.com', '$2b$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lHBe');
```

この長い文字列が「bcryptハッシュ」と呼ばれる、ハッシュ化されたパスワードです。

ログイン時にやることは、入力されたパスワードを同じ関数に通して、保存済みのハッシュ値と比較するだけです。

データベースにはパスワードの実体を持たせない。

これが正しいパスワード管理の出発点になります。

## ブルートフォース攻撃とは何か

ハッシュ化すれば安全、と単純に言い切れるわけではありません。

ハッシュ関数には「同じ入力なら同じ出力になる」という性質があるため、攻撃者は次のような手順を踏めます。

考えられるパスワードの候補を片っ端から用意し、それぞれをハッシュ化して、盗んだハッシュ値と一致するものを探す。

候補をしらみつぶしに試すこの攻撃を**ブルートフォース攻撃**（総当たり攻撃）と呼びます。

「password123」のような辞書に載っている単語や、よくあるパターンのパスワードは、この方法で短時間に割られます。

ここで問題になるのが、ハッシュ関数の計算速度です。

`SHA-256`のような汎用のハッシュ関数は、もともと高速な処理を目的に設計されています。

計算が速いということは、攻撃者が1秒間に試せる候補の数も増えるということです。

現代的なハードウェアを使えば、SHA-256で1秒に数億から数十億回のハッシュ計算ができてしまいます。

つまり、パスワードのハッシュ化に汎用ハッシュ関数を使うと、ブルートフォース攻撃に弱くなります。

この問題に対処するために作られたのが、わざと計算を遅くしたハッシュ関数です。

## bcryptがわざと遅く設計されている理由

パスワードのハッシュ化には、SHA-256のような汎用ハッシュ関数ではなく、**bcrypt**や**Argon2**を使います。

理由は、前の節で説明したブルートフォース攻撃への耐性です。

bcryptは、ハッシュ計算を意図的に何度も繰り返すことで、1回の計算に時間がかかるよう設計されています。

```javascript
const bcrypt = require('bcrypt');

// ハッシュ化(ユーザー登録時)
const SALT_ROUNDS = 10;
const hash = await bcrypt.hash('secret123', SALT_ROUNDS);
// → '$2b$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lHBe'

// 検証(ログイン時)
const isMatch = await bcrypt.compare('secret123', hash); // true
const isMatch2 = await bcrypt.compare('wrong_password', hash); // false
```

生成されたハッシュ値の `$2b$10$...` という形式の `10` は、コストファクター（ストレッチング回数）と呼ばれる値です。

この値が10だと、2の10乗、つまり1024回の計算を繰り返してから1つのハッシュ値を返します。

繰り返し計算によって、ハッシュ1回あたりの処理時間が数十から数百ミリ秒まで引き延ばされます。

人間が1回ログインする分には、数百ミリ秒の遅延はほぼ気になりません。

しかし攻撃者の視点で見ると、1秒に試せる候補が数十回程度に制限されることになり、総当たりで全パターンを試すには現実的でない年数がかかります。

`bcrypt.compare`がやっていることは、入力値を同じ手順でハッシュ化し、保存済みのハッシュ値と一致するか調べることです。

元のパスワードを復号しているわけではありません。

そもそも、一方向性を持つハッシュ関数からの復号自体が成立しません。

<Callout title="ソルトという仕組み">
  同じパスワードを複数の人が使っていても、bcryptが生成するハッシュ値は毎回ランダムな「ソルト」という値を混ぜてから計算するため、結果が異なります。

  これによって、事前に大量のパスワードをハッシュ化しておいて検索表（レインボーテーブル）と一致させる攻撃を防げます。
</Callout>

## 認証と認可は別の問いに答えている

ハッシュ化の仕組みがわかったところで、混同されがちな2つの用語を分けて整理します。

**認証**（Authentication）とは、「リクエストを送ってきた相手が、名乗っている本人かどうか」を確認することです。

**認可**（Authorization）とは、「その本人に、求めている操作をする権利があるかどうか」を確認することです。

```
ログインにパスワードを入れる  → 認証
管理画面にアクセスしようとする → 認可(管理者権限があるか)
```

「認証」は身元の確認であり、「認可」は権限の確認です。

この2つを混同したまま実装すると、典型的な失敗が起こります。

ログイン機能そのものは正しく作ったのに、各エンドポイントで権限チェックを書き忘れる、という失敗です。

その場合、ログインさえ済ませれば、本来アクセス権のないユーザーでも他人のデータを見られてしまいます。

「認証」は「誰か」を確認するだけで、「何をしていいか」を決めるのは「認可」の役割です。

両者は別の問いに答えているので、片方だけ実装しても安全にはなりません。

## ミドルウェアとは何か

これから先のコード例には「ミドルウェア」という言葉が出てきます。

先に説明しておきます。

サーバーがリクエストを受け取ってからレスポンスを返すまでの間には、複数の処理が順番に挟まることがよくあります。

たとえば「ログインしているかを確認する処理」と「実際にデータを返す処理」は、別々の処理として分けて書けます。

このように、リクエストとその最終的な処理の間に挟み込まれる、共通の前処理を担う関数を**ミドルウェア**と呼びます。

ミドルウェアは、確認が済んだら次の処理に進めるための合図（多くのフレームワークでは `next()` という関数の呼び出し）を出します。

確認に失敗した場合は、次の処理に進めずに、その場でエラーのレスポンスを返します。

「ログイン確認」と「権限確認」をそれぞれ別のミドルウェアとして書いておけば、必要なエンドポイントに必要な確認だけを組み合わせて付けられます。

これから出てくる `requireAuth` と `requireRole` は、この仕組みを使った実装です。

## サーバーはどうやって「ログイン状態」を覚えるのか

パスワードの確認は1回のリクエストの中で完結しますが、ログイン状態はページを移動しても保たれる必要があります。

先ほど説明したとおり、HTTPには元々この記憶を保つ仕組みがありません。

そこで使われるのが**セッション**という仕組みです。

サーバーはログインに成功した時点で、そのユーザー専用の合言葉のような文字列（セッションID）を新しく作ります。

このセッションIDと、対応するユーザー情報の組をサーバー側に記録しておきます(この記録先をセッションストアと呼び、Redisなどのインメモリデータベースがよく使われます)。

そして、生成したセッションIDをブラウザに渡し、ブラウザはそれをクッキーとして保存します。

以降のリクエストでは、ブラウザが自動的にこのクッキーを一緒に送ってくれます。

サーバーは届いたセッションIDをセッションストアと照合し、対応するユーザー情報が見つかれば、そのリクエストを「そのユーザーからのもの」として扱います。

実装は次のようになります。

```javascript
// ログイン処理(サーバーサイド)
app.post('/login', async (req, res) => {
  const { email, password } = req.body;

  const user = await db.findUserByEmail(email);
  if (!user) return res.status(401).json({ error: '認証失敗' });

  const isValid = await bcrypt.compare(password, user.passwordHash);
  if (!isValid) return res.status(401).json({ error: '認証失敗' });

  // ランダムなセッションIDを生成
  const sessionId = crypto.randomBytes(32).toString('hex');

  // セッションストア(Redis等)に保存
  await sessionStore.set(sessionId, { userId: user.id, role: user.role });

  // セッションIDをクッキーで返す(HttpOnly + Secure + SameSite必須)
  res.cookie('sid', sessionId, {
    httpOnly: true, secure: true, sameSite: 'Lax', maxAge: 3600 * 1000
  });

  res.json({ success: true });
});
```

このコードには、初心者がつい見落とす実装の細部がいくつかあります。

1つ目は、認証失敗のメッセージを統一していることです。

「メールアドレスが存在しません」と「パスワードが違います」を分けて返すと、攻撃者にとって、入力したメールアドレスが登録済みかどうかの情報を与えてしまいます。

そのため、どちらの場合も同じ「認証失敗」という1つのメッセージだけを返します。

2つ目は、セッションIDを `crypto.randomBytes` という暗号的に安全な乱数生成器で作っていることです。

`Math.random()` は内部の計算手順が予測しやすいため、攻撃者にセッションIDを推測されるおそれがあります。

3つ目は、ログイン成功のたびに新しいセッションIDを生成していることです。

ログイン前のセッションIDをそのまま引き継ぐと、後で説明する「セッション固定攻撃」に弱くなります。

## 認可の実装、RBACという考え方

ログインしていることが確認できたら、次に「この操作をしてよいか」を確認します。

最もよく使われる方式が**RBAC**（ロールベースアクセス制御）です。

これは、ユーザーに「ロール」（admin、editor、viewerなど、役割を表す区分）を割り当て、ロールごとに許可する操作を決めておく方式です。

```javascript
// 認証ミドルウェア
const requireAuth = (req, res, next) => {
  const sessionId = req.cookies.sid;
  const session = await sessionStore.get(sessionId);
  if (!session) return res.status(401).json({ error: 'ログインが必要です' });
  req.user = session;
  next();
};

// 認可ミドルウェア
const requireRole = (role) => (req, res, next) => {
  if (req.user.role !== role) return res.status(403).json({ error: '権限がありません' });
  next();
};

// 使い方
app.get('/admin/users', requireAuth, requireRole('admin'), (req, res) => {
  res.json(getAllUsers());
});
```

`requireAuth` は「ログインしているか」だけを確認するミドルウェアです。

確認に成功すると、後続の処理が使えるように `req.user` にユーザー情報を入れて、次の処理に進みます。

`requireRole('admin')` は、すでにログイン済みであることを前提に、「そのユーザーのロールがadminかどうか」を確認するミドルウェアです。

この2つを順番に並べることで、「ログインしていて、かつadminロールを持つユーザーだけがこのエンドポイントに到達できる」という条件を表現できます。

ここで返しているステータスコードにも意味の違いがあります。

`401 Unauthorized` は「ログインそのものができていない」ことを表します。

`403 Forbidden` は「ログインはできているが、その操作をする権限がない」ことを表します。

この2つを区別せずに同じコードを返す実装をよく見かけますが、クライアント側が「ログイン画面に戻すべきか」「権限不足のメッセージを出すべきか」を判断するための材料になるため、区別して返す必要があります。

<Callout title="デフォルト拒否の原則">
  認可の設計は「明示的に許可したもの以外は禁止する」という方向で書きます。

  「管理者以外は弾く」という書き方ではなく、「adminロールを持つ者のみ通す」という書き方にします。

  ロールの種類が増えたとき、前者の書き方は新しいロールへの対応漏れが生じやすくなります。
</Callout>

## セッションIDはランダムなだけでは安全と言えない

セッションIDが推測困難なランダム値であることは前提として必要ですが、それだけでは安全とは言えません。

理由は、セッションIDの仕組みそのものにあります。

サーバーは「正しいセッションIDを持っているリクエスト」を、そのユーザー本人からのリクエストとして扱います。

つまり、セッションIDを盗まれると、攻撃者はパスワードを知らないままログイン状態を乗っ取れます。

この被害を**セッションハイジャック**と呼びます。

攻撃者がセッションIDを入手する経路は複数あります。

- XSS(クロスサイトスクリプティング。HttpOnly属性がなければ `document.cookie` で直接取得できる)
- 通信の盗聴(HTTPで通信していれば、セッションIDは暗号化されずに流れる)
- ログファイルにセッションIDがそのまま記録される設定ミス
- セッション固定攻撃(攻撃者が事前に用意したセッションIDを、被害者に使わせる攻撃)

セッションIDは、それを持っている者が本人として扱われる値なので、パスワードと同等の秘匿価値を持っています。

この事実から、保存場所の選び方も決まってきます。

JavaScriptから読み書きできるLocalStorageに保存すると、XSSで読み取られたときにそのまま盗まれます。

`HttpOnly`属性を付けたクッキーに保存すれば、JavaScriptからの読み取り自体ができなくなるため、XSS経由での盗難を防げます。

## ここで押さえること

- HTTPはステートレスであり、ログイン状態を維持するにはセッションのような追加の仕組みが必要になる
- パスワードは平文ではなくハッシュ化して保存する。bcryptやArgon2のように意図的に低速化されたハッシュ関数を使う。SHA-256のような汎用ハッシュ関数は高速すぎてブルートフォース攻撃に弱い
- 「認証」(相手が誰か)と「認可」(何をしていいか)は別の問題であり、両方を全エンドポイントで確認する
- セッションIDはパスワードと同等の秘匿価値を持つ。HttpOnlyクッキーで管理し、ログイン成功後は再発行する
- 認証失敗のエラーメッセージは、メールアドレスとパスワードのどちらが間違っているかを教えない
