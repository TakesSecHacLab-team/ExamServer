## フロントエンドを書いていたら突然このエラーが出た

```
Access to fetch at 'https://api.example.com/data' from origin
'http://localhost:3000' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

「コードを間違えたのか」と思うかもしれない。

違う。

このエラーは、ブラウザのセキュリティ機能が意図的に通信を止めた結果です。

これが**CORS**エラーです。

何が起きているかを機構として理解すれば、直し方も自然に決まる。

逆に意味を理解せず `Access-Control-Allow-Origin: *` を貼って済ませると、セキュリティホールを開けることがあります。

このページでは、HTTPの基本からオリジンの定義、Same-Origin Policy、CORS、プリフライトリクエストまで、土台から順番に組み立てます。

## HTTPのリクエストとレスポンスとは何か

CORSの話をする前に、ブラウザとサーバーがどうやって会話しているかを確認します。

ブラウザでWebサイトを開くとき、ブラウザは**HTTP**（HyperText Transfer Protocol）という通信規約に従って、サーバーにデータを要求します。

この要求を**リクエスト**と呼びます。

リクエストには「どのURLの、どんな種類の操作を求めるか」が書かれています。

```
GET /data HTTP/1.1
Host: api.example.com
```

この `GET` の部分が「メソッド」です。

データを取得したいときは `GET`、新しいデータを送って作成したいときは `POST`、データを削除したいときは `DELETE`、というように操作の種類を表します。

サーバーはリクエストを受け取ると、結果を**レスポンス**として返します。

```
HTTP/1.1 200 OK
Content-Type: application/json

{"name": "Alice"}
```

`200 OK` の部分は「ステータスコード」で、リクエストが成功したことを示します。

リクエストとレスポンスには、それぞれ「ヘッダー」という付加情報を付けられます。

`Content-Type: application/json` は、レスポンス本文がJSON形式であることを示すヘッダーです。

CORSに関わるヘッダーも、この仕組みの上に乗っています。

サーバーがレスポンスに特定のヘッダーを付けるかどうかで、ブラウザの振る舞いが変わる。

これがCORSの土台です。

## オリジンとは何か、なぜ3要素で決まるのか

ブラウザがWebページを表示するとき、そのページが「どこから来たか」を識別する単位が**オリジン**です。

オリジンは次の3つの組み合わせで決まります。

```
https://example.com:443  ← プロトコル: https、ホスト: example.com、ポート: 443
```

```
http://example.com:80    ← プロトコルが違う → 別オリジン
https://api.example.com  ← ホストが違う    → 別オリジン
https://example.com:8080 ← ポートが違う    → 別オリジン
```

3つのどれか一つでも違えば、別のオリジンとして扱われます。

なぜこの3要素なのか。

プロトコルが違うと、通信の暗号化や信頼性の前提が変わる。

`http://` は誰でも経路上で覗き見できる平文通信だが、`https://` は暗号化されている。

同じホスト名でも、片方が平文、片方が暗号化されていれば、攻撃者が平文側に成り代わって暗号化側のクッキーを盗む経路が生まれる。

ホストが違うと、運営者が違う可能性がある。

`example.com` と `evil.example.com` は文字列としては似ているが、DNSの委任構造やサーバーの管理者は別々になりうる。

ポートが違うと、同じマシン上で動く別のサービスを指すことになる。

同じサーバーマシンの8080番ポートで開発用のテストサービスが動いていて、443番ポートで本番の決済サービスが動いている場合、ポートまで区別しないと両者が同じ扱いになってしまう。

この3要素のどれかが変われば「別の管理主体、別の信頼境界」とみなす、という設計判断がオリジンの定義です。

## fetchとは何か

ブラウザ上のJavaScriptからHTTPリクエストを送る手段の一つが**fetch**です。

```javascript
fetch('https://api.example.com/data')
  .then(response => response.json())
  .then(data => console.log(data));
```

`fetch()` を呼ぶと、ブラウザは指定したURLにHTTPリクエストを送り、返ってきたレスポンスをJavaScriptで扱える形にして渡します。

ページに表示されているボタンを押した瞬間に裏側でデータを取りに行く、いわゆる「非同期通信」の大半はこの仕組みで動いています。

ここで注意したいのは、fetchが送るリクエスト自体は、サーバーに届いて処理され、レスポンスも実際に返ってくるという点です。

止まるのは、その先の「JavaScriptがレスポンスを読み取れるかどうか」です。

次の節で扱うSame-Origin Policyは、まさにこの「読み取り」の段階を制限します。

## Same-Origin Policyという基本ルール

ブラウザには**Same-Origin Policy**（同一オリジンポリシー、SOP）という基本ルールがあります。

```
あるオリジンで読み込まれたページのJavaScriptは、
別のオリジンのリソースを読み取れない。
```

なぜこのルールが必要なのか、具体的な場面で確認します。

ユーザーが `bank.com` にログインして、認証用のクッキーがブラウザに保存されているとします。

同じブラウザで別タブを開き、`evil.com` という悪意のあるサイトを訪れたとします。

`evil.com` のページに仕込まれたJavaScriptが、裏で `fetch('https://bank.com/api/balance')` を実行したとします。

ブラウザは「同じドメイン宛のリクエストにはクッキーを自動で付ける」という仕組みを持っているため、このリクエストにも `bank.com` の認証クッキーが自動的に付いて送られます。

つまりリクエスト自体は、ユーザー本人が `bank.com` にアクセスしたときと同じ認証状態でサーバーに届く。

SOPがなければ、`evil.com` のJavaScriptはこのレスポンス（残高データ）をそのまま読み取り、攻撃者のサーバーに送信できてしまいます。

SOPは、この「リクエストは通っても、別オリジンのJavaScriptはレスポンスを読み取れない」という制限をかけることで、この攻撃を防ぎます。

別オリジンへのJavaScriptからの読み取りを禁止するのが「SOP」の役割です。

## なぜこの制限はブラウザだけのものなのか

ここまでの説明で、SOPが「ブラウザの中で動くJavaScript」を対象にしていることに気づくかもしれません。

実際、この制限はブラウザだけが持つもので、サーバー同士の通信や、curlのようなコマンドラインツールには存在しません。

理由は、SOPが防いでいる攻撃の構造そのものにあります。

SOPが防ぐ攻撃は「ユーザーが信頼しているサイトAのクッキーを使って、ユーザーが今見ているサイトBのスクリプトが、サイトAのデータを盗み読む」という構図です。

この構図が成立するには、次の3条件が同時に揃う必要があります。

1台のブラウザに、複数のサイトの認証情報（クッキー）が同時に保存されている。

ユーザーが、信頼できないサイトのJavaScriptを実行する状態に置かれている(タブを開くだけで実行される)。

そのJavaScriptが、保存済みの認証情報を使って別サイトに自動でリクエストを送れる。

curlでAPIを呼ぶとき、この3条件は成立しません。

curlは複数サイトの認証情報を勝手に溜め込んだりしないし、悪意のある第三者のスクリプトを意図せず実行する状況も発生しません。

curlを実行する人は、自分が打ったコマンドの内容を把握しています。

サーバー間通信(あるサーバーのプログラムが別のサーバーのAPIを呼ぶ場合)も同様です。

呼び出す側のプログラムは、開発者が明示的に書いたコードであり、ユーザーが意図せず実行させられた第三者のスクリプトではありません。

ブラウザだけが「無関係な第三者が書いたJavaScriptを、ユーザーの認証情報を保持した状態で、ユーザーの同意なく実行してしまう」という特殊な環境にある。

この環境特有のリスクに対応するための仕組みが、SOPであり、その先にあるCORSです。

## CORSという例外を認める仕組み

SOPは安全だが、これだけでは正当な通信もできなくなってしまいます。

フロントエンドを `localhost:3000` で動かし、APIを `api.example.com` で動かす、という構成は開発の現場でよくあります。

この2つは別オリジンなので、SOPのままでは読み取りがブロックされ続けます。

別オリジンとの通信を、サーバー側の意思で許可できるようにする仕組みが**CORS**(Cross-Origin Resource Sharing)です。

CORSは、サーバーがレスポンスに特定のヘッダーを付けることで、「このオリジンからの読み取りを許可する」と宣言する仕組みです。

```
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://myapp.example.com
```

ブラウザはレスポンスを受け取ると、このヘッダーを確認します。

リクエスト元のオリジンが `Access-Control-Allow-Origin` の値と一致していれば、JavaScriptにレスポンスを渡す。

一致しない、またはヘッダー自体が無ければ、ブロックする。

CORSの判断そのものを行うのはブラウザですが、何を許可するかを決めるのはサーバー側です。

サーバーが適切なヘッダーを返すかどうかで、結果がすべて決まります。

## プリフライトリクエストという事前確認

単純なGETやPOSTのリクエストなら、レスポンスに `Access-Control-Allow-Origin` が付いているかどうかをブラウザが確認するだけで済みます。

ただし、リクエストの内容が一定の条件を超えると、本リクエストの前にブラウザが「この操作をしてよいか」をサーバーに確認する、**プリフライトリクエスト**という手順が入ります。

プリフライトリクエストは `OPTIONS` というメソッドで送られます。

```
OPTIONS /api/data HTTP/1.1
Origin: https://myapp.example.com
Access-Control-Request-Method: DELETE
Access-Control-Request-Headers: Content-Type, Authorization
```

このリクエストには本来送りたい操作の情報、つまり「これから `DELETE` メソッドで `Content-Type` と `Authorization` ヘッダーを付けたリクエストを送りたい」という予告が含まれます。

サーバーは、この予告内容を許可するかどうかを判断し、レスポンスを返します。

```
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://myapp.example.com
Access-Control-Allow-Methods: GET, POST, DELETE
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 86400
```

許可されていれば、ブラウザは続けて本来のリクエスト(本リクエスト)を送ります。

プリフライトが必要になる条件は、次のいずれかに当たる場合です。

`GET`、`POST`、`HEAD` 以外のメソッドを使う場合(`PUT` や `DELETE` など)。

`Content-Type: application/json` のような、単純リクエストの範囲を超えるヘッダーを付ける場合。

`Authorization` ヘッダーなど、カスタムヘッダーを送る場合。

`Access-Control-Max-Age: 86400` は、このプリフライトの結果を86400秒(24時間)キャッシュしてよい、という指示です。

これがないと、同じ種類のリクエストを送るたびに毎回OPTIONSが往復することになり、通信回数が増えてレスポンスが遅くなります。

`Access-Control-Allow-Credentials` は、クッキーや認証ヘッダーを含むリクエストをCORSで許可したい場合に必要なヘッダーです。

このヘッダーを使う場合、`Access-Control-Allow-Origin` にワイルドカード(`*`)を指定することはできません。

具体的なオリジンを1つ指定する必要があります。

理由は単純です。

ワイルドカードは「どのオリジンでも読み取ってよい」という宣言であり、これと「認証情報付きで読み取ってよい」を組み合わせると、世界中のどのサイトからでもユーザーの認証済みデータを読み取れる状態になってしまうからです。

## Node.jsでのCORS設定例

実装側でどう書くかを確認します。

Express(Node.jsのWebフレームワーク)では、`cors` というライブラリを使うのが典型的です。

```javascript
const cors = require('cors');

// ワイルドカード(注意して使う)
app.use(cors({ origin: '*' }));

// 特定オリジンのみ許可
app.use(cors({
  origin: ['https://myapp.example.com', 'http://localhost:3000'],
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  allowedHeaders: ['Content-Type', 'Authorization'],
  credentials: true,  // クッキーを送る場合に必要
  maxAge: 86400
}));
```

`credentials: true` を指定する場合、`origin` には具体的なオリジンの配列を渡す必要があります。

`'*'` を指定すると、ライブラリ側でエラーになるか、ブラウザ側で拒否されます。

## curlとブラウザはなぜ違う振る舞いをするのか

CORSに関する誤解で多いのが、「CORSをサーバー側で制限している」「サーバーがCORSをブロックしている」という説明です。

これは実際の機構とは逆です。

CORSの判断、つまり「レスポンスをJavaScriptに渡すか、渡さずにエラーにするか」を行うのはブラウザです。

サーバーが行っているのは、許可する条件をヘッダーで宣言することだけです。

この違いは、curlで同じURLを叩くと確認できます。

```bash
# これはエラーにならない(curlはCORSという仕組みを持たない)
curl -H "Origin: https://evil.com" https://api.example.com/data

# ブラウザ上のJavaScriptからは、CORSヘッダーがなければ読み取りがブロックされる
fetch('https://api.example.com/data')
```

curlは、サーバーから返ってきたレスポンスをそのまま端末に表示するだけのツールです。

「別オリジンのJavaScriptから読み取られる」という状況自体が、curlには存在しません。

curlを実行しているのは開発者自身であり、レスポンスを読むのも開発者自身だからです。

したがって、「サーバーがCORSを拒否している」という言い方は正確ではありません。

正確には、「サーバーがそのオリジンへの許可を宣言していないため、ブラウザがレスポンスの読み取りを拒否している」という状態です。

この区別が実務で意味を持つのは、`Access-Control-Allow-Origin: *` を付けたときに何が起きるかという場面です。

このヘッダーが解除するのは、あくまでブラウザ側の読み取り制限だけです。

サーバー自体のアクセス制御や、APIが誰からのリクエストでも処理してしまうという問題は、CORSヘッダーとは別の話です。

APIを保護する仕組みは認証と認可であり、CORSはその代わりにはなりません。

## ここで押さえること

**HTTP**：ブラウザとサーバーが「リクエスト」と「レスポンス」をやり取りする通信規約。CORSのヘッダーもこのリクエストとレスポンスの上に乗っている。

**オリジン**：プロトコル、ホスト、ポートの組み合わせで決まる識別単位。3要素のどれかが違えば、運営主体や信頼境界が異なるとみなされる。

**Same-Origin Policy(SOP)**：別オリジンのJavaScriptによる読み取りを禁止するブラウザの基本ルール。ユーザーの認証情報を悪用した盗み読みを防ぐ。

**この制限がブラウザだけにある理由**：複数サイトの認証情報を保持したまま、無関係な第三者のスクリプトを意図せず実行してしまう、という状況がブラウザ特有だから。curlやサーバー間通信にはこの状況自体が存在しない。

**CORS**：SOPの例外を、サーバー側の宣言によって認める仕組み。`Access-Control-Allow-Origin` などのヘッダーで許可するオリジンを示す。

**プリフライトリクエスト**：単純リクエストの範囲を超える操作の前に、ブラウザが `OPTIONS` で事前確認する手順。`Access-Control-Max-Age` で結果をキャッシュできる。

**curlとブラウザの違い**：CORSの判断を行うのはブラウザだけであり、curlやサーバー間通信にはこの判断自体が存在しない。
