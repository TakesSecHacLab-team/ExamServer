## ブラウザの鍵マークは、何を保証しているのか

URL バーに鍵のマークがある。`https://` で始まるサイトです。「安全なサイト」という印象を持つ人は多いです。

でも、その理解は不正確です。

その鍵マークが保証しているのは「通信経路が暗号化されている」ただそれだけ。サイト自体の正当性、コンテンツの安全性、そのサイトがあなたを詐欺から守ってくれるかは、全然別の問題です。

フィッシングサイト、詐欺広告サイト、違法コンテンツのサイトも、ほぼ全部 https を使ってます。Let's Encrypt で無料に証明書がもらえる今、鍵マークは「通信が見られないようにされてる」という意味であって、「このサイトが信頼できる」という意味ではないんです。

ではその鍵マークは何のためにあるのか。そこから入ります。

## TLS とは

TLS（Transport Layer Security）は、インターネット上の**通信を暗号化する**プロトコルです。

昔は SSL（Secure Sockets Layer）と呼ばれていました。今は TLS に統一されていますが、業界ではまだ「SSL」と言われることも多いです。「SSL/TLS」と一緒に書くのは、そのためです。

何を暗号化するのか。あなたのブラウザが送った HTTP リクエストと、サーバーからの HTTP レスポンス——つまり**通信の中身**です。

TLS なしの HTTP では、あなたのログインパスワードやカード番号は、ネットワーク上を**平文**（ひらぶん）で流れます。通信を傍受する中間者（中間マン）からは丸見えです。TLS でその通信を暗号化すれば、傍受されても内容は読めない。

## TLS ハンドシェイク——鍵を交換する流れ

ブラウザがサーバーにアクセスしようとするとき、いきなり暗号化通信が始まるわけじゃありません。まず「これから暗号化通信をしますよ」という合意をします。その過程を **TLS ハンドシェイク**と呼びます。

深い数学には入りませんが、大まかな流れは：

**1. クライアント（ブラウザ）がサーバーへ "Hello" を送る**

「私は TLS 1.3 に対応してます」「こういう暗号化方式を使えます」という情報を伝えます。

**2. サーバーが "Hello" を返す**

「わかりました。そしたらこの暗号化方式を使いましょう」と返答。サーバーの **証明書**も一緒に送ります。

**3. 鍵交換**

ブラウザとサーバーが、以降の通信で使う**共通鍵**（秘密鍵ではない、お互いに知っている鍵）を安全に交換します。ここまでくると暗号化の準備ができた。

**4. 確認**

「これからこの鍵で暗号化して話しますね」と相互に確認。

この時点で、以降のすべての HTTP リクエスト・レスポンスが暗号化されます。

## 証明書の役割——「この鍵はたしかに example.com のもの」

ハンドシェイクの途中でサーバーが **証明書**を送ります。何のためか。

試しに、悪い人が自分のサーバーを立てて「俺が example.com だ」と名乗ったとします。暗号化通信は成立します。でも、あなたのブラウザはそれが本物の example.com なのか偽物なのか区別できません。ここで困ります。

証明書が解決する問題は：「この公開鍵はたしかに example.com のものであり、偽物ではない」という保証です。

証明書に書いてあるのは：

- **ドメイン名**：このサーバーが担当する URL（`example.com`）
- **公開鍵**：ハンドシェイクで使用する暗号化用の鍵
- **発行者**（Issuer）：誰がこの証明書を発行したか
- **有効期限**：いつまで有効か
- **署名**：発行者による電子署名（改ざんがないことを証明）

発行者とは、**CA**（認証局、Certificate Authority）のことです。Comodo、DigiCert、Let's Encrypt など、信頼された第三者機関がこれに当たります。

あなたのブラウザは、その CA を信頼しているので、「CA が署名した証明書なら、その内容は間違いない」と考えます。だから、偽のサーバーが偽の証明書を送ってきても、ブラウザは「この署名は知らない CA のもの」「不正だ」と警告を出す。

## 中間者攻撃（MITM）と TLS

TLS がなぜ必要かを理解するには、**中間者攻撃**（Man-In-The-Middle attack、MITM）を知る必要があります。

公共 Wi-Fi であなたのブラウザが通信しているとき、同じ Wi-Fi に繋がっている悪い人が、あなたのトラフィックを傍受できます。HTTP（暗号化なし）なら、内容は丸見え。ログインパスワードもカード番号も読める。

それだけじゃなく、悪い人が自分をサーバーになりすまして、あなたのブラウザへ返答を送ることもできます。あなたは本物のサーバーと通信していると思ってるのに、実は中間者経由で通信している——それが中間者攻撃です。

TLS でこれを防ぐ方法：

- **暗号化**で、通信内容を読めなくする。傍受されても意味がない
- **証明書による認証**で、サーバーが本物かどうかを検証する。中間者が偽の証明書を送っても、ブラウザが警告を出す

つまり TLS は「通信路を安全にする」技術であって、サイト自体の正当性まで保証しているわけではありません。

<Callout title="https のサイト = 安全？">
  フィッシングサイトも、ほぼ全部 https です。詐欺師も TLS を使います。なぜなら証明書取得が無料だから（Let's Encrypt）。あなたのパスワードを盗むために作られたサイトですら、その盗んだパスワードを「安全に」自分のサーバーへ送信したいので、https を使う。だから鍵マークがあるからといって「このサイトは信頼できる」と判断してはいけません。重要なのは「URL は本当に公式のドメインか」「ブラウザの警告が出てないか」「その先で求められている情報は本当に必要か」という人間の判断です。
</Callout>

## HSTS——ブラウザを http から強制的に https へ

ブラウザのアドレスバーに `http://example.com` と入力する人がいます。サーバーはそれを `https://example.com` へリダイレクトするのが一般的です。ただ、その最初の 1 回のリクエストは http で流れます。その間に中間者が介入することもありえます。

対策として、サーバーは **HSTS**（HTTP Strict Transport Security）というヘッダーを返します。

```
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

このヘッダーをもらったブラウザは「このドメインへのアクセスは絶対に http で送らず、https だけ使う」と記憶します。次回以降、ユーザーが `http://example.com` を入力しても、ブラウザが勝手に `https://` に書き換える。最初のリクエストすら http で流さない。

これで「http だから傍受できる」という穴が塞がります。

## 証明書チェーンと認証局（CA）

ブラウザに組み込まれた**ルート証明書**は、信頼された大手 CA（Comodo、Verisign、Let's Encrypt など）のものです。ブラウザはこれらを「絶対に信頼する」と決めています。

でも、すべてのサーバー証明書に直接署名していたら、ルート CA の秘密鍵が漏洩したときに大惨事です。だから **中間 CA** を作ります。

- **ルート CA**：絶対に信頼される。秘密鍵はめちゃくちゃ厳重に保管
- **中間 CA**：ルート CA が認可した下級の CA。普段の署名はここで行う
- **サーバー証明書**：中間 CA が署名したサーバーの証明書

ブラウザが `example.com` のサーバーにアクセスするとき、サーバーは「私の証明書」と「中間 CA の証明書」を両方送ります。ブラウザは：

1. サーバー証明書の署名を、中間 CA の公開鍵で検証
2. 中間 CA の証明書の署名を、ルート CA の公開鍵で検証
3. ルート CA はブラウザに組み込まれているので、信頼する

この「信頼の連鎖」を **証明書チェーン**（certificate chain）と呼びます。

Let's Encrypt は無料で証明書を発行する CA ですが、ISRG がルート CA で、Let's Encrypt が中間 CA として機能しています。無料だからといって信頼性が劣るわけではなく、ブラウザに組み込まれた ISRG ルート証明書で検証されます。

## ここで押さえること

- **TLS**（旧 SSL）は通信を暗号化するプロトコル。サーバーの正当性を保証するものではない
- **証明書**は「このサーバーが本物であることを CA が保証する」という電子署名。中間者攻撃から守る
- **ハンドシェイク**で鍵を交換し、その後のすべての通信が暗号化される
- https（TLS）の鍵マークは「通信が暗号化されている」を示すだけ。サイトが詐欺でないことは保証しない
- **HSTS** ヘッダーにより、ブラウザは http 接続を避け、常に https を使う。最初のリクエストでの MITM も防ぐ
- **証明書チェーン**でルート CA → 中間 CA → サーバー証明書と信頼を検証。Let's Encrypt も同じ仕組みで信頼される
