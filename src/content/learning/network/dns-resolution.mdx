## ブラウザに名前を打ってからページが出るまでに起きていること

ブラウザのアドレス欄に `example.com` と打って Enter を押す。
ページが表示されるまでの間に、ネットワークの中で短い問い合わせが一往復済んでいます。
かかる時間は数十ミリ秒、まばたき一回より短いくらいです。

ここでまず確認しておきたいのは、コンピュータが通信するときに実際に使っている宛先の形です。
コンピュータ同士の通信は、`93.184.216.34` のような数字の並びを宛先として届けています。
これを **IPアドレス**と呼びます。
IPアドレスについては次の章で詳しく扱うので、ここでは「コンピュータが使う宛先の数字」という理解で十分です。

人間が `example.com` という名前でサイトを覚えているのに対して、コンピュータが実際に使っているのは数字のIPアドレスです。
名前と宛先が別ものになっている。
この名前から実際の宛先を引き出す仕組みが **DNS**（Domain Name System、ドメインネームシステム）です。

## 名前と宛先を分けたのはなぜか

「最初から数字の宛先を覚えておけばよいのでは」という疑問は自然です。
実際、1970年代後半のインターネットはそうなっていました。
`hosts.txt` という1枚のテキストファイルに、すべてのコンピュータの名前と宛先が書かれていて、管理者が手元にコピーして使っていたのです。

コンピュータの台数が数十台のうちは、この方式で困りませんでした。
台数が数万に増えると、1枚のファイルがどんどん巨大になり、更新が追いつかなくなりました。
1983年にDNSが設計されたのは、この問い合わせを複数のサーバーに分散して処理する仕組みが必要だったからです。

分けておく理由はもう一つあります。
サーバーを別のデータセンターへ引っ越すと、IPアドレスは変わります。
名前を固定して宛先だけを変えれば、利用者はURLを変えずに済みます。
名前と宛先を分離しておくことで、運用側の都合とサイト名を分けて扱えるようになっています。

## 問い合わせを代わりに出してくれる係、リゾルバ

PCやスマートフォンが `example.com` のIPアドレスを知りたいとき、利用者の端末が直接調べに行くわけではありません。
代わりに問い合わせを担当する係がいます。
これを **リゾルバ**（DNSリゾルバ、フルサービスリゾルバ）と呼びます。

リゾルバを提供しているのは、契約しているISP(インターネットサービスプロバイダ)や会社のネットワーク機器、あるいはCloudflareの `1.1.1.1` やGoogleの `8.8.8.8` のような公開DNSサービスです。
家庭のルーターが「どのリゾルバを使うか」をPCへ自動で伝えているので、利用者が手動で設定しなくても動いています。

<Callout title="今使っているリゾルバを確認する">
  Windowsのコマンドプロンプトで `ipconfig /all` と打つと、「DNS サーバー」という項目にIPアドレスが表示されます。
  これが今使っているリゾルバのアドレスです。
  普段意識しない存在ですが、確認すると身近な機器が動いているとわかります。
</Callout>

## 問い合わせがたどる道筋

リゾルバが `example.com` のIPアドレスをまだ知らないとき、どこに尋ねればよいのでしょうか。
答えは1か所にまとまっていません。
インターネット規模の名前を1台のサーバーで管理するのは現実的ではないので、DNSは**階層**に分けて管理されています。

問い合わせの順番を追っていきます。

最初にリゾルバは **ルートサーバー**（root name server）へ問い合わせます。
ルートサーバーは世界に13系統あり、DNSの名前空間の頂点に位置しています。
ルートサーバーはIPアドレスを直接教えてくれません。
代わりに「`.com` についてはTLDサーバーへ聞いてください」と次の行き先を返します。

次にリゾルバは **TLDサーバー**（トップレベルドメインサーバー）へ問い合わせます。
`.com` を管理しているのはVerisignという企業です。
TLDサーバーも直接の答えは返さず、「`example.com` については権威DNSサーバーへ聞いてください」と返します。

最後にリゾルバは **権威DNSサーバー**(authoritative name server)へ問い合わせます。
権威DNSサーバーは、そのドメインについて公式に答えられる唯一のサーバーです。
ここでようやく `example.com → 93.184.216.34` という答えが返ってきます。

リゾルバはこの答えを利用者のPCへ渡し、ブラウザはそのIPアドレスへ接続します。

この一連の仕組みを **再帰問い合わせ**(recursive query)と呼びます。
利用者のPCからは「1回問い合わせたら答えが返ってくる」だけに見えますが、裏側ではリゾルバが「ルートサーバー、TLDサーバー、権威DNSサーバー」と順番に問い合わせを重ねています。
クライアント側がこの連鎖を意識しなくて済むのが、再帰問い合わせの仕組みです。

## レコードの種類で変わる、DNSが返す情報

DNSが返すのはIPアドレスだけではありません。
返す情報の種類を **レコードタイプ**と呼び、用途ごとに使い分けます。

**Aレコード**は最も基本的なレコードタイプで、IPv4アドレスを返します。
`example.com → 93.184.216.34` のような変換がこれに当たります。

**AAAAレコード**(クワッドエー)はIPv6アドレスを返します。
`2606:2800:220:1:248:1893:25c8:1946` のような形です。
IPv6に対応したネットワークではAレコードとAAAAレコードの両方が設定されていることが多く、接続できる方が選ばれます。

**CNAMEレコード**は別名(エイリアス)の定義です。
`www.example.com` を `example.com` の別名として指定するときに使います。
CNAMEレコード自体はIPアドレスを持たず、最終的にはAレコードかAAAAレコードまで解決されます。

**MXレコード**はメールサーバーを指定します。
`example.com` 宛にメールを送るとき、どのサーバーが受け取るかをこのレコードで定義します。
DNSはウェブの名前解決だけでなく、メールの配送先を決める場面でも使われています。

**TXTレコード**は任意のテキストを返します。
ドメインの所有確認や、スパム対策のメール認証(SPF、DKIM)に広く使われています。
一見IPアドレスに無関係なのにDNSが使われるのは、「そのドメインの管理者だけが書き換えられる場所」としてDNSが信頼されているためです。

## TTLとキャッシュ、問い合わせを毎回繰り返さない仕組み

毎回ルートサーバーから辿っていたら、1回の名前解決のたびに何往復もの通信が発生します。
実際には、一度得た答えを **キャッシュ**(一時保存)します。
利用者のPC、リゾルバ、ブラウザのそれぞれが、それぞれのキャッシュを持っています。

キャッシュをいつ捨てるかを決めるのが **TTL**(Time To Live、生存時間)です。
秒単位で設定されます。
DNSレコードに `TTL: 300` と書かれていれば、300秒(5分)後に古い情報を捨てて再び問い合わせます。

TTLを小さくすれば、変更が早く反映されるようになります。
TTLを大きくすれば、問い合わせの回数が減り、リゾルバとサーバーの負荷が下がります。

エンジニアがサーバーを移転するとき、「DNSの浸透に時間がかかる」という言い方をよく聞きます。
この「浸透」という表現は技術的には正確ではありません。
実際に起きているのは「TTLが切れるまで古いIPアドレスがキャッシュに残っている」という、**キャッシュの消化待ち**です。
移転前にTTLを短く(たとえば60秒に)設定しておけば、切り替え後の反映が速くなります。

<Callout title="キャッシュを確認してクリアする">
  Windowsでは `ipconfig /displaydns` でリゾルバのキャッシュ一覧を確認でき、`ipconfig /flushdns` でキャッシュを消せます。
  「サイトを新しいサーバーへ移したのにつながらない」というときは、まずここを確認します。
  ブラウザ自身も別のキャッシュを持っているので、それでも直らないときは開発者ツールでブラウザキャッシュも確認します。
</Callout>

## キャッシュがあるからこそ起きる問題

キャッシュは便利な仕組みですが、攻撃の対象になることがあります。

**DNSキャッシュポイズニング**(DNS cache poisoning)は、リゾルバのキャッシュに偽の情報を混入させる攻撃です。
`bank.example.com` のIPアドレスを攻撃者のサーバーのものに書き換えられると、利用者が正しいURLを入力しても偽サイトへ誘導されてしまいます。
もともとのDNSの設計には、返ってきた答えが本物かどうかを検証する仕組みがありませんでした。
これがキャッシュポイズニングを可能にしている根本の原因です。

対策として設計されたのが **DNSSEC**(DNS Security Extensions)です。
DNSレコードに電子署名を付けることで、改ざんがないことを検証できるようにした仕組みです。
重要なドメインでは有効化が進んでいますが、すべてのドメインに普及しているわけではありません。

キャッシュポイズニングとDNSSECの詳細は、セキュリティの章で改めて扱います。
ここでは「DNSの回答は、何もしなければ改ざんを検出できない仕組みである」という点を覚えておけば十分です。

## ここで押さえること

- 名前(`example.com`)と通信先(IPアドレス)は別ものであり、DNSがその変換を担う
- **リゾルバ**が代わりに問い合わせを追いかける。クライアント側は1回聞けば答えが返る(**再帰問い合わせ**)
- DNSはレコードタイプごとに返す情報が違う(A、AAAA、CNAME、MX、TXTなど)
- 返答は **TTL** で指定した時間だけキャッシュされる。「DNSの浸透待ち」の実体はキャッシュの消化待ち
- キャッシュは攻撃の対象になりうる(**キャッシュポイズニング**)。対策の仕組みが **DNSSEC**
