## 0.5 秒の中で起きていること

ブラウザに `example.com` を入力して Enter を押す。ページが表示されるまでの間、ほんの数十ミリ秒の中で一往復の問い合わせが済んでいます。

コンピュータ同士が通信するとき、宛先として必要なのは IP アドレスです。`93.184.216.34` のような数字の列です。でも人間は `example.com` という名前でサイトを覚えています。名前と住所が分かれているのです。

この変換——名前から IP を引く仕組み——が **DNS**（Domain Name System）です。

## なぜ名前と住所が分かれているのか

「最初から IP アドレスを覚えればいいのでは」という疑問はもっともです。実際、初期のインターネット（1970 年代後半）はそうでした。`hosts.txt` という 1 枚のファイルに全コンピュータの名前とアドレスが書いてあって、管理者がコピーして使っていました。

コンピュータが数十台のうちは問題なかった。何万台になるとファイルが巨大になり、更新が追いつかなくなりました。1983 年、DNS が設計されたのはこのためです。問い合わせを分散して処理できる仕組みを作る必要があった。

もう一つ理由があります。IP アドレスはサーバーの引っ越しで変わります。名前を固定して住所だけ変えれば、利用者は URL を変えずに済む。この分離は、現実の運用でとても重要です。

## リゾルバ——問い合わせを代わりに出す係

あなたの PC やスマホが `example.com` の IP を知りたいとき、直接調べにいくわけではありません。代わりに問い合わせてくれる係がいます。これを **リゾルバ**（DNS リゾルバ、またはフルサービスリゾルバ）と呼びます。

リゾルバは ISP（インターネットプロバイダ）や会社のネットワーク、または Cloudflare の `1.1.1.1`・Google の `8.8.8.8` のような公開 DNS サービスが担っています。あなたのルーターが「どのリゾルバを使うか」を教えているので、PC には自動で設定されています。

<Callout title="今使っているリゾルバを確認する">
  Windows のコマンドプロンプトで `ipconfig /all` と打つと「DNS サーバー」の項目が表示されます。そこに出てくる IP がリゾルバのアドレスです。普段意識しない存在ですが、確認してみると思ったより身近に動いています。
</Callout>

## 問い合わせが進む道筋

リゾルバが `example.com` の IP を知らないとき、どこへ聞くのか。答えは一か所にまとまっていません。インターネット規模の名前を一か所で管理するのは現実的でないので、**階層に分けて管理**されています。

順番に追ってみます。

まずリゾルバは **ルートサーバー**（root name server）へ問い合わせます。ルートサーバーは世界に 13 系統あり、名前空間の頂上にいます。ルートサーバーは IP アドレスを教えてくれません。代わりに「`.com` については TLD サーバーへ聞いてください」と次の行き先を返します。

次にリゾルバは **TLD サーバー**（トップレベルドメインサーバー）へ問い合わせます。`.com` を管理しているのは Verisign という企業です。TLD サーバーも直接の答えは返さず、「`example.com` については権威 DNS サーバーへ聞いてください」と返します。

最後にリゾルバは **権威 DNS サーバー**（authoritative name server）へ問い合わせます。権威 DNS サーバーとは「そのドメインについて公式に答えられる唯一のサーバー」です。ここで `example.com → 93.184.216.34` という答えが返ってきます。

リゾルバはこの答えをあなたの PC へ渡す。ブラウザはその IP へ接続する。

## レコードの種類

DNS が返すのは IP アドレスだけではありません。情報の種類を **レコードタイプ**と呼び、用途に応じて使い分けます。

**A レコード**は最も基本で、IPv4 アドレスを返します。`example.com → 93.184.216.34` のような変換がこれです。

**AAAA レコード**（クワッド A）は IPv6 アドレスを返します。`2606:2800:220:1:248:1893:25c8:1946` のような形です。IPv6 に対応したネットワークでは A と AAAA の両方が設定されていることが多く、接続できる方が使われます。

**CNAME レコード**は別名（エイリアス）の定義です。`www.example.com` を `example.com` の別名として指定するとき使います。直接 IP を持つわけではなく、最終的には A または AAAA まで解決されます。

**MX レコード**はメールサーバーを指定します。`example.com` 宛にメールを送るとき、どのサーバーが受け取るかをここで定義します。DNS はメールの配送にも使われています。

**TXT レコード**は任意のテキストを返します。ドメインの所有確認や、スパム対策のメール認証（SPF・DKIM）に広く使われています。一見 IP に無関係なのに DNS を使うのは、「そのドメインの管理者だけが書き換えられる」場所として DNS が信頼されているためです。

## TTL とキャッシュ

毎回ルートサーバーから辿っていたら、問い合わせのたびに何往復も発生します。実際には、一度得た答えを **キャッシュ**（一時保存）します。あなたの PC、リゾルバ、ブラウザのそれぞれがキャッシュを持ちます。

キャッシュをいつ捨てるかを決めるのが **TTL**（Time To Live、生存時間）です。秒単位で設定します。DNS レコードに `TTL: 300` とあれば、300 秒（5 分）後に古い情報を捨てて再問い合わせします。

TTL を小さくすれば変更が早く反映されます。大きくすれば問い合わせ回数が減り、リゾルバとサーバーの負荷が下がります。

エンジニアがサーバーを引っ越すとき「DNS の浸透に時間がかかる」という言い方をよく聞きます。正確には「TTL が切れるまで古い IP がキャッシュされている」ということです。「浸透」という表現は技術的に正確ではなく、実態は**キャッシュの消化待ち**です。引っ越し前に TTL を短く（たとえば 60 秒）設定しておくと、切り替え後の反映が早くなります。

<Callout title="キャッシュを確認・クリアする">
  Windows では `ipconfig /displaydns` でリゾルバのキャッシュ一覧を確認し、`ipconfig /flushdns` でキャッシュを消せます。「サイトを新しいサーバーへ移したのにつながらない」というとき、まずここを確認します。ブラウザ自身がキャッシュを持っていることもあるので、それでもだめなら開発者ツールでブラウザキャッシュも確認します。
</Callout>

## キャッシュがあるからこそ起きる問題

便利なキャッシュですが、攻撃対象になることがあります。

**DNS キャッシュポイズニング**（DNS cache poisoning）は、リゾルバのキャッシュに偽の情報を混入させる攻撃です。`bank.example.com` の IP を攻撃者のサーバーのものに書き換えられると、利用者が正しい URL を入力しても偽サイトへ誘導されます。もともとの DNS 設計では、返答の真正性を検証する仕組みがなかったことが根本原因です。

対策として設計されたのが **DNSSEC**（DNS Security Extensions）です。DNS レコードに電子署名を付けて、改ざんがないことを検証できるようにした仕組みです。重要なドメインでは有効化が進んでいますが、完全に普及しているとは言えない状況です。

キャッシュポイズニングや DNSSEC の詳細は、セキュリティ基礎の章で改めて扱います。ここでは「DNS の回答はデフォルトでは改ざんを検出できない仕組みである」という認識を持っておくだけで十分です。

## ここで押さえること

- 名前（`example.com`）と通信先（IP アドレス）は別もの。DNS がその変換を担う
- **リゾルバ**が代わりに問い合わせを追う。クライアントは 1 回聞けば答えが返る（**再帰問い合わせ**）
- DNS はレコードタイプごとに返す情報が違う（A・AAAA・CNAME・MX・TXT など）
- 返答は **TTL** で指定した時間だけキャッシュされる。「DNS の浸透待ち」の正体はキャッシュの消化待ち
- キャッシュは攻撃対象になりうる（**キャッシュポイズニング**）。対策の仕組みが **DNSSEC**
