## ブラウザのDevToolsを開くと見える通信の中身

ChromeやEdgeを開いてF12キーを押す。
「Network」タブをクリックする。
そこに並ぶ一覧が、見ているページの裏側で実際に行われている通信そのものです。

一覧の1行をクリックすると、「Headers」「Request」「Response」といったタブが現れます。
数字や英単語が並んでいるだけに見えますが、これがそのまま **HTTP**(Hypertext Transfer Protocol)という、ウェブの通信ルールの内容です。

難しそうに見えるのは、内容が普段は表に出ていないからです。
仕組みがわかれば、DevToolsの表示は「この情報をやり取りしているのか」と読めるようになります。

## HTTPの正体、テキストで書かれた通信の指示

HTTPの仕組みは、見た目より単純です。

ブラウザが `https://example.com/page` を開いたとき、サーバーへ何を送っているのか。
テキストで書くと次のようになります。

```
GET /page HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
```

これで全部です。
**HTTPリクエスト**とは、テキストで書かれた指示です。
「このサーバーへ」「この情報をもらいたい」「こういう条件で」という内容を、文章として送っています。

サーバーが返す内容も、同じくテキストです。

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1234

<!DOCTYPE html>
<html>...
```

成功か失敗かを示す番号(200)、データの種類(Content-Type)、最後に実際のページの内容(本文)まで、全部テキストとして書かれています。

ブラウザがこのテキストを自動で作り、自動で読んでいるので、利用者はこの形式を意識しません。
通信の中身を理解するうえでは、このテキスト形式を読めることが前提になります。

## リクエストの構造、3つの部分

HTTPリクエストは3つの部分に分かれています。

**1. リクエストライン**。
やりたいことを1行で書きます。

```
GET /page HTTP/1.1
```

`GET` が **メソッド**です。
「サーバーから情報をもらう」という意味で、ブラウザがページを見るときの大半はこのメソッドを使います。
よく使うメソッドは次のとおりです。

- **`GET`**: データをもらう
- **`POST`**: データをサーバーへ送る(フォーム送信、ログインなど)
- **`PUT`**: 既存のデータを丸ごと上書きする
- **`DELETE`**: データを削除する
- **`PATCH`**: データの一部だけを変更する

`/page` はサーバーのどこにアクセスするかを示す **パス**です。
`HTTP/1.1` はプロトコルのバージョンです。

**2. ヘッダー**。
リクエストに関する追加情報を、1行に1つ「名前: 値」の形式で書きます。

```
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Content-Type: application/json
Authorization: Bearer token_here
```

よく出てくるヘッダーは次のとおりです。

- **`Host`**: どのサーバー宛か。1つのIPアドレスを複数のサイトが共有していることもあるので、ここで指定する
- **`User-Agent`**: どのブラウザやツールから来たか
- **`Accept`**: どの形式の返答が欲しいか(HTMLかJSONか)
- **`Content-Type`**: リクエストの本文が何の形式か
- **`Authorization`**: 認証情報。ログインが必要な場合、ここにトークンを入れる

ヘッダーは何個でも追加できます。
キャッシュの指示、独自のデータ、認証情報など、通信に必要な指示はすべてここに書きます。

**3. ボディ**。
データを送る場合だけ存在します。
`GET` では通常ボディは空です。
`POST` でフォームを送信するときや、JSONデータを送るときは、ここにデータが入ります。

```
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json

{"name": "Taro", "email": "taro@example.com"}
```

この例では、最後のJSONがボディです。

## レスポンスの構造

サーバーからの返答も、同じくテキスト形式です。

**1. ステータスライン**。
通信が成功したか失敗したかを示す数字です。

```
HTTP/1.1 200 OK
```

`200` は「うまくいった」という意味の **ステータスコード**です。

`200`番台は成功、`300`番台はリダイレクト、`400`番台はクライアントのミス、`500`番台はサーバーのミスという分け方が基本です。

**2. レスポンスヘッダー**。
リクエストのヘッダーと同じ形式で、追加情報を書きます。

```
Content-Type: text/html; charset=utf-8
Content-Length: 5120
Cache-Control: max-age=3600
Set-Cookie: sessionid=abc123; Path=/
```

よく出てくるヘッダーは次のとおりです。

- **`Content-Type`**: 返答がどの形式か(HTML、JSON、画像など)
- **`Content-Length`**: ボディのバイト数
- **`Cache-Control`**: ブラウザやキャッシュサーバーがこのコンテンツをどのくらい保存してよいか
- **`Set-Cookie`**: ブラウザに保存してほしいクッキー。ログイン情報などに使われる
- **`Location`**: リダイレクト先のURL(3xx系のステータスで使う)

**3. ボディ**。
実際のデータです。
HTML、JSON、画像データなどが入ります。

```
Content-Type: application/json

{"status": "success", "data": [{"id": 1, "name": "Item 1"}]}
```

## よく出るヘッダーの役割

ブラウザとサーバーが繰り返しやり取りする中で、同じヘッダーが何度も使われます。
それぞれの役割を理解しておくと、DevToolsを見たときに「ここはこういう指示を送っているのか」と理解できます。

**`Authorization: Bearer token`**

ログインが必要なリソースへアクセスするとき、認証情報を送ります。
「このリクエストは、このユーザーからの正当なものです」という証明にあたります。
トークンが無効か期限切れの場合、サーバーは `401` を返します。

**`Set-Cookie` と `Cookie`**

サーバーが `Set-Cookie` でクッキーを設定すると、ブラウザはそれを保存します。
次回以降のリクエストでは、ブラウザが自動で `Cookie` ヘッダーにそれを乗せて送ります。
ログイン状態が保持される理由はこの仕組みです。
HTTP自体は次に説明するとおり状態を覚えていませんが、クッキーで「このユーザー」という情報を外側から付け足しています。

**`Cache-Control`**

レスポンスヘッダーに `Cache-Control: max-age=3600` とあれば、ブラウザはそのコンテンツを3600秒(1時間)の間キャッシュできます。
同じURLへのリクエストが来たとき、サーバーへ聞き直さずキャッシュから返すことで、サーバーの負荷を減らせます。
`no-cache` なら毎回サーバーへ検証を問い合わせ、`no-store` なら保存を一切許しません。

**`Content-Type`**

リクエストでは「このフォーマットのデータを送っている」、レスポンスでは「返答はこのフォーマットだ」という宣言です。
`text/html`、`application/json`、`image/png` などの値を取ります。
これが欠けているか間違っていると、ブラウザは正しく解析できません。

**`Content-Encoding`**

データが圧縮されていることを示します。
`gzip` や `brotli` で圧縮されたボディを送ると、ブラウザが自動で解凍します。
通信量を減らす目的でよく使われます。

## HTTPはステートレス、ただし状態は外付けできる

ここはよくある誤解があるので、正確に説明しておきます。

「HTTPはステートレス」とは、HTTPプロトコル自体が、通信と通信の間の履歴や関係を覚えていないという意味です。
1回目のリクエストと2回目のリクエストが関連しているかどうかを、プロトコルは区別しません。
毎回が独立した通信として扱われます。

これは「状態管理ができない」という意味ではありません。
**クッキー、セッションID、ヘッダートークン**などを使えば、ユーザーの「ログイン状態」や「カートの内容」といった情報を外側から付け足せます。
サーバーが「このセッションIDはこのユーザーのものだ」と覚えておけば、HTTP自体が何も覚えていなくても問題は起きません。

実務では、ほぼすべてのウェブサービスが何らかの方法で状態を管理しています。
HTTPがステートレスだからこそ、状態管理の責任はサーバー側にあり、その設計が難しく、また重要にもなっています。

## HTTP/1.1とHTTP/2、何が変わったか

HTTP/1.1は1997年から使われていた標準で、1回のリクエストに1つの接続が必要でした。
ページに画像が100個あれば、接続が100回発生します。
そのたびに通信を立ち上げて閉じるオーバーヘッドがかかっていました。

HTTP/2(2015年頃から普及)では、**多重化**(multiplexing)という技術によって、1つの接続で複数のリクエストを同時に送受信できるようになりました。
画像100個分のリクエストが、実質1つの接続でまとめて処理されます。
結果として、ページの表示が速くなりました。

DevToolsのProtocol列に「h2」と表示されていればHTTP/2、「http/1.1」と表示されていればHTTP/1.1です。
見た目の違いは小さくても、内部の処理は大きく異なります。

HTTP/1.1の仕組みを理解しておけば、HTTP/2で何が改善されたのかも理解できます。
ここでは「複数のリクエストが1つの接続で同時に流れるようになった」という理解で十分です。

## ここで押さえること

- HTTPはテキストベースのプロトコル。ブラウザとサーバーがテキスト形式のリクエストとレスポンスで対話する
- **リクエスト**は「メソッド(GET/POSTなど)+パス+ヘッダー+ボディ」で構成される。サーバーへの指示にあたる
- **レスポンス**は「ステータスコード+ヘッダー+ボディ」で構成される。サーバーからの返答にあたる
- **ステータスコード**(200/404/500など)で、リクエストが成功したか失敗したかが一目でわかる
- HTTP自体はステートレスだが、**クッキーやトークン**で状態を外付けする。HTTPが何も覚えていなくても、サーバーが管理すれば状態は保持できる
- DevToolsのNetworkタブを見れば、今この瞬間に起こっているHTTP通信がそのまま見える
