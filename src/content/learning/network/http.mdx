## ブラウザのDevToolsを開くと見える、あの通信の正体

Chrome や Edge のブラウザを開いて F12 を押す。「Network」タブをクリックする。そこに表示される一覧が、あなたが見ているページの裏で実際に起こっている**通信そのもの**です。

1つの通信をクリックすると、「Headers」「Request」「Response」といったタブが出てきます。何か難しい数字や単語が書いてあるように見えるけど、実はそれこそが **HTTP**（Hypertext Transfer Protocol）という、ウェブでの通信ルールをそのまま表示したものなんです。

難しそうに見える理由は、実装に隠されているだけ。仕組みが分かると、あのDevToolsの表示は「ああ、こういう情報を送り合ってるのか」とすぐに読める。

## HTTPとは——テキストベースの通信プロトコル

HTTPの本質は、思ったより単純です。

あなたのブラウザが `https://example.com/page` を開いたとき、サーバーへ何を送っているのか。テキストで書くと、こんな感じです。

```
GET /page HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
```

ここまで。つまり **HTTPリクエスト**は、テキストで書いた指示です。「このサーバーへ」「この情報をもらいたい」「こういう条件で」を文章として送ります。

サーバーが返すのも、同じくテキスト。

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1234

<!DOCTYPE html>
<html>...
```

ステータス番号（200）から、情報の種類（Content-Type）、最後に実際のページ内容（body）まで全部テキストです。

「でも、ブラウザはそんなテキストを自動で作ったり読んだりしてくれるから、ユーザーは気にしなくていい」——その通り。でもエンジニアとして通信の中身を理解するなら、このテキスト形式を読めることが大前提になります。

## リクエストの構造

HTTPリクエストは3つの部分に分かれています。

**1. リクエストライン**：やりたいことを1行で書きます。

```
GET /page HTTP/1.1
```

ここで「`GET`」がメソッドです。`GET` は「サーバーから情報をもらう」という意味。同じくよく使うのは：

- **`GET`**：データをもらう（ブラウザがページを見るときはほぼこれ）
- **`POST`**：データをサーバーへ送る（フォーム送信、ログインなど）
- **`PUT`**：既存のデータを全部上書きする
- **`DELETE`**：データを削除する
- **`PATCH`**：データの一部だけ変更する

`/page` は「サーバーのどこにアクセスするか」で、**パス**と呼びます。`HTTP/1.1` はプロトコルのバージョンです。

**2. ヘッダー**：リクエストに関する追加情報を書きます。1行1つ、`名前: 値` の形式。

```
Host: example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64)
Accept: text/html
Content-Type: application/json
Authorization: Bearer token_here
```

よく出るものは：

- **`Host`**：どのサーバーか（複数のサイトが同じ IP を共有していることもあるので、これで指定）
- **`User-Agent`**：どのブラウザ・ツールから来たか
- **`Accept`**：どの形式の返答が欲しいか（HTML なのか JSON なのか）
- **`Content-Type`**：リクエストのボディが何形式か
- **`Authorization`**：認証情報。ログインが必要なら、ここにパスワードやトークンを入れる

ヘッダーは何個でも増やせます。キャッシュの指示、カスタムデータ、認証など、通信に必要な指示を全部ここに書きます。

**3. ボディ**：データを送る場合のみ。GET では通常ボディは空。POST でフォーム送信するときや、JSON データを送るときはここにデータが入ります。

```
POST /api/users HTTP/1.1
Host: example.com
Content-Type: application/json

{"name": "Taro", "email": "taro@example.com"}
```

この場合、最後の JSON がボディです。



## レスポンスの構造

サーバーからの返答も、同じくテキスト形式です。

**1. ステータスライン**：通信が成功したか失敗したかを示す数字。

```
HTTP/1.1 200 OK
```

`200` が「うまくいった」という意味の**ステータスコード**です。よく出るのは：

| コード | 意味 | 説明 |
|--------|------|------|
| **200 OK** | 成功 | リクエストが正常に処理された |
| **301 Moved Permanently** | 永続的な移動 | サイトが別の URL に引っ越した。ブラウザは自動で新しい URL にリダイレクト |
| **302 Found** | 一時的なリダイレクト | 一時的に別の URL へ飛ばす |
| **400 Bad Request** | リクエスト不正 | クライアントがおかしなリクエストを送った |
| **401 Unauthorized** | 認証が必要 | ログインしてないか、トークンが無効 |
| **403 Forbidden** | アクセス禁止 | ログインしていても、そのリソースにアクセスする権限がない |
| **404 Not Found** | 見つからない | その URL は存在しない |
| **500 Internal Server Error** | サーバーエラー | サーバー側で予期しないエラーが起きた |
| **503 Service Unavailable** | サービス利用不可 | サーバーがメンテナンス中など |

`200` 系は成功、`300` 系はリダイレクト、`400` 系はクライアントのミス、`500` 系はサーバーのミスという分け方が基本です。

**2. レスポンスヘッダー**：リクエストのヘッダーと同じく、追加情報を書きます。

```
Content-Type: text/html; charset=utf-8
Content-Length: 5120
Cache-Control: max-age=3600
Set-Cookie: sessionid=abc123; Path=/
```

よく出るものは：

- **`Content-Type`**：返答がどの形式か（HTML、JSON、画像など）
- **`Content-Length`**：ボディのバイト数
- **`Cache-Control`**：ブラウザやキャッシュサーバーがこのコンテンツをどのくらい保存していいか
- **`Set-Cookie`**：ブラウザに保存してほしいクッキー。ログイン情報など
- **`Location`**：リダイレクトのときの遷移先 URL（3xx 系のステータスで使用）

**3. ボディ**：実際のデータ。HTML、JSON、画像データなど。

```
Content-Type: application/json

{"status": "success", "data": [{"id": 1, "name": "Item 1"}]}
```



## よく出るヘッダーの役割

ブラウザとサーバーが何度も何度もやり取りする中で、同じヘッダーが繰り返し使われます。それぞれの役割を押さえておくと、DevTools を見たときに「あ、ここはこういう指示を送ってるんだ」と理解できるようになります。

**`Authorization: Bearer token`**

ログインが必要なリソースにアクセスするときに、認証情報を送ります。「このリクエストは、このユーザーからの正当なものです」という証明書のようなもの。トークンが無効か期限切れなら `401` で返されます。

**`Set-Cookie` と `Cookie`**

サーバーが `Set-Cookie` でクッキーをセットすると、ブラウザはそれを保存します。次のリクエストでは、ブラウザが自動で `Cookie` ヘッダーにそれを入れて送ります。ログイン状態を保持するのはこれで——HTTP 自体はステートレスですが、クッキーで「このユーザー」という情報を外付けしているわけです。

**`Cache-Control`**

レスポンスヘッダーに `Cache-Control: max-age=3600` とあれば、ブラウザはそのコンテンツを 3600 秒（1 時間）の間、キャッシュできます。同じ URL へのリクエストがきたら、サーバーへ聞き直さず、キャッシュから返す。これでサーバーの負荷を減らします。`no-cache` なら毎回サーバーへ検証を問い合わせ、`no-store` なら保存を一切許さない。

**`Content-Type`**

リクエストでは「私はこのフォーマットのデータを送ってます」、レスポンスでは「返答はこのフォーマットです」という宣言です。`text/html`、`application/json`、`image/png` など。これが無いか間違っていると、ブラウザは正しく解析できません。

**`Content-Encoding`**

データが圧縮されていることを示します。`gzip` や `brotli` で圧縮されたボディを送ると、ブラウザが自動で解凍します。通信量を減らすためによく使われます。

## HTTPはステートレス——ただし、状態は外付けできる

ここ、よくある誤解なので、正確にしておきます。

「HTTP はステートレス」とは、「HTTP プロトコル自体が、通信と通信の間の履歴・関係を覚えていない」という意味です。1 回目のリクエストと 2 回目のリクエストが関連しているかいないか、プロトコルは区別しない。毎回独立した通信として扱われます。

でも「状態管理ができない」という意味じゃありません。**クッキー、セッション ID、ヘッダートークン**などで、ユーザーの「ログイン状態」「カート内容」といった情報を外付けできます。サーバーが「このセッション ID はこのユーザー」と覚えていれば、HTTP が何も覚えてなくても問題ない。

実務では、ほぼすべてのウェブサービスが何らかの方法で状態を持っています。HTTP がステートレスだからこそ、状態管理の責任がサーバーにあり、その設計が難しくもあり、重要でもあるということです。

## HTTP/1.1 と HTTP/2：何が変わったか

HTTP/1.1 は 1997 年から使われていた標準で、1 回のリクエストに 1 つの接続が必要でした。ページに画像が 100 個あれば、100 回の接続が発生。そのたびに通信を立ち上げたり閉じたりするオーバーヘッドがありました。

HTTP/2（2015 年頃から普及開始）では、**多重化**（multiplexing）という技術を使って、1 つの接続で複数のリクエストを同時に送受信できるようになりました。画像 100 個のリクエストが、実質 1 つの接続で処理される。結果、ずっと高速になった。

DevTools を見たとき、Protocol 列に「h2」と書いてあれば HTTP/2、「http/1.1」と書いてあれば HTTP/1.1 です。大きな違いはありませんが、内部的には大きく異なります。

ただ、プロトコルレベルの理解としては「HTTP/1.1 で動く世界を理解すれば、HTTP/2 の改善も理解できる」という関係なので、ここでは「複数のリクエストが1接続で同時に流れるようになった」という認識で十分。

## ここで押さえること

- HTTP はテキストベースのプロトコル。ブラウザとサーバーがテキスト形式のリクエスト・レスポンスで対話する
- **リクエスト**は「メソッド（GET/POST など）+ パス + ヘッダー + ボディ」の構成。サーバーへの指示
- **レスポンス**は「ステータスコード + ヘッダー + ボディ」の構成。サーバーからの返答
- **ステータスコード**（200/404/500 など）で、リクエストが成功したか失敗したかが一目瞭然
- HTTP 自体はステートレスだが、**クッキーやトークン**で状態を外付けする。HTTP が何も覚えてなくても、サーバーが管理すれば状態は保持できる
- DevTools の Network タブを見れば、今この瞬間に起こっている HTTP 通信が見える。仕組みが分かると、あの表示は「ああ、こういう指示・返答をしてるんだ」と読める
