## ps auxを打ったら、自分が打ったコマンド以外が大量に並んでいた

Linuxサーバーに SSH でログインして、`ps aux` というコマンドを打つと、こういう出力が返ってきます。

```
USER       PID  %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1  43344  6660 ?        Ss   10:00   0:01 /sbin/init
root         2  0.0  0.0      0     0 ?        S    10:00   0:00 [kthreadd]
alice     1234  0.1  0.5  28391  5128 pts/0    Ss   10:15   0:02 bash
alice     1456  0.0  0.2  13456  2344 pts/0    S+   10:20   0:00 grep something
```

自分はsshでログインしてbashを起動しただけなのに、それ以外にもたくさんの行が並んでいます。

これらが何で、誰が動かしているのかがわからないと、サーバーで何が起きているかも判断できません。

この記事では、Linuxにおける**プロセス**とは何か、そしてそれがどう管理されているのかを、順番に見ていきます。

## プロセスとは実行中のプログラムである

**プロセス**とは、実行中のプログラムと、それに紐づく情報（使用しているメモリ、開いているファイル、実行している権限など）をまとめたものです。

同じプログラムを2回起動すると、2つの独立したプロセスが生まれます。

```
firefoxを起動 → プロセスA（独自のメモリ領域、独自のファイル管理）
firefoxを起動 → プロセスB（独自のメモリ領域、独自のファイル管理）
```

この2つのFirefoxは、見た目は同じプログラムですが、それぞれ独立したメモリとファイル管理を持って動いています。

## PIDとPPID、プロセスの親子関係

各プロセスには、一意な番号である**PID**（Process ID）が割り当てられます。

```
PID  ...  COMMAND
1    ...  /sbin/init        PID=1はsystemdまたはinit
45   ...  sshd              PID=45
1234 ...  bash              PID=1234
```

PID=1は特別な存在です。

Linuxが起動すると、最初にPID=1のプロセス（systemdまたはinit）が立ち上がり、それ以降のすべてのプロセスは、このプロセスを起点として生まれます。

つまりPID=1は、プロセス全体のツリー構造の根にあたります。

**PPID**（Parent PID）は、あるプロセスを起動した側のプロセスのPIDです。

`pstree` コマンドを使うと、この親子関係をツリーの形で確認できます。

```
init (PID 1)
 ├── sshd (PID 45, PPID=1)
 │    └── bash (PID 1234, PPID=45)   bashはsshdの子プロセス
 │         └── grep (PID 1456, PPID=1234)   grepはbashの子プロセス
 └── ...
```

親子関係を知っておく理由は、プロセスを終了させるときの影響範囲にあります。

bash（親プロセス）を終了させると、そのbashから起動された子プロセスも合わせて終了します。

サーバーで動いているプロセスのほとんどは、最終的にPID=1から枝分かれした子プロセスです。

`ps aux` で見える見知らぬプロセスの多くは、systemdが起動した何らかのサービスか、その子プロセスです。

## forkとexec、プロセスが生まれる手順

新しいプロセスを起動するとき、OSは内部的に2つの操作を順に行います。

最初の操作が**fork**（フォーク）です。

forkは、実行中のプロセスが自分自身のコピーを作る操作です。

```
親プロセス (PID=100)
  ↓ fork実行
親プロセス (PID=100) と 子プロセス (PID=101) が並行して存在
```

forkによって生まれた子プロセスは、親プロセスの**メモリイメージのコピー**を持った状態で動き出します。

同じコード、同じ変数の値、同じファイルの開閉状態を、子プロセスも引き継いでいます。

fork実行直後、親プロセスと子プロセスは区別できるようになっています。

fork関数を呼び出したコードの中で、自分が親なのか子なのかを判定できます。

```c
pid_t child_pid = fork();
if (child_pid == 0) {
    // 子プロセス側。child_pidには0が入る
    printf("I am child\n");
} else if (child_pid > 0) {
    // 親プロセス側。child_pidには子のPIDが入る
    printf("I am parent, child PID is %d\n", child_pid);
}
```

2番目の操作が**exec**（エグゼック）です。

execは、forkで生まれた子プロセスのメモリイメージを、別のプログラムのものに置き換える操作です。

```
子プロセス (forkで生まれたばかり、親のコピー)
  ↓ exec実行
子プロセス (新しいプログラム、例えばbashで実行を開始)
```

exec実行後、メモリイメージは新しいプログラムのものに置き換わります。

それまでのコードは消え、新しいプログラムが実行されますが、PIDは変わりません。

bashが新しいコマンドを実行するとき、内部ではforkとexecの組み合わせが起きています。

```
bash (PID=1000) でlsコマンドを実行する場合

bash (PID=1000)
  ↓ fork
bash (1000) + bash (PID=1001、子プロセス)
  ↓ exec（子プロセスがlsに置き換わる）
bash (1000) + ls (PID=1001)
  ↓ lsが終了
bash (1000) だけが残る
```

<Callout title="forkだけでは別のプログラムを実行できないのはなぜか">
forkは親プロセスのコピーを作るだけなので、forkの直後はまだ親プロセスと同じコードが動いています。

別のプログラムを動かすには、execでプロセスのメモリイメージを書き換える必要があります。

forkとexecを分けることで、新しいプロセスを作る処理と、そのプロセスでどのプログラムを動かすかという処理を、別々に制御できるようになっています。
</Callout>

## フォアグラウンドとバックグラウンド

ターミナルでコマンドを実行すると、デフォルトでは**フォアグラウンド**で動きます。

```bash
sleep 100
```

このコマンドを実行すると、100秒間、ターミナルはコマンドの終了を待ち続け、次の操作ができません。

終了を待たずに次の操作をしたい場合は、コマンドの末尾に `&` を付けて**バックグラウンド**で起動します。

```bash
sleep 100 &
```

```
[1] 1234
```

実行するとすぐにプロンプトが戻ってきます。

`[1]` はこのシェルにおけるジョブ番号、`1234` はそのプロセスのPIDです。

実行中のプロセスは、一時停止と再開ができます。

```bash
sleep 100
```

```
^Z
[1]+ Stopped    sleep 100
```

`Ctrl+Z` を押すと、フォアグラウンドで動いていたプロセスが一時停止します。

```bash
jobs
```

```
[1]+ Stopped    sleep 100
```

`jobs` は現在のシェルが持つジョブの一覧を表示します。

```bash
bg
```

```
[1]+ sleep 100 &
```

`bg` は一時停止しているジョブをバックグラウンドで再開します。

```bash
fg
```

```
sleep 100
```

`fg` はジョブをフォアグラウンドに戻します。

フォアグラウンドに戻った状態で `Ctrl+C` を押すと、プロセスを停止できます。

複数のバックグラウンドジョブがある場合は、`fg %1`、`fg %2` のようにジョブ番号を指定して操作します。

## シグナルによるプロセスの終了

プロセスを終了させる操作には、複数の方法があります。

**SIGTERM**は、プロセスに終了を要求する信号です。

```bash
kill 1234
```

`kill` コマンドは、引数を指定しない場合、デフォルトでSIGTERMを送信します。

SIGTERMを受け取ったプロセスは、ファイルのクローズやリソースの解放といった後処理をしてから終了できます。

**SIGKILL**は、プロセスを強制的に終了させる信号です。

```bash
kill -9 1234
```

`-9` はSIGKILLの番号で、`kill -SIGKILL 1234` と書くこともできます。

SIGKILLを受け取ったプロセスは、後処理をする機会を与えられずに即座に終了します。

このため、ファイルが中途半端な状態で残ったり、ロックが解放されないままになったりすることがあります。

## /procでプロセスの情報をファイルとして読む

ここまでの内容より少し進んだ話として、`/proc` という仮想ファイルシステムを紹介します。

`/proc` は実際のディスク領域を持たない仮想ファイルシステムで、OSが実行中のプロセスの情報をファイルの形で見せています。

```bash
ls /proc
```

```
1/
2/
45/
1234/
```

各ディレクトリの名前は、プロセスのPIDに対応しています。

ディレクトリの中に入ると、そのプロセスに関する情報がファイルとして並んでいます。

```bash
ls /proc/1234
```

```
cmdline     起動時に渡されたコマンドライン引数
environ     環境変数
fd/         開いているファイルディスクリプタ
maps        メモリマップ
status      プロセスの状態（メモリ使用量、スレッド数など）
```

`status` ファイルを読むと、そのプロセスのメモリ使用量やスレッド数がわかります。

```bash
cat /proc/1234/status
```

```
Name:   bash
State:  S (sleeping)
Pid:    1234
PPid:   45
VmPeak:     28391 kB
VmRSS:       5128 kB
Threads:    1
```

この方法は、デバッグやセキュリティ調査において、特定のプロセスが具体的に何をしているかを調べるときに使います。

<Callout title="psコマンドの内部">
`ps` コマンドは、内部的に `/proc` を読み取り、その内容を見やすく整形して表示しています。

`/proc` に直接アクセスすることもできますが、通常は `ps` を使うほうが情報を読みやすくなります。
</Callout>

## ここまでのまとめ

**プロセス**は実行中のプログラムと、それに紐づくメモリやファイルの情報をまとめたものです。

**PID**は各プロセスの一意な識別番号、**PPID**は親プロセスのPIDで、`pstree` でこの親子関係をツリーとして確認できます。

新しいプロセスは**fork**で親プロセスのコピーが作られ、**exec**で別のプログラムに置き換わることで生まれます。

コマンドの末尾に `&` を付けるとバックグラウンドで実行され、`Ctrl+Z`、`bg`、`fg` でフォアグラウンドとバックグラウンドを切り替えられます。

プロセスを終了させるときは、後処理の機会を与える**SIGTERM**を先に送り、応答しない場合に限って強制終了の**SIGKILL**を使います。

`/proc` は実行中のプロセスの情報をファイルとして見せる仮想ファイルシステムで、`ps` コマンドはこの情報を内部で読み取って整形しています。
