## ps aux を打ったら見知らぬプロセスが大量に並んでいた

Linux サーバーに SSH でログインして、`ps aux` を叩くと、こんな出力が。

```
USER       PID  %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.1  43344  6660 ?        Ss   10:00   0:01 /sbin/init
root         2  0.0  0.0      0     0 ?        S    10:00   0:00 [kthreadd]
alice     1234  0.1  0.5  28391  5128 pts/0    Ss   10:15   0:02 bash
alice     1456  0.0  0.2  13456  2344 pts/0    S+   10:20   0:00 grep something
...
```

「あれ、俺の打ったコマンドだけじゃなく、いっぱい何かが走ってる——何なんだこれ？」と思った経験、あると思います。ここからプロセスとは何か、どう管理されているのかを見ていきます。

## プロセスとは「プログラムの実行インスタンス」

単純な定義：

**プロセス** = 実行中のプログラムと、それに付紐付く情報（メモリ、ファイルディスクリプタ、権限など）の総体

同じプログラムを2回起動したら、2つのプロセスが生まれます。独立した実行環境を持つ。

```
firefox を起動 → プロセスA（メモリ領域A、ファイル番号管理A）
firefox を起動 → プロセスB（メモリ領域B、ファイル番号管理B）

→ 2つの Firefox が独立して動く
```

## PID と PPID——「プロセスの ID」と「親プロセスのID」

各プロセスには一意な番号（**PID**）が割り当てられます。

```
ps aux の出力：
PID  ...  COMMAND
1    ...  /sbin/init        # PID=1 は systemd（ルートプロセス）
45   ...  sshd              # PID=45
1234 ...  bash              # PID=1234
```

PID=1 は特別。Linuxが起動すると最初に 1 番のプロセス（systemd または init）が立ち上がり、そこから全ての他のプロセスが生まれます。プロセスツリーの「根」です。

**PPID**（Parent PID）は「親プロセスのPID」。あるプロセスを起動したプロセスのID。

```
pstree コマンドでツリーを見る：

init (PID 1)
 ├── sshd (PID 45, PPID=1)
 │    └── bash (PID 1234, PPID=45)  ← bash は sshd の子
 │         └── grep (PID 1456, PPID=1234) ← grep は bash の子
 └── ...
```

「なぜそんなこと知る必要があるのか」——プロセスを終了したいとき、親プロセスの状態が影響することがあります。例えば bash（親）を kill すると、bash から起動した全ての子プロセスも自動的に終了します。



## fork と exec——プロセスが生まれるメカニズム

プロセスを新たに起動するとき、OS は内部的に2つのステップを踏みます。

**1. fork（フォーク）**：実行中のプロセスが「自分のコピー」を作る。

```
親プロセス (PID=100)
  ↓ fork 実行
親プロセス (PID=100) と 子プロセス (PID=101) が並行
```

子プロセスは親の **メモリイメージのコピー** を持ってスタート。当然、同じコード、同じ変数、同じファイルディスクリプタを持っています。

ここで重要：fork 直後、親と子は区別される。fork から帰ってきたコード内で「俺は親か子か」を判定できます。

```c
// C言語での例（実装は省略）
pid_t child_pid = fork();
if (child_pid == 0) {
    // 子プロセス（PID=0）
    printf("I am child\n");
} else if (child_pid > 0) {
    // 親プロセス（child_pid には子のPIDが入る）
    printf("I am parent, child PID is %d\n", child_pid);
}
```

**2. exec（エグゼック）**：子プロセスが「別のプログラム」に置き換わる。

```
子プロセス (fork で生まれたばかり、親のコピー)
  ↓ exec実行
子プロセス (新しいプログラム（例：bash）で実行開始)
```

exec 実行後、メモリイメージは新プログラムのものに置き換わります。以前のコードは消え、新しいプログラムが実行される。PID は変わりません。

**つまり**：新しいプロセスを起動する ≈ fork → exec の組み合わせ。

```
bash でコマンド実行：
bash (PID=1000)
  ↓ fork
bash (1000) + bash (PID=1001, 子)
  ↓ exec (子が ls に置き換わる)
bash (1000) + ls (PID=1001)
  ↓ ls 終了
bash (1000) だけ残る
```

<Callout title="fork は子プロセスのコピーを作るのに、なぜ exec が必要か">
  fork だけだと、親プロセスとまったく同じコードを走らせるだけで、別のプログラムを実行できません。exec で「このプロセスのメモリイメージを新しいプログラムで上書きする」ことで、別のプログラムを同じPIDで実行できるようになる。これは非常に効率的で、UNIX/Linux の設計の賢さ。
</Callout>

## フォアグラウンドとバックグラウンド、Ctrl+Z / bg / fg

ターミナルでコマンドを打つと、デフォルトはフォアグラウンドで動きます。

```
$ sleep 100
# 100秒間、ターミナルは動作待機状態
```

待ちたくないなら、コマンド末尾に `&` を付けてバックグラウンドで起動：

```
$ sleep 100 &
[1] 1234
# すぐにプロンプトが戻ってくる
# [1] はジョブ番号、1234 はPID
```

実行中のプロセスを一時停止・再開：

```
$ sleep 100
^Z              # Ctrl+Z を押す
[1]+ Stopped    sleep 100

$ jobs          # 現在のジョブ一覧
[1]+ Stopped    sleep 100

$ bg            # そのジョブをバックグラウンドで再開
[1]+ sleep 100 &

$ jobs
[1]+ Running    sleep 100 &

$ fg            # フォアグラウンドに戻す
sleep 100
# またターミナルが睨めっこ状態
^C              # Ctrl+C で停止
```

ジョブ管理はシェル（bash など）の機能。複数のバックグラウンドジョブがあれば `fg %1`, `fg %2` のように指定できます。

## シグナル——プロセスとの「対話」

プロセスを終了させたいとき、単に「kill」と言っても、実は複数の方法があります。

**SIGTERM**（シグターム）：「丁寧な終了要求」。プロセスは終了の準備（クリーンアップ）ができます。

```
kill 1234       # デフォルトで SIGTERM を送信
```

プロセス側で `try-finally` みたいにキャッチして、ファイルクローズやリソース解放をしてから終了できる。

**SIGKILL**（シグキル）：「強制終了」。プロセスは文句なく即座に落ちます。

```
kill -9 1234    # または kill -SIGKILL 1234
```

SIGKILL は無視できません。ただし、リソースのクリーンアップができないので、ファイルが壊れたり、ロックが残ったりすることも。

## /proc 仮想ファイルシステム——プロセス情報を「ファイル」として見る

ここ、少し高度ですが、知ると Linux への理解が深まります。

`/proc` は **実在しないファイル** です。OS が「プロセスの情報をあたかもファイルとして見えるようにしてくれる」仮想ファイルシステム。

```
ls /proc

1/
2/
45/
1234/
...
```

各数字はプロセスの PID に対応したディレクトリ。その中に入ると：

```
ls /proc/1234

cmdline     # コマンドラインの引数
environ     # 環境変数
fd/         # ファイルディスクリプタ（開いてるファイル）
maps        # メモリマップ
status      # プロセスの状態（メモリ使用量、スレッド数など）
...
```

例えば `cat /proc/1234/status` で、PID=1234 のプロセスのメモリ使用量やスレッド数が読める。

```
cat /proc/1234/status

Name:   bash
State:  S (sleeping)
Pid:    1234
PPid:   45
VmPeak:     28391 kB
VmRSS:       5128 kB
Threads:    1
```

セキュリティや デバッグのときに「特定のプロセスが何をしてるか」を調査するのに重宝します。

<Callout title="ps aux の内部実装">
  `ps` コマンドは内部的に `/proc` を読んで情報を整形・表示してくれている。だから `/proc` に直接アクセスすることもできるし、`ps` の方が見やすい。
</Callout>

## ここで押さえること

- **プロセス** = 実行中のプログラム + 付紐付く情報（メモリ、ファイルディスクリプタなど）
- **PID** は各プロセスの一意ID。**PPID** は親プロセスのID。`pstree` でツリー構造が見える
- **fork + exec** — プロセスが生まれるメカニズム。fork で子プロセスのコピーを作って、exec で新プログラムに置き換え
- **フォアグラウンド / バックグラウンド** — `&` でバックグラウンド起動。`Ctrl+Z`, `bg`, `fg` で管理
- **SIGTERM（丁寧）vs SIGKILL（強制）** — まずはSIGTERM で kill を試し、応答しなければ SIGKILL で
- **/proc** は仮想ファイルシステム。プロセスの情報をファイル形式で見える化
