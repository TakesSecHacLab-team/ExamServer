## USBを「安全に取り外す」ボタンはなぜ必要か

WindowsやMacでUSBメモリを挿すと、「安全に取り外す」とか「イジェクト」みたいなボタンが出てくる。「別に引っこ抜いても大丈夫じゃん」と思ったことある人、多いと思います。

でもそっちが実は危険。ここから、Linuxのファイルシステムという仕組みを通じて、なぜそれが必要なのかを解きほぐします。

## ファイルシステムとは——「ディスクの整理箱」

ファイルシステムとは、ハードディスクやSSD上に「データをどうやって保管・管理するか」というルールセット（仕組み）です。

なんで必要か。考えてみてください。ディスクは「磁気が付いてる領域 / 付いてない領域」の区別しかできません。1と0の列です。でも僕たちは「ファイル」という塊単位で考えたいし、「このファイルはどこに在るのか」「どのくらいのサイズなのか」「誰が読み書きできるのか」といった情報も必要。

それを管理するのがファイルシステム。「ここからここまでが『test.txt』だよ」「所有者はuser、読み取り権限はgroupまで」みたいなことを記録して、OSがそれを基に読み書きを仲介するわけです。

## Linux のディレクトリ構造——「根っこから枝へ」

Linuxでは全ファイルが1つのツリーの下にぶら下がっています。頂点は `/`（ルートディレクトリ）。

```
/
├── bin       → よく使うコマンド群（ls, cat, grep など）
├── etc       → システム設定ファイル（ユーザー情報、ネットワーク設定など）
├── home      → ユーザーのホームディレクトリが入る（/home/alice など）
├── var       → 時系列データ。ログ、キャッシュ、メールスプール、DBデータ
├── tmp       → 一時ファイル。再起動で消える
├── proc      → **仮想ファイル**。実際には存在しない。実行中のプロセス情報をファイル化して見せる
├── sys       → 同じく仮想。カーネルと各種ドライバの状態を見える化
├── lib       → システムが使う共有ライブラリ（.so ファイル）
├── usr       → ユーザーが入れたプログラムとそのライブラリが住む領域
├── opt       → オプション。大型のサードパーティアプリ
└── root      → root ユーザーのホームディレクトリ
```

「なぜこんなに細かく分けてるの？」——これは UNIX の古い哲学。「各ディレクトリには役割を持たせる。ログはここ、設定はここ、一時ファイルはここ」と責任を分離することで、保守しやすく、バックアップ時に何をコピーしたらいいか明確になるわけです。

Windowsとの大きな違い：Linuxはドライブレター（C:\, D:\ みたいなの）を使わない。すべて `/` の下に **マウント** されます。

## マウント——「USBをツリーに繋ぐ」という行為

ここが「安全に取り外す」の正体につながる部分です。

Linuxでは、USBメモリであろうが追加のハードディスクであろうが、何らかのストレージデバイスをシステムに **組み込む** 行為を **マウント** と呼びます。

例えば、新しい USB を挿すと、OSはそれを自動認識して `/media/user/USB_NAME` みたいなところに繋ぎます。これが「マウント」。マウント後は、ツリーの一部として `/` 配下のファイルパスでアクセス可能になります。

逆に外したいときは **アンマウント** する。これが「安全に取り外す」の正体。

なぜ必須か。ファイルシステムは「後で書き込む予定だったデータ」を一度メモリ（キャッシュ）に溜めておいて、効率良くディスクに書き出すという仕組みを持っています。USB を突然引っこ抜くと、メモリにまだ残っているデータがディスクに書き込まれないまま消える。結果、ファイルが壊れたり、ファイルシステム自体の構造が壊れたり、データロストしたりするわけです。

アンマウント時に「キャッシュをすべてディスクに書き込んでから切り離す」という流れを踏むから安全。それを OS に任せずに無理矢理引っこ抜くのは、銀行の最中にATMの電源を抜くみたいなもの。

<Callout title="マウント / アンマウントコマンド">
  Linux上でコマンドで操作する場合：
  ```
  mount /dev/sdb1 /mnt/external    # /dev/sdb1 を /mnt/external にマウント
  umount /mnt/external              # アンマウント（キャッシュをディスクに書き込んでから切り離す）
  ```
  ただしこれは root 権限が必要。普通のユーザーがGUIから「安全に取り外す」を押すと、内部で umount が実行されます。
</Callout>

## inode——「ファイル名」と「ファイルの実体」は別物

これ、かなり重要なポイントです。

Linuxのファイルシステムでは、ファイル名とファイルの実体が **分離** されています。

仕組みはこう：

- **inode** = ファイル（またはディレクトリ）の実体。中身、所有者、パーミッション、作成日時、実データがディスク上のどこに在るか、という情報をすべて持つ
- **ファイル名** = inode への「ポインタ」（参照・ショートカット）。複数のファイル名から同じ inode を指すことも可能

例えば、`test.txt` を作る。実は、OSは：

1. ディスク上の適切な場所にデータを書き込む
2. そのデータの「メタデータ」（誰のものか、何ビットか、どこに在るか）を inode として記録する
3. ディレクトリテーブルに「`test.txt` → inode 12345」という対応を記す

ファイルを削除するとき、OSは `test.txt` というエントリを削除します。でも inode 12345 の参照カウント（「何個のファイル名が俺を指してるか」の数）が0になるまで、inode は残ります。参照カウント > 0 なら、`test.txt` を削除しても、別の名前でそのファイルは存在し続ける。

これが何に効くか。**ログファイルの復旧** やデジタルフォレンジック（調査）で、ファイル名を削除しても実データが残っているという事実が基盤になります。「ごみ箱を空にしても復元できる」のは、inode と実データがまだディスク上に在るから。専門的なツール（`dd`, `strings`, `PhotoRec` など）でスキャンすれば、ファイル名が消えたあとも データ自体は読める。セキュリティの観点では「ファイルを削除した ≠ データが完全に消えた」という認識が必須です。

## 代表的なファイルシステムたち

Linuxで使われるファイルシステムは複数あります。軽く知っておくと、「なぜこのシステムでこれを選んだのか」が理解できます。

| FS | 特徴 | 向いてる場面 |
|---|---|---|
| **ext4** | 標準的。バランス型。ジャーナリング対応（クラッシュに強い） | ほとんどのLinuxサーバー、デスクトップ |
| **xfs** | 大容量ファイル向け。メタデータ処理が高速 | 大規模ストレージ、データセンター |
| **btrfs** | 新しい。スナップショット機能、圧縮、データ検証 | 高機能性が欲しい場面。まだ発展途上 |
| **tmpfs** | メモリベース。揮発性。/tmp や /dev/shm で使われる | 一時ファイル、再起動で消えてOK なデータ |

深掘りするなら各FS の白書を読むといいですが、まずは「ext4 が標準で、大規模環境は xfs」くらいで十分です。

## ここで押さえること

- **ファイルシステム** はディスク上のデータをどう管理するかというルール。OSが読み書きを仲介する
- **Linux のディレクトリツリー** は `/` 単一の根から全ファイルが枝分かれする。Windows の C:, D: みたいなドライブレターはない
- **マウント / アンマウント** — ストレージを「ツリーに繋ぐ / 外す」。アンマウント時にキャッシュをディスクに書き込むから「安全に取り外す」が必須
- **inode** は「ファイル名」と「ファイルの実体」を分離する仕組み。削除してもinodeと実データは残ることがあり、復旧やフォレンジックの前提
- **ext4, xfs, btrfs** など複数の FS がある。デフォルトはext4。大規模環境ではxfs が多い
