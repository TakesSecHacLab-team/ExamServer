## USBメモリを引っこ抜く前に「安全に取り外す」を押すのはなぜか

WindowsでもMacでも、USBメモリを挿すと「安全に取り外す」や「取り出す」というボタンが用意されています。

このボタンを押さずに直接引っこ抜いても、たいていは何も起こりません。

ファイルも壊れず、いつも通り使えます。

だからこそ「このボタン、別に押さなくてもよくない」と感じた人は多いと思います。

ところが、これは運がよかっただけです。

USBメモリへの書き込みは、OSが裏側で「あとでまとめて書き込む」という最適化をしています。

書き込み中のタイミングで引っこ抜くと、データの一部だけがディスクに反映され、ファイルやファイルシステムそのものが壊れることがあります。

この記事では、Linuxの**ファイルシステム**という仕組みを土台から説明しながら、なぜ「安全な取り外し」が必要なのかを順番に解きほぐします。

## ファイルシステムとは何か

ディスク（ハードディスクやSSD）は、もともと「ファイル」という単位を知りません。

ディスクが持っているのは、磁気やフラッシュメモリの状態として記録された、0と1の並びだけです。

人間がファイルという単位で読み書きできるようにするには、「どこからどこまでが1つのファイルか」「そのファイルの名前は何か」「誰が読み書きしてよいか」という情報を、ディスクのどこかに記録しておく必要があります。

**ファイルシステム**とは、ディスク上のデータを「ファイル」という単位で管理するためのルールの集まりです。

OSはファイルシステムのルールに従って、ファイルの読み書きをディスクとの間で仲介します。

たとえば「`test.txt` というファイルは、ディスクの何番目から何番目のブロックに書かれている。所有者はuser、グループの読み取り権限がある」といった情報を、ファイルシステムが記録して管理しています。

ユーザーが見ている「ファイル」と「フォルダ」は、ファイルシステムが提供する見せ方であり、ディスクの実体そのものではありません。

## Linuxのディレクトリ構造はなぜ1本の木になっているのか

Windowsでは、ファイルは `C:\` や `D:\` のように、ドライブごとに別々の木構造で管理されています。

Linuxではドライブという考え方を使いません。

すべてのファイルが、`/`（**ルートディレクトリ**と呼びます）を根とする1本の木構造の下にぶら下がります。

```
/
├── bin       よく使うコマンド群（ls, cat, grep など）
├── etc       システム設定ファイル（ユーザー情報、ネットワーク設定など）
├── home      各ユーザーのホームディレクトリ（/home/alice など）
├── var       時系列で増えるデータ。ログ、キャッシュ、メールスプール、DBデータ
├── tmp       一時ファイル。再起動で消える
├── proc      仮想ファイルシステム。実行中のプロセス情報をファイルの形で見せる
├── sys       仮想ファイルシステム。カーネルとドライバの状態を見せる
├── lib       システムが使う共有ライブラリ（.so ファイル）
├── usr       ユーザーが追加でインストールしたプログラムとそのライブラリ
├── opt       サードパーティ製の大型アプリケーション
└── root      root ユーザーのホームディレクトリ
```

ディレクトリがこれだけ細かく分かれているのは、UNIXという古いOS系列が持つ設計方針によります。

「設定ファイルはここ、ログはここ、一時ファイルはここ」と役割ごとに置き場所を決めておけば、バックアップを取るときに「`/etc` と `/home` だけコピーすればよい」のように、何を保護すべきかが明確になります。

`/proc` と `/sys` は実際のディスク領域を持たない**仮想ファイルシステム**です。

カーネルが管理している情報（動いているプロセスの一覧、メモリの使用状況など）を、あたかもファイルとして読めるように見せかけています。

## マウントとは何か、なぜUSBの取り外しに関係するのか

ここまでで、Linuxのファイルが1本の木構造の下に全部つながっていることがわかりました。

新しくUSBメモリやハードディスクを挿したとき、そのストレージの中身は、最初からこの木構造に含まれているわけではありません。

ストレージを木構造の中の特定のディレクトリに結びつけて、`/` から続くパスでアクセスできるようにする操作を**マウント**と呼びます。

USBメモリを挿すと、OSが自動的にこれを認識し、たとえば `/media/user/USB_NAME` のようなディレクトリにマウントします。

マウントが終わると、USBメモリの中のファイルは、ふつうのディレクトリパスとして読み書きできるようになります。

取り外すときは、木構造との結びつきを切る**アンマウント**という操作が必要です。

「安全に取り外す」ボタンの正体は、このアンマウントです。

なぜアンマウントが必要かは、ファイルシステムの書き込み方式に理由があります。

ファイルシステムは、書き込みのたびにディスクへ直接アクセスするのではなく、書き込み内容を一度メモリ上の**キャッシュ**に溜めておき、まとめてディスクに反映する仕組みを持っています。

これは、ディスクへのアクセス回数を減らして全体の速度を上げるための最適化です。

USBメモリを引っこ抜くタイミングで、キャッシュにまだディスクへ反映されていないデータが残っていると、そのデータは失われます。

ファイルの一部だけが書き込まれた状態になり、ファイルが壊れることもあれば、ファイルシステム全体の管理情報が壊れることもあります。

アンマウントは「キャッシュに残っている内容をすべてディスクに書き込んでから、木構造との結びつきを切る」という手順を踏みます。

この手順を踏まずに物理的に引っこ抜く行為は、銀行が取引データをまだ記録し終えていない瞬間に、サーバーの電源を落とすことに近いといえます。

<Callout title="マウントとアンマウントをコマンドで行う場合">
```
mount /dev/sdb1 /mnt/external
umount /mnt/external
```
`mount` は指定したデバイス（`/dev/sdb1`）を指定したディレクトリ（`/mnt/external`）に結びつけます。

`umount` はキャッシュの内容をディスクに書き込んでから結びつきを切ります。

どちらも通常はroot権限が必要です。

GUIで「安全に取り外す」を押した場合、内部的には `umount` に相当する処理が実行されています。
</Callout>

## ファイル名とファイルの実体は別物である

ファイルを削除したのに、データ復旧ソフトで内容が読めてしまうことがあります。

これは、Linuxのファイルシステムにおいて、**ファイル名とファイルの実体が別の場所に記録されている**ためです。

仕組みを整理します。

**inode**とは、1つのファイル（またはディレクトリ）の実体を表す管理情報です。

inodeには、ファイルの所有者、パーミッション、作成日時、そして実データがディスク上のどこにあるか、という情報がすべて含まれます。

inode自体には「ファイル名」という情報は含まれていません。

ファイル名は、inodeを指し示す**ポインタ**として、ディレクトリの中に別途記録されています。

1つのinodeを、複数の異なるファイル名から指すことも可能です。

`test.txt` というファイルを新しく作るとき、OSは次の手順を実行します。

1. ディスク上の空いている領域にデータを書き込む
2. そのデータの管理情報（所有者、サイズ、ディスク上の位置など）をinodeとして記録する
3. ディレクトリの中に「`test.txt` という名前はinode番号12345を指す」という対応を記録する

ファイルを削除する操作は、実際には「ディレクトリから `test.txt` という名前の対応を消す」操作です。

inode自体は、それを指しているファイル名の数（**参照カウント**と呼びます）が0になるまで残り続けます。

別の名前から同じinodeを参照しているファイルがもう1つあれば、`test.txt` を削除しても、そのもう1つの名前でファイルは存在し続けます。

ファイルを「削除した」ことと、データが「完全に消えた」ことは同じではありません。

ファイル名がディレクトリから消えても、参照カウントが0になっていなければinodeは残りますし、参照カウントが0になった直後でも、実データがディスク上の同じ場所に残っている場合があります。

`dd` や `strings`、`PhotoRec` といった専門的なツールでディスクを直接スキャンすると、ファイル名が消えたあとのデータを読み取れることがあります。

ログファイルの復旧やデジタルフォレンジック（証拠調査）は、この仕組みを前提に成り立っています。

セキュリティの実務では「ファイルを削除した」という操作が「データが読めなくなった」ことを保証しないという前提で、機密データの扱いを考える必要があります。

## 代表的なファイルシステムの種類

Linuxで使えるファイルシステムは1種類ではありません。

それぞれ得意な場面が異なります。

<table>
  <thead>
    <tr>
      <th>名称</th>
      <th>特徴</th>
      <th>向いている場面</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><strong>ext4</strong></td>
      <td>長く使われてきた標準的な方式。ジャーナリング機能を持ち、クラッシュ後の復旧に強い</td>
      <td>一般的なLinuxサーバーやデスクトップ</td>
    </tr>
    <tr>
      <td><strong>xfs</strong></td>
      <td>大容量ファイルの扱いに強く、メタデータの処理が速い</td>
      <td>大規模ストレージ、データセンター</td>
    </tr>
    <tr>
      <td><strong>btrfs</strong></td>
      <td>比較的新しい方式。スナップショット、圧縮、データ検証機能を持つ</td>
      <td>高度な機能が必要な場面</td>
    </tr>
    <tr>
      <td><strong>tmpfs</strong></td>
      <td>ディスクではなくメモリ上に作られる。再起動すると内容が消える</td>

    </tr>

**ジャーナリング**とは、ファイルシステムへの変更を一旦記録（ジャーナル）してから実際の変更を行う仕組みです。

書き込みの途中で電源が落ちても、ジャーナルを使って未完了の変更を巻き戻すか完了させるかを判断できるため、クラッシュ後にファイルシステムが壊れにくくなります。

迷ったときは「サーバーやデスクトップではext4が標準、大規模なストレージにはxfsが選ばれることが多い」という基準で十分です。

## ここまでのまとめ

**ファイルシステム**は、ディスク上のデータを「ファイル」という単位で管理するためのルールであり、OSはこのルールに従って読み書きを仲介します。

Linuxのディレクトリは、Windowsのドライブレターのような分割を持たず、`/` を根とする1本の木構造として表現されます。

ストレージを木構造に組み込む操作を**マウント**、結びつきを切る操作を**アンマウント**と呼び、アンマウント時にキャッシュの内容をディスクへ書き込むからこそ、USBメモリの安全な取り外しが必要になります。

**inode**はファイルの実体を表す管理情報で、ファイル名はinodeを指すポインタにすぎません。

この分離があるために、ファイルを削除しても実データが残ることがあり、復旧やフォレンジックの可能性につながっています。

ファイルシステムにはext4、xfs、btrfsなど複数の方式があり、用途に応じて選ばれます。
