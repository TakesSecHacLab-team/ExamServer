## マウスでクリックすれば済むのに、なぜ黒い画面にコマンドを打つのか

WindowsやMacのデスクトップでは、マウスでアイコンをクリックしたり、メニューからボタンを選んだりして操作するのが当たり前になっています。

それでもエンジニアの多くは、ターミナルと呼ばれる黒い画面を開いて、文字でコマンドを打ち込んで作業をしています。

理由は、同じ操作を何度も繰り返す必要が出てきたときに表れます。

マウスでの操作は、1回ごとに人間の手が必要です。

100個のファイルに同じ処理をしたいとき、マウスで100回クリックするのは現実的ではありません。

文字で書いたコマンドは、ファイルに保存して何度でも実行できます。

この記事では、この「黒い画面」の正体である**シェル**について、何ができるのかを順番に見ていきます。

## シェルとはOSとユーザーをつなぐプログラムである

**シェル**とは、ユーザーが入力したコマンドを解釈し、OSに指示を出し、その結果をユーザーに返すプログラムです。

```
ユーザー → シェル → OS → 実行 → 結果をシェルへ → ユーザーへ
```

マウスでボタンをクリックする操作方法を**GUI**（グラフィカルユーザーインターフェース）、文字でコマンドを入力する操作方法を**CLI**（コマンドラインインターフェース）と呼びます。

CLIがGUIより優れている点は、操作を文字として記録できることです。

同じ操作を繰り返したい場合、CLIではコマンドをファイル（**スクリプト**）に書いておき、それを実行するだけで済みます。

```bash
for file in *.jpg; do
    convert "$file" "${file%.jpg}.png"
done
```

```bash
bash script.sh
```

このスクリプトを実行すると、現在のディレクトリにある100個のJPGファイルが、すべて自動的にPNGに変換されます。

GUIでこの処理を行う場合、1個ずつ変換ソフトを開いて操作する必要がありますが、CLIではコマンドを一度書くだけで、何個のファイルにも同じ処理を適用できます。

ここで注意が必要な点があります。

スクリプトの先頭に `#!/bin/sh` と書いても、システムによって `/bin/sh` が指す実体は異なります。

多くのLinuxディストリビューションでは、`/bin/sh` は `/bin/dash` へのシンボリックリンクです。

`/bin/sh` がdashを指している環境で、bash特有の機能を使ったスクリプトを実行すると、エラーになります。

```bash
#!/bin/bash
array=(a b c)
echo ${array[0]}   # a
```

このスクリプトはbash向けに書かれており、配列という機能を使っています。

```bash
#!/bin/sh
array=(a b c)      # Syntax Error
```

同じ内容を `#!/bin/sh` で実行しようとすると、dash上では配列がサポートされていないため、構文エラーになります。

スクリプトの先頭を `#!/bin/sh` にする場合は、POSIX互換の機能だけを使い、bash特有の機能（配列や `[[ ]]` を使った条件式など）は避けます。

bashの機能を使いたい場合は、`#!/bin/bash` と明示します。

## stdin、stdout、stderrという3つの入出力

CLIでの操作を理解するには、データがどこから来てどこへ流れるかを把握する必要があります。

データの流れには3種類あります。

**stdin**（標準入力）は、プログラムへ渡されるデータの入り口で、デフォルトではキーボードからの入力です。

**stdout**（標準出力）は、プログラムが出す通常の結果で、デフォルトではターミナルへの表示です。

**stderr**（標準エラー出力）は、プログラムが出すエラーメッセージで、stdoutとは別の経路として扱われます。

```
キーボード入力
    ↓
  stdin
    ↓
[ プログラム ]
   ↓     ↓
stdout  stderr
   ↓     ↓
ターミナル画面に表示
```

stdoutとstderrは別の経路なので、両方をまとめて扱いたい場合は明示的に指定する必要があります。

```bash
ls /nonexistent 2>&1
```

`2>&1` は「stderr（番号2）をstdoutの行き先（番号1）と同じにする」という指定です。

これにより、エラーメッセージと通常の出力を同じ流れとして扱えるようになります。

## パイプとリダイレクト

CLIが強力なのは、プログラムどうしを自由につなげられる点にあります。

**パイプ**（`|`）は、あるプログラムのstdoutを、別のプログラムのstdinにつなぎます。

```bash
cat bigfile.txt | grep "error" | wc -l
```

このコマンドでは、`cat` が `bigfile.txt` の内容を出力し、その出力を `grep "error"` が受け取って「error」を含む行だけを抽出し、その結果を `wc -l` が受け取って行数を数えています。

**リダイレクト**（`>`、`>>`）は、stdoutをファイルに書き込みます。

```bash
ls > file_list.txt
ls >> file_list.txt
```

`>` は指定したファイルを上書きし、`>>` は既存の内容の末尾に追記します。

**エラーリダイレクト**（`2>`）は、stderrをファイルに書き込みます。

```bash
program 2> errors.log
program > out.log 2>&1
```

1行目はエラーメッセージだけを `errors.log` に保存します。

2行目はstdoutを `out.log` に書き込んだうえで、stderrもその同じファイルにまとめています。

パイプとリダイレクトは組み合わせて使えます。

```bash
cat data.txt | grep -v "^#" | sort | uniq > result.txt 2> errors.log
```

このコマンドは、コメント行（`#`で始まる行）を除外し、内容をソートし、重複する行を取り除いた結果を `result.txt` に保存し、その過程で出たエラーを `errors.log` に保存します。

`chmod`、`ps`、`kill` の詳しい使い方は、パーミッションとプロセスの記事で扱った内容です。

これらのコマンドは、パイプで組み合わせるとさらに役に立ちます。

```bash
cat /var/log/syslog | grep "error" | tail -20
```

このコマンドは、ログの中から「error」を含む行を抽出し、その最後の20行を表示します。

```bash
find /home -name "*.log" -mtime +30 -delete
```

このコマンドは、30日以上前に変更された `.log` ファイルを削除します。

削除を伴うコマンドなので、実行前に対象を確認してから使う必要があります。

```bash
ps aux | grep nginx | grep -v grep
```

このコマンドは、実行中のnginxプロセスを探します。

`grep -v grep` は、検索結果の中に `grep nginx` というコマンド自身が紛れ込むのを除外する書き方です。

<Callout title="シェルの補完機能">
bashやzshでは、コマンド名やファイル名を途中まで入力してから `Tab` キーを押すと、残りを自動的に補完してくれます。

`ls /ho` と入力して `Tab` を押すと `/home` に補完されます。

長いパスやコマンド名を毎回最後まで打つ必要がなくなるため、操作の速度が上がります。
</Callout>

## PATH環境変数とコマンドの実行場所

`ls` というコマンドは、実際にはディスク上のどこかにある実行ファイルです。

```bash
which ls
```

```
/usr/bin/ls
```

```bash
file /usr/bin/ls
```

```
/usr/bin/ls: ELF 64-bit LSB executable
```

`ls` の実体は `/usr/bin/ls` という実行ファイルです。

毎回フルパスを打たなくても `ls` だけで実行できるのは、**PATH**という環境変数のおかげです。

```bash
echo $PATH
```

```
/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```

シェルは、コマンドが入力されると、PATHに列挙されたディレクトリを左から順に検索します。

最初に見つかった実行ファイルが実行されます。

```
/usr/local/sbin/ls   見つからない
/usr/local/bin/ls    見つからない
/usr/sbin/ls         見つからない
/usr/bin/ls          見つかった。これを実行する
```

自作のスクリプトをコマンドのように使いたい場合は、PATHに含まれるディレクトリ（たとえば `/usr/local/bin`）に置くか、PATH自体に新しいディレクトリを追加します。

```bash
export PATH="/my/scripts:$PATH"
```

このコマンドを `~/.bashrc` に書いておくと、bashを起動するたびにPATHが更新され、`/my/scripts` の中のコマンドも名前だけで実行できるようになります。

## シェルスクリプトの基本形

コマンドを**スクリプトファイル**に書いておくことで、複数の処理をまとめて自動化できます。

```bash
#!/bin/bash
SOURCE="/home/alice/documents"
DEST="/backup/documents"

if [ -d "$SOURCE" ]; then
    cp -r "$SOURCE"/* "$DEST"
    echo "Backup completed at $(date)"
else
    echo "Source directory not found" >&2
    exit 1
fi
```

このスクリプトは、`SOURCE` で指定したディレクトリが存在するかを確認し、存在すればその内容を `DEST` にコピーし、存在しなければエラーメッセージをstderrに出して終了します。

```bash
chmod +x backup.sh
./backup.sh
```

スクリプトを実行するには、実行権限を付けてから直接呼び出すか、`bash backup.sh` のようにbashに読み込ませます。

シェルスクリプトには、条件分岐やループ、関数といった機能もあり、簡単な管理作業であれば十分な処理が書けます。

より複雑な処理が必要になった場合は、PythonやGoのような汎用プログラミング言語に切り替えるのが一般的です。

## ここまでのまとめ

**シェル**は、文字で入力したコマンドをOSに伝え、結果を返すプログラムで、同じ操作を繰り返す自動化において、マウス操作よりも効率的です。

bash、zsh、sh、dashなど複数の実装があり、互換性を重視するスクリプトでは `sh` と明示してbash特有の機能を避けます。

プログラムの入出力には**stdin**、**stdout**、**stderr**という3つの経路があり、**パイプ**（`|`）でプログラムどうしをつなぎ、**リダイレクト**（`>`、`>>`、`2>`）でファイルへの出力を制御します。

**PATH**環境変数は、コマンド名だけで実行ファイルを見つけるための検索順序を決めており、自作のコマンドもPATHに追加すれば同じように呼び出せます。

複数のコマンドをまとめたい場合は**シェルスクリプト**に書き、条件分岐やループを使ってより複雑な処理を組み立てられます。
