## サーバーが応答しない。SSHでログインはできる。次に何をするか

本番環境のサーバーが応答しなくなったとき、画面には何も手がかりが出てきません。

ブラウザでアクセスしても真っ白、あるいはタイムアウトするだけです。

ただ、サーバーにSSHでログインできるなら、手がかりはサーバーの中に残っています。

サーバー上で動いているプログラムは、自分が何をしたか、何のエラーが起きたかを**ログ**というファイルに書き残しているからです。

障害の原因を特定する作業の多くは、結局「どのログを、どこで、どう読むか」という知識に行き着きます。

この記事では、Linuxのログがどこに置かれていて、どうやって読むのかを説明します。

## ログファイルは `/var/log` の下にまとまっている

Linuxでは、システムやアプリケーションが出力するログファイルが、`/var/log/` というディレクトリの下に集約されています。

```bash
ls /var/log/
```

```
syslog              システム全般のログ
auth.log            ログインやsudoなど認証関連のログ
kern.log            カーネルのログ
nginx/              Nginx（Webサーバー）のログディレクトリ
apache2/            Apacheのログディレクトリ
mysql/              MySQLのログディレクトリ
```

代表的なログファイルを整理します。

## journalctlで見るログ

DebianやUbuntu、CentOSなど現在主流のLinuxディストリビューションは、**systemd**という仕組みでサービスを起動して管理しています。

systemdには**journald**という付属のログ収集機能があり、`journalctl` コマンドでその内容を読みます。

systemdを使う環境では、`/var/log/syslog` に書かれる内容の多くが、journaldにも記録されています。

基本的な使い方を見ていきます。

```bash
journalctl
```

何も指定しなければ、記録されているすべてのログを古い順に表示します。

```bash
journalctl -n 50
```

直近50行だけを表示します。

```bash
journalctl -f
```

`tail -f` と同様に、新しいログが記録されるたびに画面に追加していきます。

```bash
journalctl -u nginx
journalctl -u mysql
```

`-u` に続けてサービス名を指定すると、そのサービスのログだけに絞り込めます。

```bash
journalctl -p err
```

`-p` で優先度を指定すると、その優先度以上のログだけに絞り込めます。

```bash
journalctl --since "1 hour ago"
journalctl --since "2026-06-20 10:00:00" --until "2026-06-20 11:00:00"
```

`--since` と `--until` で、時間範囲を指定して絞り込めます。

<Callout title="journalctlとテキストログファイルの違い">
従来のテキストログファイルは、行ごとに自由な書式でメッセージが書かれているだけです。

journalctlが管理するログは、タイムスタンプ、優先度、出力元のサービス名といった情報が構造化された形で記録されています。

このため、`-u` や `-p` のような条件での絞り込みが正確に行えます。

ただし、再起動をまたいでログを残すには、`/var/log/journal/` への保存設定をあらかじめ有効にしておく必要があります。
</Callout>

## tail -fでいま起きていることを見る

ログファイルが大きいと、すべてを読むのは現実的ではありません。

**最新の部分だけをリアルタイムに追跡する**には、`tail -f` を使います。

```bash
tail -f /var/log/nginx/error.log
```

```bash
tail -f -n 20 /var/log/nginx/error.log
```

`-n 20` を付けると、まず最新20行を表示してから監視を始めます。

`tail -f` を実行している間、ログファイルに新しい行が追記されるたびに、その内容が画面に表示されます。

サーバーで起きていることをその場で観察するのに向いた方法です。

終了するときは `Ctrl+C` を押します。

複数のログファイルを同時に監視したいときは、tmuxやscreenで画面を分割するか、別々のターミナルタブでそれぞれ `tail -f` を実行します。

```bash
tail -f /var/log/syslog &
tail -f /var/log/nginx/error.log &
wait
```

末尾の `&` はそのコマンドをバックグラウンドで実行する指定で、`wait` はバックグラウンドのジョブがすべて終わるまで待つ指定です。

## grepでログを絞り込む

ログファイルの量は多く、目的の情報だけを抽出するには `grep` を使います。

```bash
grep "ERROR" /var/log/syslog
```

`ERROR` という文字列を含む行だけを表示します。

```bash
grep "203.0.113.42" /var/log/nginx/access.log
```

特定のIPアドレスからのアクセスだけを表示します。

```bash
grep -v "DEBUG" /var/log/app.log
```

`-v` を付けると、指定した文字列を含まない行だけを表示します。

パイプでコマンドをつなげば、抽出した結果をさらに加工できます。

```bash
grep "ERROR" /var/log/syslog | awk '{print $NF}' | sort | uniq -c
```

この一行は、ERRORを含む行から最後のフィールドを取り出し、並べ替えて、同じ内容が何回出てきたかを数えています。

```bash
tail -100 /var/log/syslog | grep "ERROR"
```

直近100行に絞ったうえでERRORを探すと、巨大なログファイル全体を検索するより速く済みます。

本番環境では、INFO以上のレベルだけを記録するのが一般的です。

DEBUGレベルの出力はログの量が膨大になるため、開発環境や、問題を詳しく調べる必要があるときだけ有効にします。

```bash
journalctl -p warn
journalctl -p err..crit
```

`-p err..crit` のように範囲を指定すると、ERROR以上CRITICAL以下のログだけに絞り込めます。

## logrotateによるログローテーション

ログは毎日増え続けます。

放置すればディスクの空き容量がなくなり、サーバー自体が動かなくなることもあります。

**ログローテーション**とは、古いログを圧縮したり削除したりして、新しいログのための領域を確保する仕組みです。

Linuxでは `logrotate` というツールが、設定に従って自動的にこの処理を実行します。

```bash
cat /etc/logrotate.d/nginx
```

```
/var/log/nginx/*.log {
    daily
    rotate 14
    compress
    delaycompress
    notifempty
    create 0640 www-data adm
    sharedscripts
    postrotate
        systemctl reload nginx > /dev/null 2>&1 || true
    endscript
}
```

この設定は、Nginxのログを毎日（`daily`）ローテーションし、14世代（`rotate 14`）保持し、古いログを圧縮（`compress`）するという内容です。

`delaycompress` は当日分のログをすぐには圧縮しない指定で、`notifempty` は空のログファイルをローテーションしない指定です。

ローテーション後のディレクトリは次のようになります。

```bash
ls -la /var/log/nginx/
```

```
access.log                 現在のログ
access.log.1               1日前のログ
access.log.2.gz            2日前のログ（圧縮済み）
access.log.3.gz
access.log.15.gz           15日前。設定上は14世代までなので次回削除される
```

圧縮された過去のログを読むには、`gunzip` で展開しながら表示します。

```bash
gunzip -c /var/log/nginx/access.log.2.gz | grep "203.0.113.42"
```

## サーバーが応答しないときに見るログの順序

サーバーが応答しないときの調査は、次の順序で進めると手がかりにたどり着きやすくなります。

最初に、systemdのログでサービスが何で止まったかを確認します。

```bash
journalctl -p err -n 50
journalctl -u nginx -n 50
journalctl -u mysql -n 50
```

ERROR以上のメッセージを直近50行確認すると、サービスが止まった理由がここで判明することが多くあります。

次に、syslogでシステム全体のイベントを確認します。

```bash
tail -100 /var/log/syslog | grep -E "ERROR|CRITICAL"
```

ディスク容量の不足やメモリ不足、カーネルパニックといった、システム全体に関わる問題はここに記録されます。

続いて、該当するサービスのエラーログを直接確認します。

```bash
tail -f /var/log/nginx/error.log
tail -f /var/log/mysql/error.log
```

アプリケーション固有のエラーは、ここでより詳しく読み取れます。

最後に、アクセスログで障害がいつから始まったかを確認します。

```bash
tail /var/log/nginx/access.log
```

記録されている最後のアクセス時刻から、サービスがいつ止まったかを推測できます。

## ローカルのログは証拠として絶対ではない

ここで、よくある誤解を訂正しておきます。

ローカルのログファイル（`/var/log` 以下）は、root権限を持つユーザーであれば書き換えや削除ができます。

つまり「ログに記録されている内容」がそのまま「実際に起きたことの証拠」であるとは限りません。

悪意のあるrootユーザーや、攻撃によってroot権限を奪った第三者は、自分の操作の痕跡をログから消すことができます。

セキュリティ対応の現場では「侵害の痕跡をそのまま残しておく」という原則がありますが、ローカルログだけに頼っていると、その原則自体が成り立たなくなる場合があります。

ログの改ざんを防ぐ、あるいは検知する方法には次のようなものがあります。

**リモートsyslog**は、ログを別のサーバーに転送して保存する方法です。

ローカルのログが書き換えられても、転送先のサーバーは独立しているため影響を受けません。

**WORM（Write Once Read Many）ストレージ**は、一度書き込んだデータを上書きできない記憶領域です。

クラウドストレージの不変ログ機能などがこれに当たります。

**監査ログ（Audit daemon）**は、通常のログよりも詳細な操作記録を残す仕組みで、改ざんが難しいように設計されています。

本番環境の運用では、ローカルのログだけで完結させず、ログを集中管理する仕組みを検討する価値があります。

## ここまでのまとめ

Linuxのログファイルは `/var/log/` の下に集約され、syslog、auth.log、kern.log、各アプリケーションのログディレクトリなどに分かれています。

systemd環境では `journalctl` が標準のログ管理ツールで、`-u` でサービスを、`-p` で優先度を、`--since` で時間範囲を指定して絞り込めます。

`tail -f` は最新のログをリアルタイムに追跡する方法で、トラブルがいま起きている最中の観察に向いています。

`grep` を組み合わせれば、ERROR、特定のIPアドレス、特定の時刻といった条件でログを絞り込めます。

ログレベルはDEBUG、INFO、WARN、ERROR、CRITICALの順に重要度が上がり、本番環境ではINFO以上を記録するのが一般的です。

`logrotate` は古いログを自動的に圧縮したり削除したりして、ディスク容量を守ります。

ローカルのログはroot権限で改ざんできるため、リモートsyslogやWORMストレージのような、改ざんに強い仕組みも合わせて検討する必要があります。
