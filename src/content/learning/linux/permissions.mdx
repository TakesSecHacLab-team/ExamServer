## 「chmod 777 にすれば権限エラーが消える」という解決策が危険な理由

プログラムが「Permission denied」と出て動かないとき、ネットでよくある解決策がこれです。

```
chmod 777 filename
```

「なるほど、777 にしたら動いた！」——でも、その瞬間に何を開放したのか理解してないと、取り返しのつかないセキュリティ穴を開けてしまいます。ここから、Linux のパーミッション（権限）の仕組みと、なぜ 777 が危険なのかを詳しく見ます。

## rwx の意味——読む・書く・実行の3つの操作

パーミッションは3つの権限の組み合わせです。

- **r**（read）＝ 読み取り。ファイルの場合は中身を見る。ディレクトリの場合はリスト表示（ls）ができる
- **w**（write）＝ 書き込み。ファイルの場合は編集・削除ができる。ディレクトリの場合はファイルの作成・削除ができる
- **x**（execute）＝ 実行。ファイルの場合は実行ファイルとして実行できる。ディレクトリの場合はその中に「入る」（アクセス）ことができる

ここ、初心者がよく混同する：「ディレクトリの x ってなに？」ですが、**x がないと ls もできない** です。r（リスト表示）はできても、ディレクトリの中身にアクセスできない。実例で示します。

```
# r-x (読み取り+実行) → リスト表示はできるし、中身にもアクセスできる
ls /home/alice

# r-- (読み取りのみ) → リスト表示はできるが、中身には入れない
ls /home/alice   # 成功
cd /home/alice   # Permission denied

# --x (実行のみ) → リスト表示はできないが、中身には入れる
cd /home/alice      # 成功
ls /home/alice      # Permission denied
```

気持ち悪いですが、これが仕様。

## ユーザー / グループ / その他の3層

パーミッションは3つの主体に対して定義されます。

- **u**（user）＝ ファイルの所有者
- **g**（group）＝ ファイルが所属するグループ
- **o**（other）＝ 上記以外のすべてのユーザー

例えば、私（alice）が作ったファイル `secret.txt` は、デフォルトではこんなパーミッションを持ちます：

```
-rw-r--r-- 1 alice alice
```

読み方：
- **u: rw-** （alice は読み取り・書き込みOK、実行はNG）
- **g: r--** （alice が所属するグループは読み取りのみOK）
- **o: r--** （その他のすべてのユーザーも読み取りのみOK）

## 数値表記（755, 644, 777）の読み方

パーミッションを短く表すために、r=4, w=2, x=1 として足し合わせる記法があります。

- **r (read)** = 4
- **w (write)** = 2
- **x (execute)** = 1

例えば `rwx` は 4+2+1 = 7。`rw-` は 4+2 = 6。`r--` は 4。

3つの主体それぞれの数字を並べるのが `chmod` の数値表記です。

```
chmod 755 script.sh
  ↓ ↓ ↓
  7 5 5
  user(rwx) group(r-x) other(r-x)
```

つまり：
- user: r(4) + w(2) + x(1) = 7 → rwx
- group: r(4) + x(1) = 5 → r-x
- other: r(4) + x(1) = 5 → r-x

よく使われるパターン：

| 数値 | 意味 | 使い方 |
|---|---|---|
| **644** | rw- r-- r-- | 通常のテキストファイル（所有者が読み書き、他はreadのみ） |
| **755** | rwx r-x r-x | 実行ファイルやスクリプト（所有者は全権、他は読み実行のみ） |
| **600** | rw- --- --- | 秘密鍵など（所有者だけが読み書き） |
| **700** | rwx --- --- | ホームディレクトリ（所有者だけが全アクセス） |
| **777** | rwx rwx rwx | **全員が全権。超危険** |

## 実例：ls -l の出力の読み方

`ls -l` でファイル一覧を見ると、左にこんな文字列が出ます。

```
-rw-r--r-- 1 alice alice 1234 Jun 20 10:30 config.txt
^^^^^^^^^^^^^^^
パーミッション部分
```

左から順に：

| 文字 | 意味 |
|---|---|
| **-** | ファイルタイプ。`-` はふつうのファイル、`d` はディレクトリ、`l` はシンボリックリンク |
| **rw-** | user（所有者 alice）のパーミッション |
| **r--** | group のパーミッション |
| **r--** | other のパーミッション |
| **1** | ハードリンク数（前のレッスンのinode参照カウント） |
| **alice** | ファイルの所有者（owner） |
| **alice** | ファイルが所属するグループ |
| **1234** | ファイルサイズ（バイト） |
| **Jun 20 10:30** | 最終更新日時 |
| **config.txt** | ファイル名 |



## chmod と chown——パーミッション変更とオーナー変更

パーミッションを変更するコマンドが `chmod`。オーナーを変更するのが `chown`。

**chmod** の使い方（数値表記）：

```
chmod 644 config.txt       # 644 に変更
chmod 755 script.sh        # 755 に変更
chmod +x script.sh         # 実行ビット (x) を付与（現在のパーミッションはそのまま）
chmod -w config.txt        # 書き込み (w) を削除
```

**chown** の使い方（オーナー / グループ変更）：

```
chown alice config.txt     # オーナーを alice に変更
chown alice:developers config.txt   # オーナーを alice、グループを developers に変更
chown :developers config.txt        # グループだけを developers に変更
```

chown は root 権限が必要。ふつうのユーザーは自分のファイルのオーナーを変えられません。

<Callout title="chmod と chown は root 権限が要るのか？">
  自分のファイル（自分が所有者）に対しては、chmod で権限を変更できます。chown でオーナーを変更する場合は、新しいオーナーになる必要があるため、root 権限が要ります。
</Callout>

## ディレクトリに対する x の意味——「中に入る」という操作

ここ、もう一度強調します。なぜならフォルダのアクセス権で詰まる人が多いから。

ディレクトリの **x** 権限は「そのディレクトリの中に入る（cd で移動する、その中のファイルにアクセスする）」という意味です。

```
drwx------ 2 alice alice    # alice だけが全権
drwxr-xr-x 2 alice alice    # alice は全権、他はリスト表示と中身へのアクセスOK
dr-xr-xr-x 2 alice alice    # alice は中身へのアクセスと実行のみ（リスト表示はできない）
```

最後の例は気持ち悪いですが、「ファイル自体は取得できるが、何が在るか一覧は見れない」という状態が作れます。セキュリティを厳しくしたいときに使われます。

## chmod 777 が危険な理由

では本題。`chmod 777` は何をするか。

```
chmod 777 filename
```

この1行で、**全ユーザー（world）が読み取り・書き込み・実行をすべてできる** という状態を作ります。

```
-rwxrwxrwx 1 alice alice
  ↑ ↑ ↑
  u g o（全員が rwx）
```

何が起きるか。

- **他のユーザーがファイルを改ざんできる**——わざとか事故か、ファイルの内容が変わってしまう
- **他のユーザーが実行ファイルを上書きできる**——マルウェアを埋め込まれたり、スクリプトを改ざんされたり
- **そのファイルが秘密鍵だったら**——だれでもあなたになりすまして通信できる

典型的なミスが「アップロードされたファイルに chmod 777 を付ける」です。「複数ユーザーからアクセスできるようにしたい」という気持ちはわかるけど、777 は過剰すぎる。ユーザーを group に追加して、group に読み取り権限を与えるのが正解です。

<Callout title="chmod 777 の代わりに">
  複数ユーザーがファイルにアクセスしたいなら：
  ```
  chmod 750 filename      # u(rwx) g(r-x) o(---)
  chown alice:developers filename  # alice が所有、developers グループがアクセス可
  ```
  または
  ```
  chmod 664 filename      # u(rw-) g(rw-) o(r--)
  ```
  ユースケースに応じて。
</Callout>

## setuid と setgid への軽い言及

パーミッションには特殊なビットがあります。ここでは概略だけ。

**setuid** (4000)：実行ファイルに付くと、その実行ファイルは「ファイルの所有者の権限で実行」されます。

例：`sudo` コマンド。実は `/usr/bin/sudo` は所有者が root で、setuid ビットが付いています。ふつうのユーザーが sudo を実行しても、root 権限で動く。これが `sudo` が権限昇格できる理由です。

```
-rwsr-xr-x 1 root root   # setuid ビット（r の代わりに s）
```

**setgid** も同様で、グループの権限で実行されます。

深掘りは後のセキュリティレッスンで。ここは「こういう仕組みがある」と知識として持っておくくらいで十分です。

## ここで押さえること

- **パーミッションは r/w/x の3つの権限** — 読み取り・書き込み・実行（またはディレクトリへのアクセス）
- **3つの主体** — user（所有者）・group（グループ）・other（その他）。それぞれに独立したパーミッションを設定できる
- **数値表記** — r=4, w=2, x=1。755 なら user が rwx、group と other が r-x
- **chmod 777 は超危険** — 全ユーザーが全権を持つ。悪意ある改ざん、秘密鍵漏洩など、様々な穴が開く
- **ディレクトリの x** — 「中に入る」という意味。r だけでは中身にアクセスできない
