## 指が2本だったら、どう数えるか

コンピュータを開いて最初に出くわす世界——それが「2進数」です。

実は、あなたが使っているOSもWebサイトもゲームもすべて、根っこでは0と1だけで動いています。なぜなら、コンピュータの心臓部であるCPUは、電気の「ON」と「OFF」の2つの状態しか認識できないように設計されているから。

ここからが本題です。もし人間の指が2本だけだったら、どうやって数を表現しますか？——それを理解することが、この章の最初の一歩です。

## なぜこれを学ぶのか

初心者がここで躓くと、後の全てが詰まります。

CTFの問題を開いたとき、こんな表記が出てきます：

```
\x41\x42\x43
```

あるいはログをhexdumpで見ると：

```
0x48656c6c6f
```

「何これ？」と思ったまま進むと、バイナリ問題、format string attack、暗号化データの解析——どれも手が出ません。でも「あ、この数字の表記法ね」と理解した瞬間、一気に霧が晴れます。

セキュリティエンジニア、リバースエンジニア、低レイヤー開発者。こういう職種にキャリアを考えているなら、ここは避けられない知識です。

## 2進数：0と1だけの世界

あなたが日常で使う数え方は「10進数」です。0, 1, 2, ..., 9 の10個の記号を使って、足りなくなると左に一桁増やす——あの方式です。

では「2進数」は何か。**0と1の2個の記号だけを使う数え方**です。

```
0
1
10    ← これは十進数の「2」
11    ← これは十進数の「3」
100   ← これは十進数の「4」
```

なぜこんなことするのか。コンピュータは電気信号を扱っています。高電位=1、低電位=0。この2つだけ。だから、データを「1」と「0」の並びで表現するのが、最も直接的で効率的なんです。

人間が後から「この並びは何を意味するのか」を読み取りやすくするために、10進数や16進数という「翻訳」を用意しているに過ぎません。CPUそのものは2進数で考えているわけではなく、ビットパターンに対して決められた操作（AND、OR、XORなど）をしているだけです。ここ、いつもミスリードされるポイント。

## 10進数 ↔ 2進数の変換

では、実際に変換してみましょう。

**10進数 → 2進数**

例：10進数の「13」を2進数に変換

13 ÷ 2 = 6 あまり 1
6 ÷ 2 = 3 あまり 0
3 ÷ 2 = 1 あまり 1
1 ÷ 2 = 0 あまり 1

下から読むと：**1101**

検証：1×2³ + 1×2² + 0×2¹ + 1×2⁰ = 8 + 4 + 0 + 1 = 13 ✓

**2進数 → 10進数**

例：2進数の「1010」を10進数に変換

```
1010 = 1×2³ + 0×2² + 1×2¹ + 0×2⁰
     = 8 + 0 + 2 + 0
     = 10
```

単純ですが、この感覚を身に付けることが土台になります。

## ビットとバイト：コンピュータの最小単位

ここ、最初みんな混同するポイントです。

- **ビット（bit）**：0または1の「1つ」を表す。コンピュータの最小単位。「binary digit」の略。
- **バイト（byte）**：ビット8個をまとめたもの。`1 byte = 8 bits`。

なぜ8ビット？——歴史的な理由です。現代的には「効率よくデータをまとめられる単位」として定着しています。

1バイトで表せる値は 0 ～ 255（2進数では `00000000` ～ `11111111`）。

<Callout title="なぜ255か？">
  8ビットで表せる組み合わせは2⁸ = 256通り。0から255だと256個の値が表現できます。最大値が255になるのは、0を含めるからです（0, 1, 2, ..., 255）。
</Callout>

例えば、文字「A」を保存するには1バイト使います。ファイルサイズが「5KB」なら、5 × 1024バイト = 5 × 1024 × 8ビット = 40,960ビット——という計算ができます。

ここで大事なのは「ビット≠バイト」。特にネットワーク通信では通信速度を「Mbps」（メガビット毎秒）で表すので、ファイルサイズの「MB」（メガバイト）と混同しやすい。1バイト = 8ビットを常に頭の片隅に置いてください。

## 16進数（hex）：2進数を短く書く方法

ここからが本当に大事なポイントです。

2進数は確かにコンピュータネイティブですが、人間が読むには長すぎます。

```
11111111111111111111111111111111
```

これは何か。32ビット全部が1——ですが、見た目で判断できません。

ここで登場するのが **16進数** です。使う記号は 0 ～ 9、A ～ F。合わせて16個。

**なぜ16か？**——2進数4ビルが16進数1桁にちょうど対応するから。2⁴ = 16。

```
2進数       16進数
0000    →  0
0001    →  1
0010    →  2
...
1001    →  9
1010    →  A
1011    →  B
1100    →  C
1101    →  D
1110    →  E
1111    →  F
```

では、先ほどの32ビットを16進数で書くと：

```
11111111 11111111 11111111 11111111  （2進数）
   FF       FF       FF       FF      （16進数）
```

大幅に短くなりました。これが使われる理由です。



## 0x プレフィックス：「ここから16進数だよ」の合図

あなたがセキュリティを学んでいると、こういう書き方をよく見ます：

```
0x48656c6c6f
0x1f
0xdeadbeef
```

先頭の `0x` は「これから書く数字は16進数ですよ」という宣言です。10進数で「0」から始まる数字と混同しないようにするため。

例：
- `0x10` = 16進数の「10」= 10進数の「16」
- `0x20` = 16進数の「20」= 10進数の「32」
- `0xff` = 16進数の「ff」= 10進数の「255」

プログラミング言語によっては `0o`（8進数）、`0b`（2進数）みたいなプレフィックスもあります。

```python
print(0x10)   # 16進数 → 出力：16
print(0o10)   # 8進数  → 出力：8
print(0b10)   # 2進数  → 出力：2
```

## \xNN 表記：バイト列を直接書く

では、こういう表記はどうか。

```
\x41\x42\x43
```

これは **バイト列** を直接書く方法です。`\x` の後に続く2桁の16進数が1バイトを表します。

- `\x41` = 16進数で「41」= 10進数で「65」
- `\x42` = 16進数で「42」= 10進数で「66」
- `\x43` = 16進数で「43」= 10進数で「67」

そして、ASCII文字コード表を見ると：

- 65 = 'A'
- 66 = 'B'
- 67 = 'C'

つまり、`\x41\x42\x43` は文字列「ABC」そのものです。

<Callout title="ASCII文字コード表を手元に">
  最初は覚える必要はありません。ただし、CTF問題を解くときに「0x41 = 'A'」「0x20 = スペース」「0x0a = 改行」くらいは知っていると、格段に読みやすくなります。
</Callout>

ファイルフォーマットの解析やシェルコード作成のときに、この表記が頻出します。「バイトの値を16進数で表しているんだな」と理解していれば、怖くありません。

## ASCII文字コードの具体例

繰り返しになりますが、ここは超重要なので例をもう一つ。

文字列「Hello」を`\xNN` 形式で書くと：

```
H → 0x48 → \x48
e → 0x65 → \x65
l → 0x6c → \x6c
l → 0x6c → \x6c
o → 0x6f → \x6f

= \x48\x65\x6c\x6c\x6f
```

これはバイナリエディタで見える「生のバイト」そのもの。プログラムの「目」で見えるのは`\x48`という文字列ではなく、バイトそのものです。

ここが腑に落ちるとバイナリ分析の大半が解ける。

## ここで押さえること

- **2進数**は0と1だけで数える。コンピュータの電気信号（ON/OFF）と対応している
- **ビット**（bit）は0か1の1個。**バイト**（byte）は8ビット。ネットワーク速度がbps単位なのに対しファイルサイズがB単位なのに注意
- **16進数**（hex）は2進数4ビルを1文字で表す。短く、読みやすい
- **0x プレフィックス**は「16進数」の宣言。`0x1f` なら10進数で「31」
- **\xNN 表記**はバイト列。`\x41` は10進数で「65」= 文字'A'。ファイルやシェルコードで頻出
- ASCII文字コードを読める力——「0x48 = 'H'」が瞬時に分かると、バイナリ分析がぐっと楽になる
