## プログラムが終わると変数はなぜ消えるのか

次のコードを実行すると、変数 `x` に42が入ります。

```c
int x = 42;
```

このプログラムを終了し、もう一度起動して `x` の値を確認しても、42は出てきません。

`x` という変数は初期化されていない状態から始まります。

一方、ファイルに書き出したデータはそうなりません。

```c
FILE *f = fopen("data.txt", "w");
fprintf(f, "Hello\n");
fclose(f);
```

このプログラムが終了したあとも `data.txt` は残ります。

10日後にこのファイルを開いても、「Hello」という内容がそこにあります。

変数とファイルで、この違いが生まれる理由は、データが置かれている場所が異なるからです。

変数は**メモリ**（RAM）に置かれ、ファイルは**ストレージ**（SSDやHDD）に置かれています。

メモリとストレージの違いを理解することは、この先の章で扱うバッファオーバーフローやメモリダンプ解析といったセキュリティ技術の前提になります。

CTF（セキュリティ技術を競う形式の演習）では、次のような問題がよく出題されます。

> このプログラムには脆弱性があります。特定のバッファに大量のデータを入力すると、プログラムの動作を制御できます。

なぜ入力データの量を増やすだけでプログラムの動作を制御できるのか。

その理由は、プログラム中の変数や関数の戻り先がメモリ上の決まった位置に置かれていて、その位置のデータを書き換えることで、プログラムに別の動作を行わせられるからです。

仕組みの詳細は後の章で扱いますが、その土台となるメモリの構造を、この章でまず押さえます。

## メモリとストレージの違い

パソコンの内部には、データを保持する装置が複数あります。

**RAM**（メモリ）は読み書きが高速ですが、電源を切るとデータが消えます。

**SSDやHDD**（ストレージ）は読み書きがRAMより低速ですが、電源を切ってもデータが残ります。

プログラムが動作しているとき、実行する命令（コード）も、操作対象の変数も、すべてRAMに置かれています。

CPUはRAMから命令を読み出し、RAM上のデータを読み書きしながら処理を進めます。

RAMの容量は有限で、現在のパソコンではおおむね8GBから32GB程度です。

電源を切った瞬間、RAM上のデータはすべて失われます。

このため、後で必要になるデータをプログラムが`fopen`や`fwrite`でストレージに保存する作業を、明示的に行う必要があります。

メモリは、プログラムが動いている間だけ使う作業領域です。

## アドレス：メモリ上の位置を示す番号

メモリの中身がどう配置されているかを見ていきます。

メモリは有限個のバイトの集まりで、各バイトには通し番号が振られています。

```
番地       格納されている値
0x1000  →  00
0x1001  →  01
0x1002  →  FF
0x1003  →  42
0x1004  →  00
```

この番号を**アドレス**（メモリアドレス）と呼びます。

住所のように、アドレス`0x1000`を指定すれば、そこに格納されている1バイトの値を読み書きできます。

プログラムの命令そのものも、メモリ上のどこかのアドレスに置かれています。

「ある関数はメモリのこのアドレスから始まる」という形で、命令の置き場所もアドレスで表されます。

後の章で扱うバッファオーバーフロー攻撃や、関数の戻り先アドレスの書き換えは、結局のところ「メモリの特定のアドレスに、想定されていないデータを書き込む」操作です。

アドレスという考え方が腑に落ちると、その先の説明がそのまま読めるようになります。

## 仮想アドレス空間：プロセスごとに用意された専用の番地

パソコンでは複数のプログラムが同時に動いています。

ブラウザ、テキストエディタ、メディアプレイヤーが、それぞれメモリを使っています。

ここで、すべてのプログラムが物理的に同じアドレス`0x1000`を直接読み書きするとどうなるか考えます。

複数のプログラムが同じ場所を取り合い、互いのデータを上書きしてしまいます。

この衝突を避けるために、OS（オペレーティングシステム）は各プログラムに**仮想アドレス空間**を用意します。

実行中の各プログラム（**プロセス**と呼びます）は、自分だけが全メモリを使っているかのように見える、専用のアドレス空間を割り当てられます。

```
ブラウザのプロセス
  アドレス 0x1000 → ブラウザのデータ

テキストエディタのプロセス
  アドレス 0x1000 → テキストエディタのデータ
```

両者は同じ`0x1000`という番号を使っていますが、OSがそれぞれを別の物理メモリ領域に割り振っているので、実際には衝突しません。

プログラムが「アドレス`0x1000`を読め」と指示すると、OSがそのプロセスの`0x1000`を実際の物理メモリ上の位置に変換してから処理します。

この変換のしくみを**メモリ保護**と呼びます。

あるプログラムが暴走して自分のメモリ領域を壊しても、他のプログラムには影響しません。

これが、現在のOSがプロセスごとにメモリを隔離している理由です。

## プログラム1つの中のメモリの分割

ここまでは複数のプログラム間の話でした。

1つのプログラムが動作しているとき、そのプログラムに割り当てられたメモリ領域自体も、用途別に分割されています。

典型的な構成は次の通りです（下に書いたものほどアドレスが小さく、上に書いたものほどアドレスが大きい領域です）。

```
┌──────────────────┐  アドレスの大きい側
│      スタック     │  関数の引数、ローカル変数が積まれる
├──────────────────┤
│   未使用領域      │
├──────────────────┤
│      ヒープ       │  動的に確保したメモリ（malloc/new）
├──────────────────┤
│    データ領域     │  グローバル変数、初期化データ
├──────────────────┤
│    コード領域     │  プログラムの機械語命令
└──────────────────┘  アドレスの小さい側
```

**コード領域**には、プログラム自体の機械語命令が入っています。

CPUはここから命令を読み出して実行します。

**データ領域**には、グローバル変数や定数が置かれます。

**ヒープ**は`malloc()`のような関数で動的に確保したメモリで、アドレスの小さい側から大きい側へ向かって領域が広がっていきます。

**スタック**は関数呼び出しやローカル変数のための領域で、アドレスの大きい側から小さい側へ向かって領域が広がっていきます。

ヒープとスタックは互いに向かい合う方向に成長するので、両者が伸び続けて衝突する位置まで近づくと、メモリが確保できなくなり、プログラムが異常終了します。

スタックとヒープそれぞれの詳しい動きは、次の章で扱います。

ここでは、プログラムのメモリが領域ごとに役割分担されているという構造だけを押さえておきます。

<Callout title="64ビットOSのアドレス空間">
  64ビットプロセスが理論上扱えるアドレス空間は2の64乗バイト、約18エクサバイトです。実際にOSが使う範囲はそれより小さく、数十テラバイト程度にとどまります。個々のプログラムが実際に消費するメモリは数ギガバイト程度で収まることが多く、この余裕で十分です。
</Callout>

## 32ビットと64ビット：アドレスの幅が決めるメモリの上限

「64ビットOSなら4GBを超えるメモリを使える」という説明を見たことがあるかもしれません。

この章で見てきたアドレスの考え方を使うと、その理由が正確にわかります。

**32ビットアーキテクチャ**では、1つのアドレスを32ビットの幅で表します。

32ビットで表現できる値の組み合わせ数は2の32乗、約43億通りです。

1つのアドレスにつき1バイトが対応するので、理論上扱える最大メモリ量はおよそ43億バイト、約4GBになります。

**64ビットアーキテクチャ**では、1つのアドレスを64ビットの幅で表します。

64ビットで表現できる値の組み合わせ数は2の64乗、約1844京通りです。

同じく1アドレスにつき1バイトが対応するので、理論上扱える最大メモリ量はおよそ1844京バイト、約16エクサバイトになります。

「64ビットの方が大きい数を扱えるから高性能」という説明では、何が大きくなっているのかがあいまいです。

正確には、アドレスを表すビット数が増えた分だけ、表現できるアドレスの個数が指数的に増え、その結果として扱えるメモリの上限が桁違いに広がっています。

実装上、現在のCPUは64ビットのアドレス空間をすべて使い切ってはおらず、実際に使える範囲は数テラバイト程度です。

それでも、上限を決めているのはアドレス空間の幅であり、CPUの計算速度や数値の大きさそのものではありません。

## この章のまとめ

**メモリ（RAM）**は、プログラムが動作している間だけデータを保持する作業領域です。

電源を切るとデータは消え、ストレージのように長期保存はできません。

**アドレス**は、メモリの各バイトに振られた番号です。

プログラムは「このアドレスのデータを読め」「このアドレスにこの値を書け」という形で、メモリを操作します。

**仮想アドレス空間**は、OSが各プロセスに用意する専用のアドレス空間です。

複数のプログラムが同じアドレス番号を使っても、実際の物理メモリ上では別の場所を指すように変換され、プロセス間の衝突を防いでいます。

1つのプログラムのメモリは、コード領域、データ領域、ヒープ、スタックに分かれています。

**32ビットと64ビット**の違いは、アドレスを表すビット数の違いです。

ビット数が増えるほど表現できるアドレスの個数が指数的に増え、扱えるメモリの上限が広がります。
