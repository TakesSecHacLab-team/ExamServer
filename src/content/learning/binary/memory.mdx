## 変数はなぜ消えるのか

あなたがプログラムを書いたとき、こんなコードを見たことがあると思います。

```c
int x = 42;
```

このプログラムが終わると、変数 `x` の値は消えます。再度起動して `x` にアクセスしても、42は出てきません。

一方、ファイルに書いたデータはどうか。

```c
FILE *f = fopen("data.txt", "w");
fprintf(f, "Hello\n");
fclose(f);
```

プログラムが終わっても、`data.txt` は残っています。10日後に読み込んでも「Hello」がそこにあります。

**この非対称性はなぜか。** 変数はメモリに置かれ、ファイルはディスクに置かれているから——その答えがこの章の全て。そしてこれを理解することが、セキュリティの半分を理解することでもあります。

## なぜこれを学ぶのか

メモリのしくみが分からないと、バッファオーバーフローの意味が分かりません。

CTFでこんな問題が出ます：

> このプログラムには脆弱性があります。特定のバッファに大量のデータを入力すると、プログラムの動作を制御できます。

「えっ、なぜ制御できるの？」——それはメモリ上に置かれた変数や関数のアドレスが固定で、その位置を書き換えることで別の動作を強制するから。

また、セキュリティ研究やマルウェア解析の世界では「メモリダンプ」という手法がよく使われます。実行中のプロセスのメモリ全体をファイルに落とし、その中身を分析する。実行中のデータがメモリのどこにあるか理解していないと、その分析も始まりません。

ここを押さえると、世界がグッと広がります。

## メモリ（RAM）とは何か

まず、正確に理解しましょう。

あなたのパソコンには、いくつかの「保管庫」があります。

- **RAM**（メモリ）：高速だが電源を切ると消える。プログラムの作業台
- **SSD / HDD**（ストレージ）：低速だが電源を切ても残る。長期保管庫

プログラムが動くとき、その指示（コード）も、操作する変数も、すべてRAMに置かれます。CPUはRAMから指示を読み込み、RAMのデータを読み書きして動作します。

電源を切った瞬間、RAMの内容は全て失われます。容量も限られています（例：16GB）。だからプログラムは終わるたびに「このデータは重要だからディスクに保存しよう」と `fopen` や `fwrite` をします。

**メモリは、プログラムの「一時的な作業台」。** ここがイメージの基本です。

## アドレス：メモリの番地

では、メモリの中はどうなっているのか。

メモリは無限ではなく、有限のバイト数を持っています。そしてそれぞれのバイトに「番地」が付いています。

```
アドレス    値（バイト）
0x1000  →  00
0x1001  →  01
0x1002  →  FF
0x1003  →  42
0x1004  →  00
...
```

この「番地」を **アドレス** または **メモリアドレス** と言います。郵便番号と番地みたいなもの。アドレス0x1000にアクセスすれば、そこに置かれたバイトが読める。「ここに何を書き込もう」と決めれば、そのアドレスに値を置ける。

プログラムのコード自体もメモリに置かれています。その場合、アドレスは「この関数はメモリのこのアドレスから始まる」という形で決まります。

後で出てくる「バッファオーバーフロー攻撃」や「リターンアドレスの書き換え」というのは、つまり「メモリのある特定のアドレスに意図しないデータを書き込む」ことです。ここが分かると、クリアです。

## 仮想アドレス空間：プロセスごとの「自分の世界」

ここ、最初は直感的じゃないので、ていねいに説明します。

あなたのパソコンで複数のプログラムが同時に動いています。Webブラウザ、テキストエディタ、メディアプレイヤー。それぞれがメモリを使っています。

もし全てのプログラムが「アドレス0x1000」を見たら、同じ場所を指してしまいます。これでは衝突して壊れます。

**そこで、OS（Operating System）が用意するのが「仮想アドレス空間」です。**

各プログラム（プロセス）は、あたかも自分だけで全メモリを使っているかのような「仮想的なアドレス空間」を与えられます。

```
ブラウザのプロセス
  アドレス0x1000 → ブラウザのデータ

テキストエディタのプロセス
  アドレス0x1000 → テキストエディタのデータ

（実は物理的には全く違うメモリ領域にあるが、見た目は同じアドレス）
```

つまり、プログラムが「アドレス0x1000を読め」と指示しても、OSが「ああ、このプロセスのアドレス0x1000なら、物理メモリのここだな」と変換してくれます。これを **メモリ保護** と呼びます。

一つのプログラムが暴走してメモリを壊しても、他のプログラムには影響しない——これが現代的なOSの基本原則。かなり賢い仕組みです。

## プログラムのメモリレイアウト

では、1つのプログラムが動くとき、メモリはどう分割されているのか。

典型的には以下のような構成です（下のアドレスが低い）：

```
┌──────────────────┐ ← アドレス空間の「上」（大きいアドレス）
│      スタック     │  関数の引数、ローカル変数が積まれる
├──────────────────┤
│   未使用領域      │
├──────────────────┤
│      ヒープ       │  動的に確保したメモリ（malloc/new）
├──────────────────┤
│    データ領域     │  グローバル変数、初期化データ
├──────────────────┤
│    コード領域     │  プログラムの機械語命令
└──────────────────┘ ← アドレス空間の「下」（小さいアドレス）
```

- **コード領域**：プログラム自体が入っている。CPU がここから命令を読んで実行
- **データ領域**：グローバル変数や定数が置かれる
- **ヒープ**：`malloc()` で動的確保したメモリ。下から上に向かって成長
- **スタック**：関数呼び出しやローカル変数。上から下に向かって成長

この「スタック」と「ヒープ」が近づいてきたとき、メモリが足りなくなり、プログラムはクラッシュします。

地味だけど超重要：メモリレイアウトを頭に入れておくと、後のセクションで出てくる「スタックオーバーフロー」が理解できます。

## 32ビットと64ビット：アドレス幅と使えるメモリ

ここで、「なぜ64ビットOSで4GB以上のRAMが使えるのか」という質問の正確な答えが出てきます。

**32ビットアーキテクチャ**：アドレスが32ビル幅
- 最大アドレス数 = 2³² = 約43億
- 1アドレス = 1バイト
- 理論上の最大メモリ = 43億バイト ≈ 4GB

**64ビットアーキテクチャ**：アドレスが64ビル幅
- 最大アドレス数 = 2⁶⁴ = 約1844京
- 1アドレス = 1バイト
- 理論上の最大メモリ = 1844京バイト ≈ 16EB（エクサバイト）

「64ビットなら大きい」ではなく、「アドレスを指すために64ビル必要だから、扱えるメモリ量が文字通り桁違いに増える」——これが正確な理由です。

実装上、現代のCPUは64ビルアドレスの一部しか使わないので、実際には数テラバイトまでです。でも原理的には、アドレス空間が広いから使えるメモリが増える。数学の話ではなく、**アドレス空間の幅** の話です。

## ここで押さえること

- **メモリ（RAM）**はプログラムの「作業台」。電源を切ると消える
- **アドレス**：メモリの各バイトに付いた番地。プログラムは「このアドレスのデータを読め / 書け」という指示を出す
- **仮想アドレス空間**：各プロセスに独立した「自分だけの世界」を見せるOSの仕組み。プロセス間の衝突を防ぐ
- **プログラムのメモリレイアウト**：コード領域・データ領域・ヒープ・スタックに分かれている
- **32ビット vs 64ビット**：アドレス幅の違い。64ビルなら2⁶⁴通りのアドレスを表現できるから、扱えるメモリが増える
