## 再帰の果てにあるもの

あなたがプログラムを学ぶとき、必ず「再帰関数」という概念に出くわします。関数が自分自身を呼び出す、という。

```c
int factorial(int n) {
  if (n <= 1) return 1;
  return n * factorial(n - 1);  // 自分自身を呼び出す
}
```

このコードで `factorial(50000)` を実行すると、大抵のマシンでクラッシュします。エラーメッセージ：

```
Stack overflow
```

「えっ、何が溢れたの？」——その答えがこのセクション全体です。

## なぜこれを学ぶのか

バッファオーバーフロー攻撃（古典的なセキュリティ脆弱性）は、スタックを理解していないと絶対に理解できません。

CTFでこんな問題が出ます：

> このプログラムに入力を与えると、任意のコードを実行できます。ペイロードを作成してください。

その「ペイロード」というのは何か。スタック上に積まれたリターンアドレスを上書きして、プログラムの制御フローを乗っ取るデータです。

スタックとヒープの違い、リターンアドレスの位置、バッファの配置——これらが分かると、「あ、ここに大量のデータを入れたら、ここを書き換えられるな」という攻撃が設計できます。

現代のセキュリティ防御（ASLR、スタックカナリア、DEP）も全て、この古典的な攻撃を防ぐための仕組みです。知らないと、その防御も理解できません。

## スタック：LIFO の世界

スタック（Stack）とは、**LIFO**（Last In, First Out）というデータ構造です。日本語では「後入れ先出し」。

実生活の例：食器棚に皿を積む。上に積まれた皿から取る。最後に置いた皿が最初に取られる——それがLIFO。

プログラムにおいてスタックは、メモリ上に実装されています。

```
  スタックポインタ（現在位置）
           ↓
  ┌───────────────┐
  │     値: 42    │ ← 最後に積まれたデータ（最初に取られる）
  ├───────────────┤
  │     値: 3.14  │
  ├───────────────┤
  │     値: 100   │ ← 最初に積まれたデータ
  └───────────────┘
```

データを積む操作を **PUSH**、取る操作を **POP** と呼びます。

## 関数呼び出しとスタックフレーム

では、プログラムが関数を呼び出すたびに何が起きるか。

```c
void greeting(char *name) {
  printf("Hello, %s\n", name);
}

int main() {
  greeting("Alice");
  return 0;
}
```

この `main()` が `greeting()` を呼び出すとき、スタックは以下のように動きます：

1. `greeting()` の引数（`"Alice"`へのポインタ）がスタックに PUSH される
2. **リターンアドレス** がスタックに PUSH される（「`greeting()` が終わったら `main()` のどこに戻るか」を示す値）
3. `greeting()` のローカル変数（例：フォーマット文字列）がスタックに PUSH される
4. `greeting()` の処理が実行される
5. `greeting()` が終わると、これらが逆順で POP される

この「引数 + リターンアドレス + ローカル変数」の束を **スタックフレーム** と呼びます。関数が呼ばれるたびに1つのスタックフレームが積まれ、関数が終わると破棄される。



<Callout title="リターンアドレスが重要">
  関数が終わるとき、「どこに戻るか」の情報がスタック上にあります。これを書き換えると、プログラムの実行フローを乗っ取れます。バッファオーバーフロー攻撃の本質。
</Callout>

## スタックポインタと深すぎる再帰

スタック上の「現在位置」を示すレジスタを **スタックポインタ**（SP）と呼びます。

再帰呼び出しが深くなるたびに、スタックフレームが1つ、また1つと積まれていきます。

```c
void deep_recursion(int n) {
  if (n == 0) return;
  deep_recursion(n - 1);  // 自分を呼び出す → スタックフレームが積まれる
}

deep_recursion(100000);  // この呼び出しだけで100,000個のフレームが積まれる
```

スタックの容量は有限です（Linux のデフォルト 8MB、Windows は 1MB）。100,000個のスタックフレームがスタック領域を埋め尽くし、さらに深い呼び出しが来ると、スタック領域の上限に達します。

そこで発生するのが **Stack Overflow**——スタックが「溢れた」状態。以降のプログラムの動作は未定義。大抵のOSはプロセスを強制終了します。

## ヒープ：動的にメモリを確保する

一方、**ヒープ** は異なる性質を持ちます。

スタックは「関数の呼び出しと戻りに自動で積み上げと破棄が起きる」ため、使うのが簡単です。一方、「事前に必要な容量が分からない」ケースはどうするのか。

```c
char *buffer = malloc(1024);  // 1024バイト、動的に確保
// ... 処理 ...
free(buffer);  // 使い終わったら明示的に解放
```

`malloc()` で確保したメモリはヒープに置かれます。スタックと異なり：

- **手動管理**：`free()` で明示的に解放する必要がある
- **容量が大きい**：OSが許す限り、かなり大容量のメモリを確保できる
- **生存期間が自由**：関数が終わっても、`free()` するまで生き続ける

```c
void example() {
  int *ptr = malloc(sizeof(int));  // ヒープから確保
  *ptr = 42;
  // 関数が終わっても ptr は有効（解放されていないから）
  return;
}
// ← ここで ptr へのアクセスは undefined。メモリリークが発生
```

## スタック vs ヒープ：使い分け

では、どちらを使うべきか。

| スタック | ヒープ |
|---|---|
| **速度** → 高速 | **速度** → 低速 |
| **大きさ** → 制限がある（数MB程度） | **大きさ** → 制限が大きい（GBオーダー） |
| **管理** → 自動（関数の終了で破棄） | **管理** → 手動（`free()` が必要） |
| 配列のサイズが固定 | 配列のサイズが動的 |
| ローカル変数 | 長く生き続ける必要があるデータ |

ローカル変数や固定サイズの配列はスタック。「実行時にサイズが決まる」「関数を超えて生き続ける」データはヒープ。

## メモリリークと二重解放

ヒープの怖いところ。

**メモリリーク**：`free()` し忘れ
```c
for (int i = 0; i < 1000000; i++) {
  char *buf = malloc(1024);
  // free(buf) を忘れた
}
// ← 1M × 1KB = 1GB のメモリが失われた
```

**二重解放**：同じメモリを2度 `free()`
```c
char *buf = malloc(1024);
free(buf);
free(buf);  // ← エラー。予測不能な動作
```

現代的な言語（Python, Java, Rust）は自動メモリ管理を提供し、こうした問題を減らしています。ただしCやC++での低レイヤー開発では自分で管理する必要があり、セキュリティ問題の温床になります。

<Callout title="Rustの登場">
  C/C++ の代替言語として注目されている Rust は、「所有権」という概念で、メモリリークも二重解放も、コンパイル時に防ぎます。難しいですが、セキュリティが必要な領域で急速に普及中。
</Callout>

## スタックオーバーフロー：本物の脆弱性

では、セキュリティで言う「Stack Overflow」とは何か。

これは、スタック上のバッファに意図より大量のデータを書き込み、**隣接するメモリ（特にリターンアドレス）を上書き** する攻撃です。

```c
void vulnerable() {
  char buffer[8];  // 8バイトのバッファ
  // ... ユーザー入力を buffer に読み込む ...
  gets(buffer);  // ← 危険。入力サイズをチェックしない
}
```

ユーザーが100バイトを入力したら、buffer は溢れ、その先のメモリを上書きします。その先には、リターンアドレスがあるかもしれません。

```
┌────────────────────┐
│  リターンアドレス   │ ← 上書きされた！
├────────────────────┤
│   ローカル変数等   │
├────────────────────┤
│   buffer (8bytes)  │ ← ここからオーバーフロー開始
└────────────────────┘
```

攻撃者が「ここに自分のコードを仕込んだメモリアドレスを入力」すれば、関数が終わったときに、プログラムはその攻撃者のコードを実行させられます。これが **バッファオーバーフロー攻撃** の仕組み。

1990年代から2000年代はこれが最も一般的な脆弱性でしたが、現代のOSやコンパイラは防御機構（ASLR、スタックカナリア、DEP）を備え、難しくなりました。ただし原理は同じです。

## ここで押さえること

- **スタック**は関数呼び出しで自動的に積み上げられるメモリ。LIFO 構造
- **スタックフレーム**：関数が呼ばれるたびに、引数・リターンアドレス・ローカル変数が積まれる
- **リターンアドレス**：関数が終わった後どこに戻るかを示す。これを書き換えると実行フローを乗っ取れる
- **スタックオーバーフロー**（再帰が深すぎる場合）：スタック容量を超える
- **ヒープ**は動的にメモリを確保。手動管理が必要（C/C++の場合）
- **バッファオーバーフロー攻撃**：スタック上のバッファに大量入力し、リターンアドレスを上書きして実行フローを乗っ取る古典的な脆弱性
