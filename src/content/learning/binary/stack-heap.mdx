## 再帰関数が深すぎるとなぜ落ちるのか

関数が自分自身を呼び出す書き方を、**再帰関数**と呼びます。

```c
int factorial(int n) {
  if (n <= 1) return 1;
  return n * factorial(n - 1);
}
```

この `factorial` を `factorial(50000)` のように大きな値で呼び出すと、多くの環境でプログラムが異常終了します。

エラーメッセージには次のような表示が出ます。

```
Stack overflow
```

「溢れた」と表示されている対象が何なのか、この章で順に確認していきます。

前章では、1つのプログラムのメモリがコード領域、データ領域、ヒープ、スタックに分かれていることを見ました。

この章では、そのうちスタックとヒープの内部で何が起きているかを見ていきます。

バッファオーバーフロー攻撃（古典的なセキュリティ脆弱性の一種）は、スタックの仕組みを把握していないと意味が読み取れません。

CTFでは次のような問題がよく出題されます。

> このプログラムに入力を与えると、任意のコードを実行できます。ペイロードを作成してください。

ここでの「ペイロード」とは、スタック上に置かれた関数の戻り先アドレスを上書きして、プログラムの制御の流れを乗っ取るために用意するデータです。

スタックとヒープの違い、戻り先アドレスがどこに置かれているか、バッファがどう配置されているかがわかると、「ここに大量のデータを入れれば、この先のアドレスを書き換えられる」という攻撃の道筋が見えるようになります。

現在のセキュリティ対策であるASLR、スタックカナリア、DEPも、すべてこの古典的な攻撃を防ぐための仕組みです。

これらの対策がなぜ必要なのかも、スタックの構造を理解してから読むと筋が通ります。

## スタック：最後に積んだものを最初に取り出す構造

**スタック**は、**LIFO**（Last In, First Out、後入れ先出し）という規則に従ってデータを並べる構造です。

食器棚に皿を積み重ねる動作がこれにあたります。

皿は上から積まれ、使うときも上から取られます。

最後に置いた皿が、最初に取り出されます。

プログラムの中のスタックは、メモリ上の一領域として実装されています。

```
  スタックポインタ（現在位置）
           ↓
  ┌───────────────┐
  │     値: 42    │  最後に積まれたデータ（最初に取り出される）
  ├───────────────┤
  │     値: 3.14  │
  ├───────────────┤
  │     値: 100   │  最初に積まれたデータ
  └───────────────┘
```

データを積む操作を**PUSH**、取り出す操作を**POP**と呼びます。

## 関数呼び出しのたびに何が積まれるか

プログラムが関数を呼び出すとき、スタックの上で何が起きているかを見ていきます。

```c
void greeting(char *name) {
  printf("Hello, %s\n", name);
}

int main() {
  greeting("Alice");
  return 0;
}
```

`main` が `greeting` を呼び出す処理は、次の順に進みます。

1. `greeting` の引数（`"Alice"` を指すポインタ）がスタックにPUSHされる
2. **戻り先アドレス**（`greeting` が終わったら `main` のどこに戻るかを示す値）がスタックにPUSHされる
3. `greeting` のローカル変数がスタックにPUSHされる
4. `greeting` の処理本体が実行される
5. `greeting` が終わると、積まれたデータが逆の順番でPOPされる

この「引数、戻り先アドレス、ローカル変数」をまとめた一区切りを**スタックフレーム**と呼びます。

関数が呼ばれるたびに1つのスタックフレームが積まれ、その関数の処理が終わると、対応するスタックフレームが取り除かれます。

<Callout title="戻り先アドレスが攻撃の標的になる理由">
  関数が終わったあとどこに戻るかという情報は、スタック上の値として保持されています。この値を書き換えると、関数の終了後にプログラムを攻撃者の指定した場所へ移動させられます。これがバッファオーバーフロー攻撃の中心にある仕組みです。
</Callout>

## スタックポインタと再帰の深さ

スタック上の現在の積み上げ位置を保持するレジスタ（CPU内部の小さな記憶領域）を**スタックポインタ**（SP）と呼びます。

再帰呼び出しが深くなるほど、スタックフレームが1つずつ積み重なっていきます。

```c
void deep_recursion(int n) {
  if (n == 0) return;
  deep_recursion(n - 1);
}

deep_recursion(100000);
```

この呼び出しでは、`deep_recursion` が10万回ネストして呼ばれるので、10万個のスタックフレームが積まれます。

スタック領域の容量は有限です。

Linuxのデフォルト設定では8MB、Windowsでは1MB程度に制限されています。

スタックフレームの数が増え続けてこの容量を使い切ると、それ以上の呼び出しを積む場所がなくなります。

この状態が**スタックオーバーフロー**です。

スタックオーバーフローが起きたあとのプログラムの動作は保証されておらず、多くのOSはプロセスを強制終了させます。

冒頭で見た `factorial(50000)` のクラッシュは、この仕組みで起きていたものです。

## ヒープ：必要な分だけ自分で確保するメモリ

スタックとは異なる性質を持つ領域として、**ヒープ**があります。

スタックは関数の呼び出しと終了に合わせて自動で積み上げと取り崩しが行われるので、使う側が容量を意識する必要がありません。

一方、「プログラムを書いている時点では必要な容量がわからない」という場面では、スタックの仕組みは使えません。

このような場面で使うのがヒープです。

```c
char *buffer = malloc(1024);
free(buffer);
```

`malloc()` を呼ぶと、指定したバイト数のメモリがヒープ上に確保され、そのメモリの先頭アドレスが返されます。

使い終わったら `free()` を呼んで、確保した領域を明示的に解放します。

スタックと比べると、ヒープには次の特徴があります。

**手動管理**：確保したメモリは`free()`を呼ぶまで解放されません。

**容量が大きい**：OSが許す範囲で、スタックよりずっと大きなメモリを確保できます。

**生存期間が自由**：確保した関数が終了しても、`free()`が呼ばれるまでメモリは存在し続けます。

```c
void example() {
  int *ptr = malloc(sizeof(int));
  *ptr = 42;
  return;
}
```

この関数が終了したあとも、`ptr` が指していたメモリは解放されていないので存在し続けます。

ただし `ptr` という変数自体は関数のローカル変数なので、関数の終了とともにスタックから取り除かれます。

確保したメモリへの参照を失ったまま、メモリだけが残り続ける状態になります。

これが次に説明する**メモリリーク**です。

## メモリリークと二重解放

ヒープを手動で管理することによって生じる典型的な不具合を2つ見ます。

**メモリリーク**は、確保したメモリを`free()`し忘れることで起きます。

```c
for (int i = 0; i < 1000000; i++) {
  char *buf = malloc(1024);
}
```

このループは100万回実行されるたびに1024バイトを確保しますが、`free(buf)`を呼んでいません。

ループを抜けた時点で、合計約1GBのメモリが、参照する変数を失ったまま確保され続けます。

**二重解放**は、同じメモリ領域に対して`free()`を2回呼ぶことで起きます。

```c
char *buf = malloc(1024);
free(buf);
free(buf);
```

2回目の`free(buf)`は、すでに解放済みの領域を解放しようとする呼び出しで、動作が保証されません。

ヒープの内部管理情報が壊れ、その後の`malloc()`や`free()`が予測できない動作をするようになります。

CやC++で低レイヤーの開発をする場合、これらの管理はプログラマの責任です。

PythonやJava、Rustといった言語では、自動メモリ管理の仕組みによってメモリリークや二重解放の多くが防がれます。

<Callout title="Rustの所有権という考え方">
  Rustは「所有権」という規則をコンパイラが検査することで、メモリリークと二重解放をコンパイル時に防ぎます。学習コストは高いものの、セキュリティが重視される分野で採用が広がっています。
</Callout>

## バッファオーバーフロー攻撃のしくみ

最後に、セキュリティの文脈で語られる「スタックオーバーフロー」を見ます。

これは、再帰の深さが原因で起きる前述のスタックオーバーフローとは別の事象です。

スタック上に確保したバッファに、想定より大きいデータを書き込むことで、**隣接するメモリ領域**（特に戻り先アドレス）を上書きする攻撃を指します。

```c
void vulnerable() {
  char buffer[8];
  gets(buffer);
}
```

`buffer` は8バイトしか確保されていませんが、`gets()` は入力の長さを確認せずにそのまま書き込みます。

利用者が100バイトの文字列を入力すると、`buffer` の領域を超えて、その先のメモリが上書きされます。

スタックフレームの構造上、`buffer` の先には戻り先アドレスが置かれています。

```
┌────────────────────┐
│   戻り先アドレス     │  上書きされた
├────────────────────┤
│   ローカル変数等     │
├────────────────────┤
│   buffer (8バイト)  │  ここからオーバーフローが始まる
└────────────────────┘
```

入力データの中に、攻撃者が用意したコードのアドレスを含めておけば、`vulnerable` が終了した時点で、プログラムはそのアドレスへ「戻り」、攻撃者の用意したコードを実行します。

これが**バッファオーバーフロー攻撃**です。

1990年代から2000年代にかけて、この攻撃は最も一般的な脆弱性の一つでした。

現在のOSやコンパイラはASLR（メモリ配置をランダム化する仕組み）、スタックカナリア（書き換えを検知する仕組み）、DEP（データ領域でのコード実行を禁止する仕組み）といった防御機構を備えており、攻撃の難易度は大きく上がっています。

これらの対策があっても、攻撃が利用している「スタック上の隣接領域を上書きする」という原理そのものは変わっていません。

## この章のまとめ

**スタック**は関数呼び出しに合わせて自動的に積み上げと取り崩しが行われるメモリ領域で、LIFO構造に従います。

**スタックフレーム**は、関数が呼ばれるたびに積まれる「引数、戻り先アドレス、ローカル変数」の一区切りです。

**戻り先アドレス**は関数終了後の復帰先を示す値で、これを書き換えると実行の流れを乗っ取れます。

再帰呼び出しが深くなりすぎてスタック容量を使い切ると、**スタックオーバーフロー**が起きます。

**ヒープ**は`malloc()`で動的に確保するメモリで、C/C++では`free()`による手動管理が必要です。

管理を誤ると、メモリリークや二重解放が発生します。

**バッファオーバーフロー攻撃**は、スタック上のバッファに大量のデータを書き込み、隣接する戻り先アドレスを上書きして実行の流れを乗っ取る古典的な脆弱性です。
