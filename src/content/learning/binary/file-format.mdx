## 拡張子を書き換えても中身は変わらない

「image.jpg」というファイルをテキストエディタで開くと、見慣れない記号が並んだ表示になります。

このファイルを「image.txt」にリネームして、もう一度テキストエディタで開いてみても、表示は変わりません。

依然として記号が並ぶだけです。

拡張子はファイル名の一部に過ぎず、ファイルの内容そのものとは無関係だからです。

では、ファイルの種類をコンピュータが見分けるとき、実際には何を見ているのか。

この章ではその仕組みを順に確認します。

CTFのファイル解析問題では、次のような出題が頻出します。

> このバイナリファイルを分析してください

> ファイルの中に隠されたデータを抽出してください

ファイルの種類が判別できなければ、どの道具で開けばよいかも決まりません。

`file` コマンドにせよバイナリエディタにせよ、フォレンジック（デジタル証拠を調べる手法）の道具にせよ、出発点は「このファイルは何の形式か」を特定することです。

セキュリティの実務でも、ファイルの先頭部分を見て形式を判定する作業は頻繁に行われます。

マルウェア解析、ログファイルの読み込み、ネットワークパケットの解析は、いずれもバイナリの先頭数バイトから「このフォーマットだ」と判定する手順を含んでいます。

## バイナリファイルとテキストファイルの違い

まず、両者の違いを明確にします。

**テキストファイル**は、人間が読める文字だけで構成されたファイルです。

```
Hello, World!
```

このような内容のファイルをテキストエディタで開くと、そのまま「Hello, World!」と表示されます。

ファイルの中身は文字コード（ASCIIなど、文字と数値を対応づける規則）に基づく数値の並びで、その数値が文字に対応しているために表示できます。

**バイナリファイル**は、文字コードとして解釈できるとは限らない、任意のバイト値を含むファイルです。

```
FF D8 FF E0 00 10 4A 46 49 46 ...
```

このような内容のファイルをテキストエディタで開くと、記号が並んだ表示になります。

`FF` や `D8` という値は、文字コードとしての対応を持たず、JPEG画像のための指示として定義された値だからです。

テキストエディタはこの値を文字コードとして解釈しようとして失敗し、対応する文字がない部分を記号で代用して表示します。

実際には、すべてのファイルはバイト列、つまりバイナリです。

「テキストファイル」と呼ばれるのは、そのバイト列が人間の読める文字コードの範囲だけで構成されている場合の呼び方であり、それ以外の値を含む場合を「バイナリファイル」と呼んでいるに過ぎません。

両者のあいだに、ファイル形式としての違いはありません。

## マジックバイト：ファイルの先頭に置かれた識別情報

コンピュータがファイルの種類を判定する方法を見ます。

判定の基準になるのは、**ファイルの先頭数バイト**です。

この数バイトを**マジックバイト**、または**ファイルシグネチャ**と呼びます。

代表的な形式のマジックバイトは次の通りです。

```
JPEG  → FF D8 FF
PNG   → 89 50 4E 47
PDF   → 25 50 44 46
GIF   → 47 49 46 38
ZIP   → 50 4B 03 04
ELF   → 7F 45 4C 46
```

各ファイル形式の仕様には、「先頭のバイト列はこの値にする」という取り決めがあり、その取り決めに基づいて作成されたファイルには、常に同じマジックバイトが付きます。

Linuxの`file`コマンドは、このマジックバイトを読み取ってファイルの種類を判定します。

```bash
$ file image.jpg
image.jpg: JPEG image data, JFIF standard 1.01

$ file document.pdf
document.pdf: PDF document, version 1.4

$ file data.bin
data.bin: data
```

拡張子は参照せず、ファイルの内容だけを見て判定しています。

<Callout title="拡張子で判定するOSとマジックバイトで判定するOS">
  Windowsは拡張子に基づいて「どのアプリケーションで開くか」を決めます。

  LinuxやUnix系のOSは拡張子を無視し、`file`コマンドでマジックバイトを確認します。

  拡張子を偽って詐称しても、マジックバイトを見るコマンドであれば実際の形式を判定できます。
</Callout>

## 主な形式のマジックバイト

実務で出会う頻度の高いマジックバイトを、形式ごとに見ていきます。

**JPEGファイル**

```
FF D8 FF E0 ...  (JFIF標準)
FF D8 FF E1 ...  (EXIF標準)
FF D8 FF E2 ...  (ICCプロファイル付き)
```

先頭3バイトは常に `FF D8 FF` です。

**PNGファイル**

```
89 50 4E 47 0D 0A 1A 0A
```

ASCIIとして読むと `\x89PNG\r\n\x1a\n` になります。

`\x89` は表示用の文字コードを持たない制御用の値なので、テキストエディタでは記号として表示されます。

**PDFファイル**

```
25 50 44 46
```

ASCIIとして読むと `%PDF` になります。

PDFはテキストとして読める部分を多く含むフォーマットなので、先頭が印字可能な文字になっています。

**ZIPファイル**（および同じ内部形式を使うJAR、DOCX、XLSXなど）

```
50 4B 03 04
```

ASCIIとして読むと `PK\x03\x04` になります。

「PK」は、ZIP形式を考案したPhil Katzのイニシャルです。

DOCXやXLSXのようなOffice文書は、内部構造がZIP圧縮されているため、同じシグネチャを持ちます。

**ELFファイル**（Linuxの実行可能ファイル形式）

```
7F 45 4C 46
```

ASCIIとして読むと `\x7FELF` になります。

`\x7F` はDEL（削除）を意味する制御用の値です。

<Callout title="複数のシグネチャを持つファイル">
  JPEGファイルの末尾にZIPのデータを追加すると、画像として開くツールは先頭のJPEGシグネチャに反応して画像を表示し、アーカイブとして開くツールは末尾のZIPシグネチャに反応してアーカイブとして展開します。

  どちらの形式でも開けるファイルが作れることになります。
</Callout>

## hexdumpとxxd：バイト列を直接見る

ファイルの内容を直接確認する方法を見ます。

`hexdump` または `xxd` というコマンドを使うと、ファイルの内容を16進数で表示できます。

```bash
$ xxd image.jpg | head -3
00000000: ffd8 ffe0 0010 4a46 4946 0001 0100 0001  ......JFIF........
00000010: 0001 0000 ffdb 0043 0003 0202 0202 0202  .......C........
00000020: 0303 0302 0304 0605 0405 0609 080a 0906  ................
```

各行の左側はアドレス（ファイル先頭からのバイト数を示すオフセット）、中央は16進数表記の内容、右側はASCII表現（印字可能な文字はそのまま、それ以外は`.`で表示)です。

先頭に見える `ffd8` と `ffe0` が、先ほど確認したJPEGのマジックバイトです。

```bash
$ xxd document.pdf | head -1
00000000: 2550 4446 2d31 2e34 0a25 e2e3 cfd3 0a31  %PDF-1.4.%......1
```

先頭の `2550 4446` が `%PDF` というPDFのシグネチャに対応しています。

## fileコマンドの内部にある仕組み

`file` コマンドがどのようにファイルの種類を判定しているかを確認します。

多くのLinux環境では、`/etc/magic` や `/usr/share/misc/magic` という設定ファイルに、マジックバイトとそれに対応するファイル形式の一覧が定義されています。

```
0       string          %PDF            PDF document
0       beshort         0xffd8          JPEG image data
0       string          \x89PNG         PNG image
0       string          PK\003\004      Zip archive data
0       string          \x7FELF         ELF
```

各行はおおまかに「ファイル先頭からのオフセット」「値の種類」「期待する値」「該当時の表示文字列」という構成になっています。

`file` コマンドは、与えられたファイルの先頭バイトをこの一覧と順に照合し、最初にマッチした行に書かれた形式を結果として返します。

`-i` オプションを付けると、MIMEタイプ（インターネット上で広く使われるファイル種別の表記形式）の形式で結果を返します。

```bash
$ file -i image.jpg
image.jpg: image/jpeg; charset=binary

$ file -i data.zip
data.zip: application/zip; charset=binary
```

## ステガノグラフィへの入口

ここまでの内容を踏まえると、**ステガノグラフィ**という技術の仕組みが理解できます。

ステガノグラフィとは、あるデータを別のデータの中に隠す技術です。

暗号化がデータの内容を読めなくする技術であるのに対し、ステガノグラフィはデータが隠されているという事実そのものを隠します。

次の手順を考えます。

```
1. JPEGファイル（画像）を用意する
2. ZIPファイル（圧縮したテキストデータ）をJPEGファイルの末尾に追加する
3. fileコマンドで判定すると「JPEG」と表示される（先頭のマジックバイトしか見ないため）
4. 末尾にZIPシグネチャがあることを知っている人がZIPツールで開くと、隠したデータが取り出せる
```

通常の画像ビューアでこのファイルを開くと、画像がそのまま表示され、不審な点は見当たりません。

しかし同じファイルをZIPツールで開くと、アーカイブとして展開できます。

JPEGとZIPの両方の形式が、先頭または末尾だけを見て判定する設計になっているために、この二重性が成立します。

CTFのステガノグラフィ問題では、「見た目は画像だが、実際には複数のフォーマットが重なっている」という出題がよくあります。

マジックバイトの仕組みを理解していれば、「ファイルの末尾も確認してみる」という発想が自然に出てきます。

<Callout title="strings コマンドで文字列を抜き出す">
  バイナリファイルから、印字可能な文字列だけを抜き出す`strings`コマンドもあります。

  マルウェアの解析や、バイナリに埋め込まれた手がかりとなる文字列を探す場面で使われます。
</Callout>

## この章のまとめ

**バイナリファイル**と**テキストファイル**の違いは、含まれるバイト値が文字コードの範囲に収まっているかどうかの違いに過ぎません。

すべてのファイルは、中身はバイナリ（バイト列）です。

**マジックバイト**（ファイルシグネチャ）は、ファイルの先頭数バイトに置かれた、形式を示す識別情報です。

代表的なシグネチャには、JPEGの`FF D8 FF`、PNGの`89 50 4E 47`、PDFの`25 50 44 46`、ZIPの`50 4B 03 04`、ELFの`7F 45 4C 46`があります。

拡張子はファイル名の一部であり、内容の判定には使えません。

Linuxの`file`コマンドは拡張子を無視し、マジックバイトを照合してファイル種別を判定します。

`hexdump`や`xxd`を使うと、ファイルの内容を16進数とASCII表現で直接確認できます。

**ステガノグラフィ**は、複数のフォーマットの判定が先頭や末尾の一部しか見ないという特性を利用して、別の形式のデータを1つのファイルに重ねて隠す技術です。
