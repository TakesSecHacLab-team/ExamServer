## 拡張子は見かけに過ぎない

あなたのパソコンに「image.jpg」というファイルがあります。これをテキストエディタで開くと、文字化けした不可思議な記号が並びます。

で、ためしに「image.jpg」を「image.txt」にリネームして、もう一度テキストエディタで開くと——やはり文字化けしています。

「えっ、拡張子を変えてもダメ？」——そうです。拡張子はただの名前。ファイルの中身とは無関係です。では、ファイルの「正体」をコンピュータはどうやって判定しているのか。それが、このセクションの全てです。

## なぜこれを学ぶのか

CTFのファイル問題では必ず出ます。

> このバイナリファイルを分析してください

> ファイルの中に隠されたデータを抽出してください

「これは何のファイル？」が分からないと、どの道具で開いたらいいか分かりません。`file` コマンドも、バイナリエディタも、フォレンジック道具も、全ては「このファイルは何か」を知ることから始まります。

また、セキュリティエンジニアは頻繁に「ファイルのヘッダーを見る」という作業をします。マルウェア解析、ログの読み込み、ネットワークパケット解析——どれもバイナリの先頭を見て「あ、このフォーマットだな」と判定します。

ここを理解していると、一気に解けるようになる。地味だけど、超強い武器になります。

## バイナリファイル vs テキストファイル

まず、区別を明確にしましょう。

**テキストファイル**：人間が読める文字だけで構成
```
Hello, World!
```

をテキストエディタで開くと、「Hello, World!」と表示されます。ファイルの中身は文字コード（ASCIIなど）で、その値が文字に対応しているから。

**バイナリファイル**：任意のバイト値を含む
```
FF D8 FF E0 00 10 4A 46 49 46 ...
```

をテキストエディタで開くと、文字化けします。なぜなら、`FF` や `D8` という値は「テキスト用の文字コード」ではなく、「JPEG画像としての指示」だから。テキストエディタは「これは文字か？」と判定して失敗し、化け文字で表示する。

正確には、**すべてのファイルはバイナリです**。ただ「テキストファイル」は、そのバイト値が人間が読める文字（A-Z, 0-9など）だけで構成されているケース。バイナリファイルは、そうではない任意の値を含むケース。両者に本質的な区別はありません。

## マジックバイト：ファイルの正体を示す署名

では、コンピュータはどうやってファイルの種類を判定するのか。

答え：**ファイルの先頭数バイト**（**マジックバイト** または **ファイルシグネチャ**）を見る。

例えば：

```
JPEG     → FF D8 FF
PNG      → 89 50 4E 47
PDF      → 25 50 44 46
GIF      → 47 49 46 38
ZIP      → 50 4B 03 04
ELF      → 7F 45 4C 46
```

各ファイル形式は「最初のバイト列はこれね」という取り決めがあります。ファイルフォーマットの仕様書に書いてあります。

Linux の `file` コマンドは、このマジックバイトを見てファイル種別を判定します。

```bash
$ file image.jpg
image.jpg: JPEG image data, JFIF standard 1.01

$ file document.pdf
document.pdf: PDF document, version 1.4

$ file data.bin
data.bin: data
```

拡張子は無視。中身を見てます。

<Callout title="拡張子 vs マジックバイト">
  Windows は拡張子に基づいて「どのアプリで開くか」を決めます。Linux/Unix は拡張子を無視し、`file` コマンドでマジックバイトを見ます。セキュリティの観点では Linux のアプローチが安全。拡張子を詐称しても、`file` なら見破ります。
</Callout>

## 代表的なマジックバイト

実務でよく見かけるマジックバイトを覚えておくと、便利です。

**JPEGファイル**
```
FF D8 FF E0 ... (JFIF標準)
FF D8 FF E1 ... (EXIF標準)
FF D8 FF E2 ... (ICC プロファイル付き)
```
必ず `FF D8 FF` で始まります。

**PNGファイル**
```
89 50 4E 47 0D 0A 1A 0A
```
ASCII で読むと `\x89PNG\r\n\x1a\n`。`\x89` は制御文字なので、テキストエディタでは化ける。

**PDFファイル**
```
25 50 44 46
```
ASCII で読むと `%PDF`。PDF はテキストベースのフォーマットなので、最初が印字可能文字。

**ZIPファイル**（とそのバリエーション：JAR, DOCX, XLSX など）
```
50 4B 03 04
```
ASCII で読むと `PK\x03\x04`。「PK」は ZIP の創作者 Phil Katz のイニシャル。DOCX や XLSX は内部が ZIP 圧縮なので、同じシグネチャを持ちます。

**ELFファイル**（Linux の実行可能ファイル）
```
7F 45 4C 46
```
ASCII で読むと `\x7FELF`。`\x7F` は DEL 制御文字。

<Callout title="容易な偽装の方法">
  JPEGファイルにZIPシグネチャを付けると（両方の先頭に必要なバイト列を足す）、JPEGビューアは "これは画像" と認識し、ZIP ツールは "これはアーカイブ" と認識します。どちらでも開けます。ステガノグラフィの一種。
</Callout>

## hexdump と xxd：バイナリの目で見る

では、実際にファイルの中身を見るにはどうするか。

`hexdump` または `xxd` コマンドを使います。

```bash
$ xxd image.jpg | head -10
00000000: ffd8 ffe0 0010 4a46 4946 0001 0100 0001  ......JFIF........
00000010: 0001 0000 ffdb 0043 0003 0202 0202 0202  .......C........
00000020: 0303 0302 0304 0605 0405 0609 080a 0906  ................
...
```

左側は **アドレス**（ファイルの先頭からのバイトオフセット）。中央は **16進数**。右側は **ASCII表現**（印字可能なら文字、でなければ`.`）。

最初の `ffd8` と `ffe0` が見えます。これが JPEG のシグネチャ。一目瞭然。



```bash
$ xxd document.pdf | head -5
00000000: 2550 4446 2d31 2e34 0a25 e2e3 cfd3 0a31  %PDF-1.4.%......1
```

最初の `2550 4446` は `%PDF`——PDF のシグネチャ。

## file コマンドの仕組み

では、`file` コマンドはどう動いているのか。

`/etc/magic` または `/usr/share/misc/magic` という設定ファイルに、マジックバイトと対応するファイル型が列挙されています。

```
0       string          %PDF            PDF document
0       beshort         0xffd8          JPEG image data
0       string          \x89PNG         PNG image
0       string          PK\003\004      Zip archive data
0       string          \x7FELF         ELF
```

`file` コマンドは、与えられたファイルを開き、先頭のバイトをこの辞書と照合し、最初にマッチしたエントリを返します。

-i（MIME type）オプションで MIME タイプ形式で返すこともできます。

```bash
$ file -i image.jpg
image.jpg: image/jpeg; charset=binary

$ file -i data.zip
data.zip: application/zip; charset=binary
```

## ステガノグラフィへの足がかり

ここで重要なセキュリティ概念：**ステガノグラフィ** を軽く紹介します。

ステガノグラフィとは、「データを他のデータの中に隠す」技術です。暗号化と異なり、隠されたデータの存在そのものが秘密。

例：

```
1. JPEGファイル（画像）を用意
2. ZIPファイル（テキストデータ圧縮）をJPEGの末尾に追加
3. file コマンド？ → "JPEG" と判定（最初のマジックバイトを見るから）
4. 特別な知識がある人？ → 末尾のZIPシグネチャに気づき、ZIPツールで抽出
```

JPEGビューアは画像を表示し、一見何ともありません。でも ZIP として開くこともできる——双方をサポートするフォーマットだから可能。

CTF の「ステガノ問題」では、「見た目は画像だが、実は複数のフォーマットが重なっている」というケースがよくあります。ここを理解していると、「あ、末尾を見たら何かあるかな」という勘が働くようになります。

<Callout title="strings コマンドも便利">
  バイナリファイルから「印字可能な文字列」だけを抽出する `strings` コマンドもあります。マルウェアやバイナリに埋め込まれた文字列情報を探すとき活躍します。
</Callout>

## ここで押さえること

- **バイナリファイル**と**テキストファイル**の区別は相対的。すべてのファイルはバイナリ
- **マジックバイト**（ファイルシグネチャ）：ファイルの先頭数バイト。ファイル種別を示す
- 代表的なシグネチャ：JPEG = `FF D8 FF`、PNG = `89 50 4E 47`、PDF = `25 50 44 46`、ZIP = `50 4B 03 04`、ELF = `7F 45 4C 46`
- **拡張子は無意味**。Linux の `file` コマンドは拡張子を無視し、マジックバイトを見て判定
- **hexdump / xxd** でバイナリを16進数で表示。アドレスと内容が読める
- **ステガノグラフィ**：画像の中に別のデータを隠す。複数のマジックバイトを持つハイブリッドファイルが実現
- **strings** コマンド：バイナリから文字列情報を抽出
