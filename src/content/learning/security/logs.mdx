## ログの1行から何が読み取れるか

SSH サーバーのログに、こんな1行が出ていました。

```
Failed password for invalid user admin from 192.168.1.100 port 54321 ssh2
```

これは何を意味するか。そこから何が読み取れるか。

——これを解きほぐせることが、セキュリティ検知の出発点です。

**注：このレッスンは「ログの内容を読み解く」にフォーカスします。** 「ログファイルがどこにあるか」「tail や grep でどう見るか」は、Linux基礎の別章（`/learn/linux/logs`）で扱っています。ここは「データが目の前にあるとき、何を読むか」という解釈スキルです。

## ログとは何か——監査証跡

ログとは、システムが過去に何をしたかの記録です。より正確には：

> イベント（ユーザーのログイン、ファイルアクセス、エラー、リクエスト等）を、タイムスタンプと共に記録したもの

ログなしに、セキュリティ検知は不可能です。ログがあれば、侵害が起きた後、「何がいつ起きたか」を追跡できます。

ただし：

- ログがあっても「見なければ」気づけない
- ログが残されないように攻撃者は隠蔽を試みる
- ログが膨大すぎると、重要な情報が埋もれる

だから、セキュリティの現場では「ログの記録」と「ログの監視」の両方が大事です。

## 認証失敗ログから brute force を読み解く

さっきのログを分析してみましょう：

```
Failed password for invalid user admin from 192.168.1.100 port 54321 ssh2
```

1段の情報：

| データ | 意味 |
|---|---|
| `Failed password` | パスワード認証が失敗 |
| `invalid user admin` | ユーザー名 `admin` は存在しない（ユーザーが存在しない場合の表示） |
| `from 192.168.1.100` | 攻撃元の IP アドレス |
| `port 54321` | 攻撃元が使用したポート |
| `ssh2` | SSH プロトコルバージョン2 |

1行だけなら「どこかから不正ログイン試行があった」だけです。

でも、ログをもっと見てみましょう：

```
Failed password for invalid user admin from 192.168.1.100 port 54321 ssh2
Failed password for invalid user root from 192.168.1.100 port 54322 ssh2
Failed password for invalid user test from 192.168.1.100 port 54323 ssh2
Failed password for invalid user oracle from 192.168.1.100 port 54324 ssh2
... [同じIP から数百行続く]
Failed password for invalid user user from 192.168.1.100 port 54500 ssh2
```

これは明らかに **brute force 攻撃**——同じIPから短時間に大量のユーザー名を試している。典型的な「パスワード推測攻撃」の痕跡です。

さらに、複数のIPからも同じパターンが来ていたら？それは **ボットネット** による分散攻撃かもしれません。

## アクセスログの異常パターン

Webサーバーのアクセスログを見てみます。通常は：

```
192.168.1.50 - - [20/Jun/2026 10:15:32 +0900] "GET /index.html HTTP/1.1" 200 1234
192.168.1.50 - - [20/Jun/2026 10:15:33 +0900] "GET /style.css HTTP/1.1" 200 5678
```

ユーザーがブラウザで閲覧している、普通のアクセス。ステータスコード `200`（成功）が返っています。

では、攻撃者のアクセスはこう見えるかもしれません：

**1. ポート・ディレクトリスキャン**
```
192.168.100.1 - - [20/Jun/2026 14:00:01 +0900] "GET /admin HTTP/1.0" 404 -
192.168.100.1 - - [20/Jun/2026 14:00:02 +0900] "GET /administrator HTTP/1.0" 404 -
192.168.100.1 - - [20/Jun/2026 14:00:03 +0900] "GET /wp-admin HTTP/1.0" 404 -
192.168.100.1 - - [20/Jun/2026 14:00:04 +0900] "GET /admin.php HTTP/1.0" 404 -
```

短時間に異なるディレクトリへのアクセス、全て `404`（見つかりません）。これは「隠れた管理画面がないか探している」スキャン活動です。

**2. SQLインジェクション試行**
```
192.168.100.2 - - [20/Jun/2026 14:30:01 +0900] "GET /search.php?q=' OR '1'='1 HTTP/1.1" 200 9999
192.168.100.2 - - [20/Jun/2026 14:30:02 +0900] "GET /search.php?q='; DROP TABLE users; -- HTTP/1.1" 500 -
```

URL に SQL構文が含まれている。`500`（内部サーバーエラー）が返っているのは、攻撃が一部成功したかもしれません。

**3. 異常な User-Agent**
```
192.168.100.3 - - [20/Jun/2026 15:00:01 +0900] "GET / HTTP/1.1" 200 - "sqlmap/1.5.0"
192.168.100.3 - - [20/Jun/2026 15:00:02 +0900] "GET /api/users HTTP/1.1" 200 - "nikto/2.1.5"
```

User-Agent が自動スキャンツール（sqlmap、nikto など）。通常のブラウザではありません。

**4. エラーレートの急増**
```
[集計] 2026-06-20 14:00: 状態別アクセス
  200: 1200件（成功）
  404: 50件（見つかりません）
  500: 5件（エラー）

[集計] 2026-06-20 15:00: 状態別アクセス
  200: 1250件
  404: 850件 ← 急増！
  500: 120件 ← 急増！
```

通常の数十倍のエラーが出ている。何か異常が起きています。

## ログの解釈の注意点——タイムゾーン

ここ、うっかり引っかかる人が多いです。

Webサーバーやデータベース、ログ管理システムは、往々にして **UTC（協定世界時）** でログを記録します。一方、管理者は**JST（日本標準時、UTC+9）** で考えています。

```
[2026-06-20T14:00:00Z] User login from 192.168.1.50
```

この `Z` は UTC を意味します。つまり、日本時間では「2026-06-20 の 23:00:00」です。

ログを見るたびにタイムゾーン変換を忘れると、「このアクセスはいつ起きた」という因果関係の把握が狂います。

複数システムのログを相互参照するときは、**全て同じタイムゾーンに統一** することが重要です。

<Callout title="複数システムのログを比較するには">
- サーバーAのログ：UTC
- データベースBのログ：JST
- ファイアウォールCのログ：UTC

こんな混在状態では、「どのイベントが連鎖しているか」の因果関係が掴めません。ログ管理システム（SIEM）を使う際は、タイムゾーン統一が前提です。
</Callout>

## ログ相関——複数ログを組み合わせて攻撃の流れを追う

1つのシステムのログだけ見ていても、全体の攻撃像は見えません。

複数のログを組み合わせることで、初めて「侵害の全体像」が浮かぶ。これを **ログ相関**（log correlation）と呼びます。

**例：多段階の攻撃**

1. **ファイアウォールログ**：
   ```
   2026-06-20 10:00:00 Connection attempt from 203.0.113.50 to internal_db_server:3306 BLOCKED
   ```
   外部からデータベースへの直接アクセス試行が止められた。

2. **Webサーバーログ**：
   ```
   2026-06-20 10:05:00 GET /api/users?id=1' OR '1'='1 from 203.0.113.50
   ```
   ファイアウォールを迂回して、Webアプリの SQLインジェクション脆弱性を狙っている。

3. **データベースログ**：
   ```
   2026-06-20 10:10:00 SELECT * FROM users; [user: webapp]
   ```
   SQLインジェクションが成功し、ユーザーテーブル全体が読み取られた。

4. **ネットワークフロー（NetFlow）**：
   ```
   2026-06-20 10:15:00 Internal_db_server -> 203.0.113.50 : 500MB transfer
   ```
   データベースサーバーから外部へ大量のデータが流出している。

1つ1つのログは「何か起きた」という情報。でも組み合わせると「Webの脆弱性を突いて侵入 → データベースから盗み出した → 外部へ流出」という攻撃の全体像が見えます。



## SIEM（Security Information and Event Management）

SIEM は、複数のログソース（サーバー、ファイアウォール、DNS、認証システムなど）を一箇所に集約し、自動的に異常を検知するシステムです。

人間が数百万行のログを眺めることは現実的ではありません。SIEM は：

- 複数のログを統一フォーマットに変換
- あらかじめ定義されたルールで異常を検知
- 関連するイベントを自動で相関付け
- アラートで管理者に通知

有名な SIEM ツール：

- **Splunk**：エンタープライズ向けスタンダード
- **ELK Stack**（Elasticsearch, Logstash, Kibana）：オープンソース、スケーラブル
- **Datadog**：クラウドネイティブ向け
- **Azure Sentinel**：Microsoft クラウド環境向け

## MTTD と MTTR——検知と復旧のスピードが生死を分ける

セキュリティのKPI（重要業績評価指標）として、以下の2つがよく使われます：

**MTTD（Mean Time To Detect）**
- 侵害がどのくらい後に検知されるか
- 短いほど良い（理想は数時間以内）
- Verizon DBIR 2025 によると、平均は約 5 日間

**MTTR（Mean Time To Recover）**
- 侵害から、システムが正常に戻るまでどのくらいかかるか
- 短いほど良い
- バックアップからの復旧速度、対応組織の体制に依存

**被害の大きさ＝ (侵害から検知までの時間) × (検知から復旧までの時間)**

MTTD が長いと、攻撃者は長時間ネットワーク内を探索し、データを盗み放題。MTTR が長いと、システムが長時間ダウンし、事業継続に支障。

だから、組織は以下に投資します：

- 検知精度を上げる（SIEM、自動化ルール）
- インシデント対応の体制を整える（24/7 の監視、決定プロセスの高速化）
- バックアップの信頼性を上げる（定期テスト、複数拠点保持）

## アラート設定と false positive の罠

SIEM でルールを設定する際の難題は、**false positive（偽陽性）** です。

「侵害の可能性がある」と判定したが、実は正常な活動だった。

```
[Alert] 深夜2時にシステム管理者がRDPでログイン
→ 実はシステム障害対応で、管理者が緊急ログイン
```

false positive が多すぎると、アラートが信頼されなくなり、本当の脅威も見落とされます（アラート疲れ）。

アラートの設定は、精度のバランスを取る技術です：

- しきい値を厳しく設定：見落としが減るが、誤報が増える
- しきい値を甘く設定：誤報が減るが、本当の脅威を見逃す

組織は、この「感度」を、継続的にチューニングします。

## ここで押さえること

- **ログとは監査証跡**。「見なければ」被害に気づけない。MTTD（検知時間）が長いほど被害は大きい
- 認証失敗ログの大量出現は **brute force 攻撃** の痕跡
- Webアクセスログの **404 多発・異常な User-Agent・エラー率急増** は、スキャンや攻撃の兆候
- **ログ相関**：複数のログを組み合わせることで、攻撃の全体像が見える
- **タイムゾーン統一** は必須。UTC と JST が混在すると因果関係が狂う
- **SIEM** は複数ログを自動監視して異常を検知するが、false positive（誤報）とのバランスが大事
