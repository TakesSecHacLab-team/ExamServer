## ログの1行から何が分かるか

SSHサーバーのログに、こんな1行が記録されていました。

```
Failed password for invalid user admin from 192.168.1.100 port 54321 ssh2
```

このレッスンが扱うのは、この1行から何を読み取れるか、という解釈の力です。

ログファイルがどこに置かれているか、`tail` や `grep` でどう開くかといった操作は、Linux基礎の別の章（`/learn/linux/logs`）で扱っています。

ここでは「データが目の前にあるとき、そこから何を読み取るか」だけに絞ります。

## ログとは何か

**ログ**とは、システムが過去に何をしたかを、タイムスタンプ付きで記録したものです。

ユーザーのログイン、ファイルへのアクセス、エラーの発生、リクエストの受信といった出来事が、発生した時刻とともに残されます。

ログがなければ、侵害がいつ何が起きたのかを後から追うことはできません。

ただし、ログが残っているだけでは不十分です。

記録されていても、誰かが見なければ気づけません。

攻撃者の側も、自分の行動の記録を消そうとします。

ログの量が増えすぎると、重要な情報がその中に埋もれてしまいます。

だから、セキュリティの現場では「ログを記録すること」と「ログを監視すること」の両方が必要になります。

## 認証失敗ログからbrute force攻撃を読む

最初に見たログを、項目ごとに分解してみます。

```
Failed password for invalid user admin from 192.168.1.100 port 54321 ssh2
```

  <thead>
    <tr>
      <th>記録</th>
      <th>意味</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td><code>Failed password</code></td>

    </tr>

      <td><code>admin</code> というユーザー名がそもそも存在しない</td>
    </tr>
    <tr>

      <td>接続を試みた側のIPアドレス</td>
    </tr>
    <tr>
      <td><code>port 54321</code></td>
      <td>接続を試みた側が使用したポート番号</td>
    </tr>
    <tr>
      <td><code>ssh2</code></td>
      <td>SSHプロトコルのバージョン2を使用した接続</td>
    </tr>
  </tbody>
</table>

この1行だけでは「どこかから不正なログイン試行があった」ことしか分かりません。

同じサーバーのログを、もう少し広く見てみます。

```
Failed password for invalid user admin from 192.168.1.100 port 54321 ssh2
Failed password for invalid user root from 192.168.1.100 port 54322 ssh2
Failed password for invalid user test from 192.168.1.100 port 54323 ssh2
Failed password for invalid user oracle from 192.168.1.100 port 54324 ssh2
... [同じIPから数百行続く]
Failed password for invalid user user from 192.168.1.100 port 54500 ssh2
```

同じIPアドレスから、短時間に大量の異なるユーザー名で接続が試みられています。

これは**brute force攻撃**（総当たり攻撃）の痕跡です。

ありそうなユーザー名とパスワードの組み合わせを片っ端から試し、当たれば侵入する手口です。

複数のIPアドレスから同時に同じパターンが届いていれば、一台の攻撃者ではなく、乗っ取られた多数の機器が一斉に動く**ボットネット**による分散攻撃の可能性があります。

## アクセスログの異常パターンを読む

次に、ウェブサーバーのアクセスログを見ます。

通常のアクセスはこう記録されます。

```
192.168.1.50 - - [20/Jun/2026 10:15:32 +0900] "GET /index.html HTTP/1.1" 200 1234
192.168.1.50 - - [20/Jun/2026 10:15:33 +0900] "GET /style.css HTTP/1.1" 200 5678
```

ブラウザでページを開いた利用者のアクセスで、ステータスコード `200`（成功）が返っています。

攻撃者によるアクセスは、いくつかの典型的なパターンで見分けられます。

1つ目は、ディレクトリスキャンです。

```
192.168.100.1 - - [20/Jun/2026 14:00:01 +0900] "GET /admin HTTP/1.0" 404 -
192.168.100.1 - - [20/Jun/2026 14:00:02 +0900] "GET /administrator HTTP/1.0" 404 -
192.168.100.1 - - [20/Jun/2026 14:00:03 +0900] "GET /wp-admin HTTP/1.0" 404 -
192.168.100.1 - - [20/Jun/2026 14:00:04 +0900] "GET /admin.php HTTP/1.0" 404 -
```

短時間に異なるパスへのアクセスが続き、いずれも `404`（ページが見つからない）が返っています。

これは、隠れた管理画面がないかを順番に試す、スキャン活動の痕跡です。

2つ目は、SQLインジェクションの試行です。

```
192.168.100.2 - - [20/Jun/2026 14:30:01 +0900] "GET /search.php?q=' OR '1'='1 HTTP/1.1" 200 9999
192.168.100.2 - - [20/Jun/2026 14:30:02 +0900] "GET /search.php?q='; DROP TABLE users; -- HTTP/1.1" 500 -
```

URLの中にSQL構文がそのまま含まれています。

2行目で `500`（サーバー内部エラー）が返っているのは、入力した文字列がデータベースへの問い合わせとして実際に処理された可能性を示します。

3つ目は、自動スキャンツールによるアクセスです。

```
192.168.100.3 - - [20/Jun/2026 15:00:01 +0900] "GET / HTTP/1.1" 200 - "sqlmap/1.5.0"
192.168.100.3 - - [20/Jun/2026 15:00:02 +0900] "GET /api/users HTTP/1.1" 200 - "nikto/2.1.5"
```

末尾の `User-Agent` フィールドが `sqlmap` や `nikto` のような、脆弱性スキャン専用ツールの名前になっています。

通常のブラウザがこの値を送ることはありません。

4つ目は、エラー率の急増です。

```
[集計] 2026-06-20 14:00: 状態別アクセス
  200: 1200件（成功）
  404: 50件（見つかりません）
  500: 5件（エラー）

[集計] 2026-06-20 15:00: 状態別アクセス
  200: 1250件
  404: 850件
  500: 120件
```

1時間の間に、`404` と `500` の件数が通常の十倍以上に増えています。

正常な利用パターンからの急な変化は、何らかの異常が起きているサインです。

## タイムゾーンを揃えて読む

ログの解釈で見落としやすいのが、タイムゾーンです。

ウェブサーバーやデータベース、ログ管理システムの多くは、**UTC**（協定世界時）でログを記録します。

一方、日本国内で運用にあたる担当者は、**JST**（日本標準時、UTC+9）で時刻を考えています。

```
[2026-06-20T14:00:00Z] User login from 192.168.1.50
```

末尾の `Z` はUTCを意味する記号です。

この時刻は日本時間に直すと、6月20日の23時00分00秒になります。

ログを見るたびにこの変換を忘れると、「どのアクセスがどの順番で起きたか」という、最も基本の因果関係が狂います。

複数のシステムのログを照らし合わせるときは、全てを同じタイムゾーンに統一してから比較します。

<Callout title="複数システムのログを比較するとき">
サーバーAのログはUTC、データベースBのログはJST、ファイアウォールCのログはUTCで記録されている、という状況は珍しくありません。

タイムゾーンが混在したままでは、どのイベントがどのイベントの後に起きたのかを正しく追えません。

ログ管理システム（SIEM）を導入する際は、まずタイムゾーンを統一する設定から始めます。
</Callout>

## 複数のログを組み合わせて攻撃の全体像を見る

1つのシステムのログだけでは、攻撃の全体像は見えません。

複数のログを組み合わせて初めて、侵害がどう進んだかが浮かび上がります。

この作業を**ログ相関**（log correlation）と呼びます。

次の4つのログは、同じ攻撃の異なる段階を記録しています。

ファイアウォールのログには、外部からデータベースへの直接アクセスが拒否された記録が残っています。

```
2026-06-20 10:00:00 Connection attempt from 203.0.113.50 to internal_db_server:3306 BLOCKED
```

ウェブサーバーのログには、5分後に同じIPアドレスからSQLインジェクションの試行が記録されています。

```
2026-06-20 10:05:00 GET /api/users?id=1' OR '1'='1 from 203.0.113.50
```

データベースへの直接アクセスを防がれた攻撃者は、ウェブアプリケーションの脆弱性を経由する迂回路を試しています。

データベースのログには、さらに5分後、ユーザーテーブル全体への問い合わせが記録されています。

```
2026-06-20 10:10:00 SELECT * FROM users; [user: webapp]
```

SQLインジェクションが成功し、ユーザーテーブルの内容が読み取られたことを示しています。

ネットワークの通信量を記録したログ（NetFlow）には、さらに5分後、データベースサーバーから外部への大量のデータ転送が記録されています。

```
2026-06-20 10:15:00 Internal_db_server -> 203.0.113.50 : 500MB transfer
```

この4つを別々に見れば、それぞれ「何かが起きた」という断片の情報にしかなりません。

時系列に並べて組み合わせると、「ウェブの脆弱性を突いて侵入し、データベースから情報を盗み出し、外部へ転送した」という一連の流れが見えてきます。

## SIEMによる自動監視

**SIEM**（Security Information and Event Management）は、サーバーやファイアウォール、DNS、認証システムなど複数のログを一箇所に集め、異常を自動で検知するシステムです。

人がログを数百万行読み続けることは現実的ではありません。

SIEMは、形式の異なる複数のログを統一したフォーマットに変換し、あらかじめ定義したルールに沿って異常を検知し、関連するイベントを自動で結びつけ、検知した内容を管理者に通知します。

代表的なSIEMには、エンタープライズ向けの **Splunk**、オープンソースで構成する **ELKスタック**（Elasticsearch、Logstash、Kibana）、クラウドネイティブな **Datadog**、Microsoftのクラウド環境に統合された **Azure Sentinel** があります。

## 検知と復旧の速さが被害を決める

セキュリティの状態を測る指標として、検知と復旧の速さがよく使われます。

**MTTD**（Mean Time To Detect、平均検知時間）は、侵害が起きてから検知されるまでの時間です。

短いほどよく、Verizonのデータ侵害調査報告書（DBIR）2025年版によると、平均は約5日とされています。

**MTTR**（Mean Time To Recover、平均復旧時間）は、侵害からシステムが正常な状態に戻るまでの時間です。

バックアップからの復旧の速さや、対応にあたる組織の体制によって変わります。

MTTDが長いほど、攻撃者は長時間ネットワーク内に留まり、データを探索し続けられます。

MTTRが長いほど、システムが止まっている時間が延び、事業の継続に支障が出ます。

この2つを短くするために、組織は検知精度を上げる仕組み(SIEM や自動化されたルール)、24時間体制の監視や対応プロセスの整備、定期的にテストされたバックアップへの投資を進めます。

## アラートのしきい値と誤報の扱い

SIEMでルールを設定するときに直面する難題が、**false positive**（偽陽性、誤報）です。

「侵害の可能性がある」と判定されたものが、実際には正常な業務だった、というケースです。

```
[Alert] 深夜2時にシステム管理者がRDPでログイン
→ 実際はシステム障害への対応で、管理者が緊急にログインしていた
```

誤報が増えすぎると、アラートそのものが信頼されなくなり、本当の脅威が出ても見過ごされやすくなります。

これを**アラート疲れ**と呼びます。

アラートのしきい値を厳しく設定すれば見落としは減りますが、誤報も増えます。

しきい値を緩めれば誤報は減りますが、本当の脅威を見逃しやすくなります。

どちらに寄せても別の問題が生まれるため、組織はこの感度を継続的に調整します。

## ここで押さえること

ログは監査証跡であり、見なければ被害には気づけません。

認証失敗ログが同じIPアドレスから大量に出現していれば、brute force攻撃の痕跡です。

ウェブアクセスログの404の多発、自動スキャンツールのUser-Agent、エラー率の急増は、いずれもスキャンや攻撃の兆候です。

複数のログを組み合わせるログ相関によって、単独のログでは見えない攻撃の全体像が浮かび上がります。

タイムゾーンの統一は、複数システムのログを照らし合わせる前提条件です。

SIEMは複数のログを自動で監視しますが、誤報とのバランスを取り続ける必要があります。
