## 攻撃者の目でシステムを見てみる

城を守るとき、城主がまずすることは何か。自分の城の「どこから敵が攻撃してくるか」を眺めることです。城壁の弱い部分。門。水路。抜け道。——地図がなければ何を守ればいいか分かりません。

セキュリティも同じです。攻撃者視点で「自分のシステムには、どんな入口がある」を整理するスキルが、防御の第一歩になります。

その入口の総体を **アタックサーフェス**（attack surface）と呼びます。

## アタックサーフェスとは

システムの「攻撃可能な表面積」のこと。より正確には：

> システムの外部から、または内部から、攻撃者が侵入・悪用できる可能性のある全ての接点

これには「サービスが提供している機能」だけでなく、「設計・実装の欠陥」「設定の不備」「依存する外部コンポーネント」も含まれます。

大規模なシステムほどアタックサーフェスは広いです。機能が多い、チームが大きい、外部サービスとの連携が多い——全て、攻撃者の選択肢を増やします。

## 外部アタックサーフェス

インターネットに公開されている部分から、攻撃者は観察を始めます。

**Webアプリ・API**
- ブラウザでアクセスできるページ、API エンドポイント
- デフォルト画面に存在する情報漏洩（バージョン番号、エラーメッセージなど）

**DNS レコード**
- サブドメイン一覧から、隠れたサービスを発見できることがある
- `mail.example.com`, `admin.example.com`, `api.example.com` など

**公開サービス**
- SSH、RDP、FTP など、外部からアクセス可能なリモートログインサービス

**Webサーバーのメタ情報**
- HTTP ヘッダーのサーバー情報、バージョン
- `Server: Apache/2.4.41` という情報から「このバージョンに既知の脆弱性がないか」を攻撃者は調べる

**お忘れのWebアプリ**
- 昔作った開発用サーバーをそのまま外部公開していた
- テスト環境へのアクセス
- デフォルトクレデンシャル（username: admin, password: admin）

攻撃者は **Shodan** や **Censys** といったサーチエンジンを使い、こうした公開サービスをスキャンします。それは Google で検索するのと似ていますが、ネットワーク側面に特化しています。

<Callout title="自分のサービスは何が見えているか">
実際に Shodan でドメイン名を検索してみると「世界から見えている自分たちのサービス」が何か分かります。想定外のサービスが公開されていることは珍しくありません。
</Callout>

## 内部アタックサーフェス

外部からアクセスできなくても、一度「内側」に入られたら、攻撃者は横展開を始めます。

**権限設定の甘さ**
- ファイル共有フォルダに全員がアクセスできる
- データベースの読み取り権限が営業チーム全員に付与されている
- クラウドリソースにパブリックアクセスが有効になっている

**依存ライブラリの脆弱性**
- npm / pip / maven などで導入した外部ライブラリ
- 古いバージョンのまま放置されている
- その中に既知の脆弱性が存在

**設定ミス**
- デフォルトのパスワードのまま
- デバッグモードが本番環境で有効
- ログが過度に詳細で、敏感情報が記録されている

**内部ネットワークの透過性**
- 一度ノートPCが感染すると、内部ネットワーク全体が見える
- セグメント分けがないため、営業PC から 医療データサーバーまで自由にアクセス可能

**人為的ミス**
- 誤送信の添付ファイルに認証情報が含まれていた
- GitHub にプライベートキーをコミットしてしまった

## サプライチェーン攻撃——「自分たちの脆弱性」ではない脅威

ここ10年で増えているのが **サプライチェーン攻撃** です。

あなたのシステムが直接攻撃されるのではなく、**あなたが依存している外部パッケージ・サービスが侵害される**という流れです。

**有名な例：SolarWinds 事件（2020年）**
- SolarWinds は数千社の企業に使われるシステム管理ツール
- そのアップデートが改ざんされ、マルウェアが混入
- アップデートを適用した企業ネットワークが一斉に侵害された

**npm パッケージの悪意ある導入**
- オープンソースパッケージに似た名前のパッケージをアップロード
- 少しの違いだけで、暗号資産を盗むコードが仕込まれていた

つまり、あなたが「セキュアに作った」と思っているシステムも、依存するライブラリが侵害されたら、防ぎようがないということです。

## 攻撃の一般的な流れ

アタックサーフェスから、攻撃は一連の段階を経ます：

1. **偵察（Reconnaissance）**
  - 対象の IP、サーバーOS、Webサーバー、使用ライブラリを調べる
  - DNS、WHOIS、スキャンツールで情報を収集

2. **侵入（Initial Access）**
  - 外部アタックサーフェスの脆弱性を悪用
  - フィッシングメールのリンクをクリックさせる
  - デフォルトパスワードでログイン

3. **横展開（Lateral Movement）**
  - 一度入った後、内部ネットワークを探索
  - 他のシステムへのアクセスを試みる
  - 権限昇格を狙う

4. **永続化（Persistence）**
  - バックドアを仕込んで、消えない痕跡を作る
  - 検知を逃れるためバックアップに隠れる

5. **目的達成（Goal Achievement）**
  - データを盗む、改ざんする、エンクリプタンサムウェアを仕掛ける、など



## アタックサーフェスを減らす——最小権限原則

防御の基本戦略は **アタックサーフェスそのものを減らす** ことです。

**入口を塞ぐ**
- 不要な外部向けサービスは公開しない
- デフォルトサービスを無効化

**権限を必要最小限に**
- ユーザーには「その仕事に必要な権限だけ」を付与
- 全員が全リソースにアクセスできる設定は論外
- これを **最小権限原則（Principle of Least Privilege）** と呼びます

**依存を把握する**
- 使っているライブラリのバージョンを記録
- 定期的に脆弱性情報（CVE）をスキャン
- アップデートする仕組みを作る

**分離する**
- ネットワークセグメント：営業PC から医療データへは直接繋げない
- アプリケーション：管理画面と一般向けページを別のサーバーに置く
- コンテナ：Docker や Kubernetes で隔離

**監視する**
- 外部からのアクセス試行を記録
- 内部での権限昇格の試みを検知
- 異常なトラフィックパターンを注視

## ここで押さえること

- **アタックサーフェス**は「攻撃者が侵入・悪用できる全ての接点」
- 外部向けの公開サービスから、内部の権限設定の甘さまで、層の厚い入口が存在
- **サプライチェーン攻撃**では「自分たちが直接つくったコード」ではない部分が侵害される。依存ライブラリも自分の責任
- 攻撃は **段階的**。1段階で防御できれば後は起こらない
- 防御の原則は「入口を減らす」「権限を絞る」「分離・監視する」
