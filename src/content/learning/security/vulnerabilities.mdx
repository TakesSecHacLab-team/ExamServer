## CVE番号とCVSSスコアをどう読むか

セキュリティのニュースをスクロールしていると、こんな表記を見かけます。

```
CVE-2024-12345: Linux Kernel 権限昇格の脆弱性、CVSS 9.8
```

CVEとは何を表す番号なのか、CVSS 9.8という数字がどれくらいの危険を意味するのか。

この2つを読み解けるようになると、セキュリティの情報を「自分には関係のない雑音」と「対応が必要な警告」に分けられるようになります。

順番に整理していきます。

## 脆弱性とは何か

**脆弱性**とは、ソフトウェアやシステムの設計、実装、運用に存在する意図しない欠陥で、攻撃者がそれを利用して不正な行為に及べる状態を指します。

欠陥が一つもないソフトウェアは存在しません。

数百万行のコードがあれば、バグは統計的に必ず混ざります。

脆弱性という言葉が指すのは、そのバグの中でも「攻撃に利用される可能性を持つもの」です。

代表的な例を3つ挙げます。

**SQLインジェクション**は、利用者が入力した値をそのままデータベースへの問い合わせ文に組み込んでしまう設計上のミスです。

攻撃者はこの隙を使って、データベースの内容を読み取ったり書き換えたりします。

**バッファオーバーフロー**は、メモリに書き込む際の境界チェックが不十分な実装上のミスです。

攻撃者はこの隙を使ってメモリの内容を破壊し、想定外のコードを実行させます。

**設定ミス**は、デフォルトでデバッグモードが有効になっている、パスワードが弱いままになっているといった、運用段階での人為的なミスです。

ここで誤解しやすいのが、「脆弱性が存在する」と「攻撃が成功する」は同じ意味ではない、という点です。

その脆弱性が、そもそも自分の環境に当てはまるかどうかを確認するところから始まります。

## CVEという世界共通の番号

**CVE**（Common Vulnerabilities and Exposures）は、見つかった脆弱性一つひとつに付けられる、世界共通の識別番号です。

形式は `CVE-[年]-[連番]` で、`CVE-2024-1234` であれば、2024年に報告された脆弱性のうち1234番目に登録されたもの、という意味になります。

この番号が必要な理由は、同じ脆弱性を異なる組織が異なる呼び方で説明することがあるためです。

「Linuxカーネルの〇〇に関する問題」という説明と、「Kernel: privilege escalation in module XYZ」という説明が、同じ脆弱性を指していることがあります。

CVE番号があれば、世界中の技術者が「CVE-2024-1234のことだ」と同じ対象を指し示せます。

ソフトウェアの開発元はそのCVE番号に対応づけてパッチの情報を公開し、利用者はその番号で対応状況を追跡できます。

CVE番号は、脆弱性に関する情報を世界中で流通させるための共通言語として機能しています。

<Callout title="CVE番号を検索すると何が分かるか">
CVE番号をブラウザで検索すると、その脆弱性について次のことを調べられます。

どのソフトウェアのどのバージョンに存在するか。

どんな影響があるか（情報漏洩か、サービス停止か)。

修正パッチが公開されているか。

実際の攻撃に利用された記録があるか。

これらの情報は、NVD（National Vulnerability Database）やGitHub Advisoriesのようなデータベースで公開されています。
</Callout>

## CVSSという危険度の数値化

CVE番号があれば「この脆弱性が存在する」ことは分かります。

ただ、同じ日に10件のCVEが報告されたとき、どれを先に対応すべきかは、番号だけでは判断できません。

そこで使われるのが**CVSS**（Common Vulnerability Scoring System）です。

脆弱性の危険度を、0.0(危険なし)から10.0(最も危険)までの数値で表します。

CVSSスコアは、次の8つの要素を組み合わせて算出されます。

    <tr>

      <td>攻撃を成立させる条件が単純か、複雑な準備が必要か</td>
    </tr>
    <tr>

ネットワーク経由で、認証不要で、複雑な準備なしに悪用でき、重大な影響を及ぼす脆弱性は、CVSS 9.8のような高い値になります。

物理的なアクセスが必須で、影響も限定的な脆弱性は、CVSS 3.5のような低い値になります。

## スコアの高さと自分の環境へのリスクは別問題

セキュリティ対応を始めたばかりの担当者は、こう考えがちです。

「CVSSスコアが9.0以上だから、今すぐパッチを当てなければいけない」。

この考え方には、危険な見落としがあります。

CVSSスコアが表しているのは、その脆弱性が持つ潜在的な危険度です。

その危険が「自分の環境に実際に存在するか」は、全く別の問題です。

3つの例で確認します。

1つ目は、対象のバージョンを使っていない場合です。

Apache 2.2.15に見つかったリモートコード実行の脆弱性(CVSS 9.8)が報告されても、自分の環境でApache 2.4.41を使っているなら、そのバージョンには脆弱性が存在しないため、リスクはありません。

2つ目は、攻撃の前提条件が自分の環境に当てはまらない場合です。

認証後にのみ悪用できるリモートコード実行の脆弱性(CVSS 8.0)が報告されても、その機能を社内ネットワークだけに公開しているなら、外部の攻撃者にとってのリスクはなく、内部者による悪用のリスクに限定されます。

3つ目は、影響の前提となる機能を使っていない場合です。

特定のUnicodeフォント処理に見つかった不具合(CVSS 7.5)が報告されても、そのフォントを使用しておらず自動ダウンロードも無効にしているなら、リスクはほぼありません。

これらの例から、セキュリティ対応の手順は次のように整理できます。

CVEを発見し、CVSSスコアで潜在的な危険度を確認し、その脆弱性が自分の環境に実際に存在するかを確認し、存在する場合はその脆弱性がどの程度の確率で悪用されるか(攻撃のコードが既に公開されているかなど)を判断し、リスクと対応コストを見比べて優先順位を決めます。

CVSSスコアは、3番目の「自分の環境に存在するか」という確認の前に見る、参考情報の一つに過ぎません。

## パッチを当てるという作業の現実

脆弱性が自分の環境に存在すると分かったら、次に行うのは**パッチ**の適用です。

パッチとは、開発元が脆弱性を修正したプログラムの更新版です。

これを適用することで、その脆弱性を塞ぎます。

パッチ管理には、いくつかの現実的な制約があります。

新しく見つかった脆弱性の場合、修正版がまだ公開されていないことがあります。

アップデートによって他の機能が動かなくなる可能性があります。

サービスを止めてからでないと適用できない場合があります。

あるライブラリを更新したら、別のライブラリとの組み合わせに不整合が生じることがあります。

こうした制約があるため、組織は重要度と緊急度、テスト環境での動作確認の結果を見比べながら、段階的にパッチを適用します。

## ゼロデイとNデイの違い

脆弱性が公開されるまでの経緯は、攻撃のリスクに大きく影響します。

**ゼロデイ脆弱性**は、ソフトウェアの開発元がその脆弱性の存在を知る前に、攻撃者が先に悪用を始めている状態です。

防御側はまだ対策を講じる手段を持たず、危険度が高くなります。

**Nデイ脆弱性**は、開発元が脆弱性を公開し、修正パッチが既に出ている状態です。

「N日」とは、公開からN日が経過していることを表します。

パッチを適用していれば、この種の脆弱性への対応は完了しています。

CVEのデータベースやShodanのようなサービスでは、ある脆弱性が実際の攻撃に使われているかどうかも調べられます。

ゼロデイ脆弱性が見つかると、セキュリティのニュースは一斉に取り上げます。

## ここで押さえること

脆弱性とは設計や実装に存在する欠陥であり、完全に欠陥のないソフトウェアは存在しません。

CVE番号は脆弱性に付けられる世界共通の識別子で、情報の流通を効率にします。

CVSSスコア(0.0から10.0)は脆弱性の潜在的な危険度を表しますが、自分の環境への実際のリスクとは別物です。

スコアが高くても、自分の環境にその脆弱性が当てはまるかどうかを先に確認します。

パッチ管理は、見つかった脆弱性を即座に全て塞ぐ作業ではなく、リスクと対応コストを見比べて優先順位をつける作業です。

ゼロデイとNデイでは、防御側が取れる対応の余地が大きく異なります。
